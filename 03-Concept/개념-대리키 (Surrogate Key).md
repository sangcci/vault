## 한 문장 정의

> (사전적) 비즈니스 의미 없이 행을 유일하게 식별하기 위해 인위적으로 추가한 키. 주로 auto-increment 정수나 UUID를 사용.
> (이해용) 도메인과 무관하게 "그냥 번호표"를 붙이는 것.

---

## 해결하는 문제

- 자연키(natural key)의 불안정성 — 이름, 이메일 등은 변경될 수 있음
- 복합 자연키 사용 시 발생하는 조인 복잡도
- 외부 시스템과의 키 노출 최소화

---

## 치르는 비용

- 정규화 이론상 "의미 없는 속성 추가"로 순수 정규화 원칙과 긴장 관계
- 자연키의 유일성 제약이 사라지므로 별도 UNIQUE 제약 필요

---

## 동작 원리

```
자연키 방식:
┌──────────┬──────────┬──────────┐
│ 학생이름  │ 과목명   │ 강사이름  │
└──────────┴──────────┴──────────┘
  → 동명이인 발생 시? 이름 변경 시? 취약

Surrogate key 방식:
┌──────────┬──────────┬──────────┐
│ 학생ID   │ 과목ID   │ 강사ID   │  ← 각각 FK
└──────────┴──────────┴──────────┘

학생 테이블: 학생ID(PK) | 학생이름
과목 테이블: 과목ID(PK) | 과목명
강사 테이블: 강사ID(PK) | 강사이름 | 과목ID(FK)
```

### 정규화와의 관계

Surrogate key를 사용하면 각 Entity가 자기 테이블로 분리되면서 **구조적으로 BCNF 위반이 해소**되는 경향이 있음

```
BCNF 위반 사례:
강사 → 과목  (강사가 후보키가 아님)

Surrogate key 도입 후:
강사 테이블 자체가 강사ID(PK) → 과목ID(FK) 구조로 분리
→ 위반이 구조적으로 사라짐
```

단, 이는 surrogate key가 정규화를 보장하는 것이 아니라 **Entity 분리의 부산물**로 달성되는 것.
