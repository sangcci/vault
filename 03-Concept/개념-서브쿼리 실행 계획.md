## 한 문장 정의

> (사전적) 서브쿼리 실행 계획은 SQL 안의 하위 `SELECT`를 planner가 비용 기반으로 `InitPlan`, `SubPlan`, `Semi Join`, `Anti Join`, `Materialize` 같은 실행 형태로 바꿔 실행하는 방식이다.
> (이해용) 서브쿼리는 “괄호 안을 먼저 실행”하는 고정 절차가 아니라, planner가 더 싸다고 판단한 모양으로 다시 배치되는 작은 query tree다.

## 해결하는 문제

- 서브쿼리는 항상 안쪽부터 실행된다는 오해를 막는다.
- SQL 문장 순서와 실제 실행 계획 순서가 다를 수 있음을 설명한다.
- `IN`, `EXISTS`, scalar subquery, FROM subquery가 왜 서로 다른 plan node로 보이는지 연결한다.

## 치르는 비용

- planner가 항상 사람이 기대한 최적 형태로 바꾸지는 못한다.
- 상관 서브쿼리는 outer row마다 실행되는 `SubPlan`으로 남을 수 있다.
- `Materialize`가 생기면 중간 결과를 저장하고 재사용하지만, row가 크면 memory/disk 비용이 생긴다.

## 동작 원리

```text
SQL
 |
 v
Parser
- subquery를 query tree 안의 하위 query tree로 표현
 |
 v
Planner / Optimizer
- 독립 서브쿼리인가?
- outer query column을 참조하는가?
- EXISTS/IN을 join으로 바꿀 수 있는가?
- materialize가 더 싼가?
 |
 v
Executor
- InitPlan / SubPlan / Semi Join / Anti Join / Materialize 등 실행
```

핵심은 planner가 쿼리 문장 순서를 그대로 따르는 실행기가 아니라는 점이다. 같은 결과를 내는 여러 path를 만들고 비용을 추정한 뒤 더 빠를 것으로 보이는 계획을 고른다.

```sql
SELECT *
FROM orders
WHERE user_id IN (
  SELECT id
  FROM users
  WHERE status = 'ACTIVE'
);
```

가능한 물리 실행:

```text
Hash Semi Join
  -> Scan orders
  -> Hash
       -> Scan users
          Filter: status = 'ACTIVE'
```

즉 `IN (SELECT ...)`라고 해서 반드시 subquery를 row마다 실행하지 않는다. 독립적인 subquery는 한 번 계산하거나 join으로 풀릴 수 있다.

상관 서브쿼리는 다르다.

```sql
SELECT *
FROM orders o
WHERE EXISTS (
  SELECT 1
  FROM payments p
  WHERE p.order_id = o.id
);
```

가능한 물리 실행:

```text
Nested Loop Semi Join
  -> Scan orders o
  -> Index Scan payments p
       Index Cond: p.order_id = o.id
```

상관되어 있어도 planner가 decorrelation할 수 있으면 semi join으로 바꾼다. 못 바꾸면 `SubPlan`으로 남고 outer row마다 실행될 수 있다.
