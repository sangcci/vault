# 개념: 차단 I/O (Blocking I/O)

## 📝 정의
I/O 작업(읽기, 쓰기 등)을 요청했을 때, 요청한 작업이 완료되거나 특정 조건이 충족될 때까지 호출한 프로세스/스레드의 제어권을 OS가 회수하여 대기 상태로 만드는 방식입니다.

## ⚙️ 핵심 원리

### 1. 제어권의 반납
- 프로세스가 시스템 콜(예: `read()`)을 호출하면, 제어권이 커널로 넘어갑니다.
- 데이터가 준비되지 않았다면, 커널은 해당 프로세스를 실행 대기 명단에서 제외하고 **Wait Queue(대기 큐)**로 이동시킵니다.

### 2. 문맥 교환 (Context Switch)
- 잠든 프로세스 대신 다른 '실행 가능' 상태의 프로세스가 CPU를 점유합니다. 이 과정에서 `[[개념-문맥 교환 (Context Switch)]]`이 발생합니다.

### 3. 깨움 (Wakeup)
- I/O 데이터가 도착하거나 버퍼에 공간이 생기면, 커널은 대기 큐에 있던 프로세스를 다시 **Ready(준비)** 상태로 깨웁니다.

## OS 관점 vs User Mode 관점

Blocking이 "문제"로 보이는 이유는 **관점의 차이** 때문이다.

### OS 관점: 모든 것은 Non-blocking

```text
User Thread          OS Scheduler         CPU
    │                     │                │
    │── read() syscall ──>│                │
    │                     │ 데이터 없음     │
    │                     │── sleep ───────> [Wait Queue로 이동]
    │   (잠듦)             │                │
    │                     │                │── 다른 스레드 실행 (CPU 낭비 없음)
    │                     │                │
    │                     │ I/O 완료 인터럽트│
    │                     │── wake ────────> [Ready Queue로 이동]
    │<── 데이터 반환 ──────│                │
```

- OS는 I/O 대기 중 스레드를 **Busy Waiting** 시키지 않는다 — Sleep 상태로 전환해 CPU를 다른 스레드에 양보
- **OS 입장에서는 항상 Non-blocking** — CPU가 멈추는 일은 없다

### User Mode 관점: Thread를 재사용할 수 없다

```text
Thread Pool (예: Tomcat Worker)
┌────────────────────────────────────────────────────┐
│ Thread-1 ──── DB 조회 시작 ──────> [sleeping] ← 재사용 불가
│ Thread-2 ──── 외부 API 호출 ─────> [sleeping] ← 재사용 불가
│ Thread-3 ──── 파일 I/O 시작 ─────> [sleeping] ← 재사용 불가
│ Thread-4 [대기] ← 신규 요청 할당 가능
│ (이후 Thread 소진 시 → 신규 요청 대기 큐 적체)
└────────────────────────────────────────────────────┘
```

- OS가 스레드를 통째로 재우기 때문에, **sleeping 중인 스레드는 다른 요청에 할당 불가**
- User mode에서 보면 "스레드가 묶여 사용 불가 = Blocking"
- Thread Pool 소진 → 신규 요청 대기 → 처리량 한계 도달

> 이벤트루프(Node.js, Redis)와 코루틴/고루틴/Virtual Thread가 이 문제를 해결하는 방식은 [[개념-이벤트 루프]], [[개념-코루틴 (Coroutine)]] 참고.

## 💡 본질과의 연결
- **`[[본질-역압 (Backpressure)]]`의 구현**: 파이프나 소켓에서 버퍼가 가득 찼을 때 생산자를 멈추게 하는 물리적 수단이 바로 이 차단 I/O입니다.
- **`[[본질-추상화 (Abstraction)]]`**: 개발자는 복잡한 스케줄링 로직을 몰라도, 단순히 "데이터를 읽을 때까지 기다린다"는 직관적인 코드를 짤 수 있습니다.
