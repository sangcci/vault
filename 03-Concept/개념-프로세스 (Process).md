## 한 문장 정의

> (사전적) 실행 중인 프로그램(a program in execution)으로, 주소 공간·CPU 상태·열린 자원을 포함하는 실행 컨텍스트 전체.
> (이해용) 프로그램이 디스크에 있는 "레시피"라면, 프로세스는 그 레시피로 실제 요리를 하고 있는 "주방 전체" — 재료(메모리), 도구(레지스터), 진행 상황(PC)까지 포함.

---

## 해결하는 문제

- 하나의 CPU에서 여러 프로그램을 동시에 실행하는 것처럼 보이게 하려면, 각 실행 단위를 독립적으로 관리할 수 있는 단위가 필요
- 프로그램 간 메모리·자원 격리를 보장하는 경계 단위

---

## 치르는 비용

- 프로세스 생성/전환마다 [[개념-문맥 교환 (Context Switch)]] 오버헤드 발생
- 프로세스 간 통신(IPC)은 스레드 간 통신보다 비용이 큼 (주소 공간이 분리되어 있으므로)

## 동작 원리

### 프로그램 vs 프로세스

```text
[프로그램 (Program)]          [프로세스 (Process)]
  수동적 실체                   능동적 실체
  디스크의 실행 파일             메모리에 로드된 실행 상태
  하나의 프로그램 →              여러 프로세스 가능 (동일 프로그램 다중 실행)
```

### 프로세스 = 컨텍스트

프로세스는 단일 객체가 아니라 **컨텍스트 전체**:

```text
┌─── 프로세스 (Context) ────────────────────┐
│                                           │
│  [주소 공간]     Text | Data | Heap | Stack│
│  [CPU 상태]      PC, SP, GPR, Flags       │
│  [열린 파일]     fd 0(stdin), 1, 2, ...    │
│  [IO 장치]       할당된 디바이스            │
│  [메타데이터]    PID, 상태, 우선순위        │
│                                           │
└───────────────────────────────────────────┘
```

같은 명령어라도 **다른 컨텍스트**에서 실행하면 다른 결과 — "In what context?"와 같은 원리.

### 프로세스 상태 전이

```text
         ┌──────────┐
   생성 →│   New    │
         └────┬─────┘
              ↓ (admitted)
         ┌──────────┐     dispatch      ┌──────────┐
         │  Ready   │ ───────────────→  │ Running  │
         └──────────┘ ←───────────────  └──────────┘
              ↑        interrupt/preempt     │
              │                              │ I/O or event wait
              │         ┌──────────┐         ↓
              └──────── │ Waiting  │ ←───────┘
           event done   └──────────┘
```
