## 한 문장 정의

> (사전적) 애플리케이션 프로세스 내부 메모리에 데이터를 저장하여 네트워크 없이 즉시 조회할 수 있는 캐시.
> (이해용) JVM 힙 메모리 안에 데이터를 넣어두는 것. Redis 없이도 빠르지만, 서버가 여러 대면 각자 따로 들고 있어 값이 달라질 수 있음.

## 해결하는 문제

- 외부 캐시(Redis) 네트워크 트래픽 없이 응답속도 및 처리량 개선.
- 외부 서비스 운영 비용 없이 캐시 효과 확보.
- "자주 변하지 않고, 완벽히 실시간 동기화가 필요 없는 조회용 데이터"에 적합.

## 치르는 비용

- **서버 간 불일치**: 다중 인스턴스 환경에서 각 서버의 캐시가 달라질 수 있음 ([[본질-일관성의 종류 (Consistency Types)]] → 최종적 일관성).
- **힙 메모리 소비**: GC 부담 증가.
- **재시작 시 캐시 소멸**: 콜드 스타트 발생.

## 동작 원리

### Spring @Cacheable — AOP 프록시 흐름
```text
클라이언트 요청
  └→ AOP Proxy (메서드 진입 전 인터셉트)
       ├─ 캐시 HIT  → 메서드 미실행, 캐시 value 즉시 반환
       └─ 캐시 MISS → 실제 메서드 실행
                         └→ DB 조회
                              └→ 반환값 전체를 캐시에 저장
                                   └→ 반환
```
- **저장 단위**: 메서드 반환값 전체 (List, Page, 단일 객체 등). 레코드 단위 저장 없음.
- **갱신 단위**: `@CachePut`도 전체 덮어쓰기. 부분 업데이트(partial update) 개념 없음.
- **AOP 프록시 한계**: 같은 클래스 내 self-invocation은 프록시를 우회해 캐시 미적용 → [[현상-AOP Self-Invocation 문제]]

### 캐시 키 설계 원칙
```text
기본 키 = 메서드 파라미터 조합

findAll("ASC")  → key: "ASC"  → 독립 엔트리
findAll("DESC") → key: "DESC" → 독립 엔트리
                                    ↑
                             Spring이 자동 재정렬하지 않음.
                             재정렬이 필요하면 키를 정규화하고
                             애플리케이션 코드에서 직접 sort.
```
- 파라미터가 정렬 조건만 다를 경우 → 캐시 미스가 증가하거나 중복 엔트리가 누적됨.
- 권장: 정렬을 제외한 데이터 식별자만 키로 쓰고, 캐시 조회 후 메모리에서 sort.

### Caffeine — Read 경로 (lock-free)
```text
Read 요청
  └→ StripedBuffer (링 버퍼 분산, lock-free)
       └→ 여러 스레드가 동시에 읽어도 경합 없음 (MVCC 유사)
            └→ W-TinyLFU 알고리즘으로 eviction 비동기 결정

Write 요청
  └→ 버퍼에 기록 후 비동기 정리 (drain)
```

### Spring CacheManager — sync 옵션
```text
sync=false (기본):
  캐시 미스 시 여러 스레드가 동시에 계산 진입 → 중복 계산 (thundering herd)

sync=true:
  캐시 미스 시 하나의 스레드만 계산, 나머지 대기 → 중복 계산 방지
```

### Cache vs Buffer 구분
```text
┌──────────────────────────────┬──────────────────────────────┐
│ Cache (캐시)                  │ Buffer (버퍼)                 │
├──────────────────────────────┼──────────────────────────────┤
│ 빠른 장치 ↔ 빠른 장치         │ 빠른 장치 ↔ 느린 장치         │
│ 동일 데이터를 반복 읽기        │ 데이터는 단 한 번만 읽고/씀   │
│ 양쪽 모두 캐싱 사실을 모름     │ 역압(backpressure) 조절 역할  │
│ 예: CPU L1, Caffeine, Redis   │ 예: I/O 버퍼, TCP 수신 버퍼  │
└──────────────────────────────┴──────────────────────────────┘
```
