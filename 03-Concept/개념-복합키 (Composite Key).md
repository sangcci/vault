## 한 문장 정의

> (사전적) 두 개 이상의 컬럼을 조합하여 하나의 기본키(Primary Key)를 구성하는 방식.
> (이해용) 주민등록번호 대신, 이름+생년월일+전화번호를 합쳐서 신분증으로 쓰는 것.

---

## 해결하는 문제

- M:N 관계의 중간 테이블에서 별도 대리키(surrogate key) 없이 **자연스러운 유일성**을 보장.
- 통계성 테이블에서 (날짜, 카테고리) 조합 자체가 식별자인 경우.

---

## 치르는 비용

### 1. FK 전파 비용

```text
[OrderDetail]                    [Delivery]
┌─────────────────────┐          ┌────────────────────────────┐
│ PK: (order_id,      │          │ FK: order_id    (필요)     │
│      product_id)    │◄─────────│ FK: product_id  (필요)     │
│ quantity            │          │ address                    │
└─────────────────────┘          └────────────────────────────┘
                                   ↑ 복합키 전체를 FK로 가져와야 함
```

- 복합키를 참조하는 **모든 FK 테이블**에 구성 컬럼이 통째로 복제됨.
- 복합키 컬럼이 변경되면 관련 FK 테이블 **전부 수정** 필요.

### 2. 인덱스 효율 제약

```text
복합키 (order_id, product_id) 인덱스 동작:

  ✅ WHERE order_id = ?                      ← 선두 컬럼, 인덱스 사용
  ✅ WHERE order_id = ? AND product_id = ?   ← 전체 사용
  ❌ WHERE product_id = ?                    ← 선두 누락, 풀스캔
```

- 선두 컬럼 순서가 쿼리 패턴과 맞지 않으면 인덱스를 타지 못함.
- **카디널리티가 높은 컬럼을 선두에 배치**해야 스캔 범위가 좁아짐.

### 3. 이상 현상과의 연결

복합키의 구성 컬럼에 **부분 함수적 종속**이 존재하면 [[현상-DB 이상 현상]]이 발생한다.

```text
[수강 테이블] - 비정규화 상태
PK: (학생ID, 과목코드)

학생ID | 과목코드 | 학생이름 | 과목명   | 성적
─────────────────────────────────────────────
  1    | CS101   | 김철수  | 운영체제 | A
  1    | CS102   | 김철수  | 데이터베이스 | B
  2    | CS101   | 이영희  | 운영체제 | A+

문제: 학생이름은 학생ID만으로 결정됨 (부분 종속)
     과목명은 과목코드만으로 결정됨 (부분 종속)

→ 갱신 이상: 김철수 이름 변경 시 2행 모두 수정 필요
→ 삽입 이상: 수강 없는 신규 과목을 등록할 수 없음 (PK 불완전)
→ 삭제 이상: 2번 학생의 수강 삭제 시 CS101 과목 정보도 소실
```

이것이 **제2정규형**이 부분 함수적 종속을 제거하는 이유이며, [[개념-함수적 종속성 (Functional Dependency)]]에서 수학적으로 증명된다.

### 4. JPA 복합키 순서 문제

JPA의 `@IdClass`/`@EmbeddedId`는 내부적으로 **알파벳 순서**로 복합키를 구성한다. DB에서 의도한 인덱스 순서와 달라질 수 있으므로, DDL에서 직접 복합키를 정의하고 JPA는 이를 매핑만 하는 방식이 안전하다.

## 동작 원리

### 복합키를 고려할 수 있는 경우

| 상황 | 설명 |
| :--- | :--- |
| **통계성 테이블** | (날짜, 지역) 자체가 식별자이며 FK 참조가 없음 |
| **조회 전용** | 다른 테이블이 이 테이블을 FK로 참조하지 않음 |

### 대리키(surrogate key) 방식이 우세한 경우

| 상황 | 설명 |
| :--- | :--- |
| **FK 참조 존재** | 참조 테이블에 복합키 전체를 복제하는 비용 회피 |
| **제약 조건 변경 가능성** | PK 컬럼 변경 시 연쇄 수정 범위 최소화 |
| **일반적인 CRUD** | AUTO_INCREMENT ID가 단순하고 효율적 |
