## 한 문장 정의

> (사전적) 커널의 I/O 이벤트 알림(epoll/kqueue)을 사용해, 단일 스레드가 수만 개의 I/O 작업을 동시에 처리하는 무한 루프.
> (이해용) 알람을 여러 개 맞춰두고 울릴 때만 가서 처리하는 방식. 알람이 울리는 동안 딴 일을 할 수 있다.

---

## 해결하는 문제

- [[개념-차단 IO (Blocking IO)]]: I/O 완료 대기 중 CPU가 낭비되는 문제
- C10K 문제: [[개념-소프트웨어 스레드 (Software Thread)]]를 요청마다 생성하면 수만 연결에서 [[개념-문맥 교환 (Context Switch)]] 비용 폭증

---

## 치르는 비용

- **CPU Bound 작업에 취약**: 콜백이 CPU-intensive하면 루프가 막혀 모든 요청 대기 (Head-of-Line Blocking)
- await는 I/O 대기만 양보. CPU 연산은 양보하지 않음 → 개발자가 직접 청크로 쪼개야 함
- 디버깅 어려움: 비동기 흐름 추적

## 동작 원리

### epoll (Linux 커널 API)

이벤트루프의 핵심 엔진. "어떤 fd에 데이터가 준비됐는지" 커널이 직접 알려준다.

```
epoll_create()         ← epoll 인스턴스 생성
epoll_ctl(ADD, fd)     ← 감시할 fd 등록 (소켓, 파일 등)
epoll_wait()           ← 준비된 fd가 생길 때까지 블로킹 대기
  │ 커널: "fd5, fd12 준비됨"
  ▼
반환: [fd5, fd12, ...]  ← O(1), 준비된 것만 반환
```

- `select`/`poll`은 매번 전체 fd 목록을 순회 → O(N)
- `epoll`은 커널이 ready 목록을 관리 → O(1)

### 이벤트루프 전체 흐름

```
           ┌─────────────────────────────────┐
           │         Event Loop              │
           │                                 │
   ──────▶ │  epoll_wait()                   │
           │    │ (준비된 fd 반환)            │
           │    ▼                            │
           │  callback 실행                  │
           │    │                            │
           │    ├─ I/O 작업 → epoll 등록     │
           │    │   (커널에게 맡기고 즉시 반환)│
           │    │                            │
           │    └─ CPU 작업 → 루프 점유 ⚠️   │
           │         (epoll_wait 호출 불가)   │
           └─────────────────────────────────┘
```

### nginx / redis vs Node.js 비교

| | nginx / redis | Node.js / Coroutine |
|---|---|---|
| 이벤트루프 구현 | C로 직접 구현 (ngx_event, ae.c) | libuv 제공 |
| 커널 API | epoll / kqueue | epoll / kqueue (libuv가 래핑) |
| CPU 블로킹 통제 | 불가 (내부 설계자 책임) | 가능, but 개발자 책임 |
| CPU 블로킹 실사례 | redis `KEYS *` (O(N) 전체 스캔) | `await heavyCpuWork()` 남용 |

- nginx/redis는 내부 설계자가 명령어를 짧게 유지한다는 전제 위에 동작
- Node.js는 그 책임이 개발자에게 있음
