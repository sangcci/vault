## 한 문장 정의

> (사전적) 속성 A의 값을 알면 속성 B의 값이 유일하게 결정될 때, B는 A에 함수적으로 종속된다고 하며 `A → B`로 표기한다.
> (이해용) A를 알면 B를 찾아낼 수 있는 관계. 학번을 알면 이름을 알 수 있는 것처럼.

---

## 해결하는 문제

- 데이터를 어떤 기준으로 분리해야 중복 없이 정보를 보존할 수 있는지에 대한 수학적 근거를 제공
- [[개념-DB 정규화 (Normalization)]]의 각 정규형이 "무엇을 제거하는가"를 정의하는 기반

---

## 구성 요소

- **결정자 (Determinant)**: `A → B`에서 왼쪽 A. 값을 결정하는 주체
- **종속자 (Dependent)**: 오른쪽 B. 결정자에 의해 결정되는 대상

---

## 동작 원리

### 종류

```
[완전 함수적 종속] — 2NF 달성 조건
복합키 (학번, 과목코드) 전체에 의해서만 결정됨

(학번, 과목코드) ──────────→ 성적   ✅


[부분 함수적 종속] — 2NF 위반
복합키의 일부만으로도 결정됨

(학번, 과목코드) ──→ 이름           ❌
     └─ 학번 ────→ 이름   (학번만으로 결정 가능)


[이행적 함수적 종속] — 3NF 위반
비키 속성이 다른 비키 속성을 결정함

학번 → 학과코드 → 학과명
 └──────────────→ 학과명   ❌ (간접 종속)
```

### 분석 순서

```
1. 각 속성이 무엇에 의해 결정되는가? (도메인 지식 기반)
2. 기본키와 비교하여 부분/이행적 종속 여부 판단
3. 위반 발견 시 해당 FD를 별도 테이블로 분리
```
