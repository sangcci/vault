## 한 문장 정의

> (사전적) 텍스트·이미지 등을 고차원 숫자 배열(벡터)로 변환해 저장하고, 의미적 유사도 기반으로 검색하는 데이터베이스.
> (이해용) 단어가 아닌 의미로 검색하는 DB — "강아지"를 검색해도 "개"가 나온다.

## 해결하는 문제

- 키워드 검색은 단어가 일치해야만 결과가 나옴
- LLM 기반 시스템은 의미적으로 유사한 문서를 찾아야 RAG가 효과적

## 치르는 비용

- Embedding 변환 비용 (저장 시, 검색 시 모두 발생)
- 키워드 검색보다 인프라 복잡도 높음
- 정확한 키워드 검색이 필요한 경우엔 오히려 부적합

## 동작 원리

```
[ Embedding: 텍스트 → 벡터 ]

"강아지" → [0.21, 0.79, 0.11, 0.44, ...]
"개"     → [0.20, 0.81, 0.10, 0.43, ...]  ← 가까움 (유사)
"데이터" → [0.91, 0.12, 0.73, 0.05, ...]  ← 멂 (비유사)

[ 검색 흐름 ]
질문 → Embedding 모델 → 질문 벡터
    → Vector DB에서 가장 가까운 벡터들 탐색 (kNN)
    → 유사 문서 반환 → RAG로 컨텍스트 주입
```

키워드 검색 vs Vector 검색:
| | Elasticsearch (키워드) | Vector DB |
|---|---|---|
| 검색 기준 | 단어 일치 | 의미 유사도 |
| "N+1 문제" 검색 | "N+1" 포함 문서만 | "쿼리 성능 저하" 문서도 검출 |
| 적합한 경우 | 정확한 용어 검색 | 자연어 질문 |
