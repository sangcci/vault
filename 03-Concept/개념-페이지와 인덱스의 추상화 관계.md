# 개념: 페이지와 인덱스의 추상화 관계

## 📝 정의
인덱스는 **논리적인 검색 구조(B+Tree 등)**이며, 페이지는 이 구조를 물리적으로 담아내기 위한 **범용적 컨테이너(Container)**입니다. 인덱스는 페이지라는 추상화된 단위를 빌려 자신의 논리적 노드(Node)를 구성합니다.

## ⚙️ 추상화 계층 구조

| 계층 | 역할 | 구현체 | 비유 |
| :--- | :--- | :--- | :--- |
| **Logical Layer** | 검색 로직 및 정렬 구조 관리 | **인덱스 (B+Tree)** | **건축 설계도** (방들의 배치와 경로) |
| **Storage Layer** | 고정 크기 데이터 관리 및 I/O | **DB 페이지 (Slotted Page)** | **표준 규격의 방(Room)** |
| **Physical Layer** | 실제 비트 저장 및 하드웨어 제어 | **OS 페이지 / 디스크 섹터** | **벽돌과 시멘트** |

## 🔄 상호작용 핵심

### 1. 1 노드 = 1 페이지 (1 Node = 1 Page)
- B+Tree 알고리즘에서 말하는 하나의 '노드'는 물리적으로 **하나의 DB 페이지**에 대응됩니다.
- 인덱스 로직은 "다음 노드로 이동하라"고 명령하지만, 실제로는 "다음 Page ID를 Buffer Pool에서 읽어오라"는 동작으로 변환됩니다.

### 2. Slotted Page의 재활용
- 인덱스 페이지 역시 가변 길이의 `Key`를 담아야 하므로, 데이터 페이지와 동일한 **Slotted Page** 구조를 가집니다.
- 다만 내용물이 '행 전체'가 아니라 **'(Key, RID)'**라는 특수한 데이터 조각일 뿐입니다.

## 💡 본질과의 연결
- **`[[본질-추상화 (Abstraction)]]`**: 인덱스 알고리즘은 디스크의 물리적 헤드 이동이나 섹터 번호를 고민하지 않습니다. 오직 '페이지'라는 추상화된 단위 위에서 논리적인 트리 탐색에만 집중합니다.
- **`[[본질-최소 단위의 불일치 (Granularity Mismatch)]]`**: 인덱스 노드 하나(16KB)를 수정해도 실제 하드웨어는 4KB 단위로 쓰기 때문에, 인덱스 페이지 역시 Torn Page 현상으로부터 자유롭지 못하며 이를 위한 복구 로직이 필요합니다.
