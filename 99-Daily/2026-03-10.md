# 기록

### Spring Boot 설정 파일 구조 정리

**목표**: `application.yml`을 단일 진실 소스(Single Source of Truth)로, 프로필별 파일은 환경변수만.

**변경 사항**:
- `src/test/resources/application.yml` 생성 → main의 `application.yml`을 **shadow**하여 테스트 환경 완전 격리
- `application-test.yml` 삭제 → 내용을 test의 `application.yml`로 통합
- `@ActiveProfiles("test")`는 유지 → `@Profile("!test")` 빈 분기용
- `logging`, `hikari.keepalive-time` 기본값을 main `application.yml`에 추가
- `loki.url`을 `${LOKI_URL:}` 환경변수로 전환 → prod/stag 파일에서 loki 설정 제거
- 루트 `docker-compose.yml` 삭제 (PostgreSQL, Redis VM 직접 운영)

**핵심 개념 - Shadow**:
- `src/test/resources/application.yml`이 `src/main/resources/application.yml`보다 classpath 우선순위가 높음
- 빈 파일이라도 main 설정을 완전히 가려버림 (shadow)
- 테스트에서 main 설정 변경의 영향을 받지 않음

### VM에 PostgreSQL 직접 설치 (Docker 대신)

**이유**: Docker의 overlay2 스토리지 드라이버가 디스크 I/O에서 약 5~10% 오버헤드 발생. DB는 I/O 집약적이라 체감됨.

**반면 Docker로 띄워도 되는 것들**:
- Redis: 메모리 기반이라 디스크 I/O 부담 없음
- Prometheus: 순차 쓰기(append-only), 페이지 단위 랜덤 I/O 아님
- Loki: chunk 단위 압축 저장, 인덱스만 관리
- Grafana: 읽기 전용 대시보드

**PostgreSQL vs Prometheus/Loki 저장 방식 차이**:

| | PostgreSQL | Prometheus/Loki |
|---|---|---|
| 쓰기 방식 | 랜덤 I/O + WAL fsync | 순차 쓰기 (append) |
| 저장 단위 | 8KB 페이지 | 세그먼트/chunk (SSTable 기반) |
| 트랜잭션 보장 | ACID (매 쓰기마다 fsync) | 최종 일관성 |
| I/O 빈도 | 매 트랜잭션 | 주기적 flush |

### 모니터링 스택 아키텍처

```
Host OS (모니터링)
├── Docker: Prometheus + Loki + Grafana + Cloudflared
│
├── KVM Guest 1 (web)  - Docker: Nginx + nginx-exporter + node-exporter
├── KVM Guest 2 (was)  - Docker: Spring Boot (Blue-Green) + node-exporter
├── KVM Guest 3 (db)   - PostgreSQL + postgres-exporter + node-exporter
└── KVM Guest 4 (cache) - Docker: Redis + redis-exporter + node-exporter
```

**메트릭 수집 방향**:
- Prometheus → 각 VM의 exporter를 **pull** (scrape)
- Spring Boot → Loki로 로그를 **push** (loki4j appender)

**Exporter 정리**:

| Tier | Exporter | 포트 | 비고 |
|------|----------|------|------|
| Nginx | nginx-prometheus-exporter | 9113 | stub_status 필요 |
| Spring Boot | actuator/prometheus | 8080 | exporter 불필요 (내장) |
| PostgreSQL | postgres-exporter | 9187 | pg_monitor 권한 필요 |
| Redis | redis-exporter | 9121 | |
| VM 리소스 | node-exporter | 9100 | 모든 VM + Host에 설치 |

### Docker 컨테이너에서 virbr0 (KVM) 네트워크 접근

**문제**: Prometheus Docker 컨테이너에서 `192.168.122.x` (virbr0 대역) VM에 접근 불가

**원인**: libvirt의 iptables 규칙이 Docker bridge(docker0)에서 virbr0로의 FORWARD를 차단

**해결**:
```bash
sudo iptables -I FORWARD -i docker0 -o virbr0 -j ACCEPT
sudo iptables -I FORWARD -i virbr0 -o docker0 -j ACCEPT

# 영속화
sudo apt install iptables-persistent
sudo netfilter-persistent save
```

**ufw로는 불가**: ufw는 INPUT/OUTPUT 체인만 관리, FORWARD 체인은 직접 제어 불가.
`/etc/ufw/before.rules`를 수정하면 가능하지만 직접 iptables가 더 명확.

**시도했다 철회한 방법 - `network_mode: host`**:
- Prometheus를 host 네트워크로 변경하면 virbr0 접근 가능
- 하지만 Cloudflared 등 다른 Docker 컨테이너에서 Prometheus 접근이 복잡해짐
- `extra_hosts`, 포트 변경 등 설정이 꼬임 → iptables FORWARD 규칙이 근본적 해결

### Cloudflare Tunnel로 모니터링 접근

**문제**: ISP가 3000번 포트 차단 (9090은 허용, 3000은 불가)

**해결**: Host OS에 별도 Cloudflare Tunnel(`monitoring`) 생성
- 기존 `allreva` Tunnel은 VM 내부 cloudflared가 connector → Host의 localhost 접근 불가
- 별도 Tunnel을 Docker Compose에 cloudflared 서비스로 추가

**라우팅**:
```
grafana.domain.com    → http://grafana:3000
prometheus.domain.com → http://prometheus:9090
loki.domain.com       → http://loki:3100
```

**Private 접근**: Cloudflare Zero Trust → Access Policy로 본인 이메일만 허용

### 로그 유실과 법적 문제

**감사 로그(Audit)**: 로그인, 결제, 개인정보 접근 → DB에 저장 (유실 불가, 법적 요건)
**운영 로그(Ops)**: 에러, 성능, 디버깅 → Loki로 충분 (유실 허용)

Loki가 유실을 전제로 설계된 건 아님. WAL도 있고 재시도도 함.
다만 PostgreSQL처럼 매 건마다 fsync 보장하지 않는다는 뜻.

### KVM VM에서 외부 네트워크 접근 불가 (GitHub Actions Runner Offline)

**증상**: was-tier VM의 self-hosted runner가 offline. 로그에 `Runner connect error: The HTTP request timed out after 00:01:40` 반복.

**진단 과정**:
1. VM 내부에서 `curl https://github.com` → 무한 hang (아웃바운드 443 차단 확인)
2. `ip route` → default gateway `192.168.122.1` (virbr0) 정상
3. 게이트웨이 ping → 정상
4. DNS → 정상
5. Host `sysctl net.ipv4.ip_forward` → `1` 정상
6. Host `iptables -t nat -L POSTROUTING` → **virbr0 대역 MASQUERADE 규칙 누락**
   - Docker(172.17.0.0/16, 172.18.0.0/16)와 다른 네트워크만 존재
   - KVM의 `192.168.122.0/24` 대역 규칙 없음

**원인**: libvirt가 자동 생성해야 할 NAT MASQUERADE 규칙이 누락됨. Docker 설치 후 iptables 규칙이 덮어써지거나 libvirtd 재시작 시 발생 가능.

**해결**:
```bash
sudo iptables -t nat -A POSTROUTING -s 192.168.122.0/24 ! -o virbr0 -j MASQUERADE
```

**영구 적용 방법 (택 1)**:
- `sudo netfilter-persistent save` (iptables-persistent 패키지)
- `sudo virsh net-destroy default && sudo virsh net-start default` (libvirt가 규칙 재생성)

### prod 프로필 Spring Boot 기동 시 로그 미출력 → 해결

**증상**: `docker logs app-blue` 시 배너와 logback WARN 1줄만 출력, 이후 로그 없음.
```
07:25:57,390 |-WARN in Logger[org.jboss.logging] - No appenders present in context [default] for logger [org.jboss.logging].
```

**원인**: `<springProfile>`이 `<root>`/`<logger>` 태그 **안에** 중첩되어 있었음.

```xml
<!-- 문제 구조 -->
<root level="INFO">
    <springProfile name="prod">
        <appender-ref ref="CONSOLE"/>
    </springProfile>
</root>
```

logback은 `<springProfile>`을 `<configuration>` 바로 아래에서만 정상 처리함. `<root>` 내부에 중첩하면 profile 매칭이 무시되어 appender가 0개로 등록됨.

**해결**: `<springProfile>`을 `<root>`/`<logger>` 바깥으로 이동.

```xml
<!-- 수정 후 구조 -->
<springProfile name="stag,prod">
    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="ASYNC_LOKI"/>
    </root>
    <logger name="com.backend.allreva" level="INFO" additivity="false">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="ASYNC_LOKI"/>
    </logger>
</springProfile>
```

**핵심**: logback-spring.xml에서 `<springProfile>`은 항상 `<configuration>` 직속 자식이어야 함. 내부 중첩 불가.

### FCM 초기화 실패 (Docker 환경) → 해결

**증상**:
```
Caused by: java.io.FileNotFoundException: class path resource [firebase/firebase-service-account-key.json] cannot be opened because it does not exist
```

**원인**: `application.yml`에 FCM 키 경로가 `classpath:`로 하드코딩됨.

```yaml
fcm:
  service-account-key: classpath:firebase/firebase-service-account-key.json
```

Docker 볼륨으로 `/app/firebase/firebase-service-account-key.json`에 마운트해도, `classpath:`는 JAR 내부만 탐색하여 외부 파일시스템을 찾지 못함.

**해결**: 환경변수로 분리하여 각 환경에서 직접 주입.

```yaml
fcm:
  service-account-key: ${FCM_SERVICE_ACCOUNT_KEY_PATH}
```

| 환경 | 값 |
|---|---|
| prod (Docker) | `file:/app/firebase/firebase-service-account-key.json` |
| local (dev) | `classpath:firebase/firebase-service-account-key.json` |

`file:` 프리픽스 → Spring `ResourceLoader`가 파일시스템에서 직접 읽음. Docker 실행 시 `-e FCM_SERVICE_ACCOUNT_KEY_PATH=file:/app/firebase/firebase-service-account-key.json` 주입.
