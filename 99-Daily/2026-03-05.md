### 로그 구성
- 보안에 민감한 로그들은 절대 stag, prod에 노출되면 안됨 -> 운영 시 모니터링 시스템이 내부망을 사용하진 않음. Slack 등 외부 API와 통신이 필수적이고 Grafana 자체가 여러 CVE가 나오는 소프트웨어이기도 함. 
- 개인정보보호법에 의거해서 필요하지 않는 데이터 보관 및 관리 시 처벌 당할 수 있음
- CI 환경 등에 중요 파라미터 값 노출하지 않도록 주의 필요, CI는 대부분 mock key이거나 임의 데이터겠지만 그래도 혹시나가 사람잡음.

너무 로그를 남기지 않으면 로그 분석할 수 없음. 로그를 너무 많이 남긴다면 스토리지 용량 이슈 발생함. 그 사이 적절선에서 관리하는 전략이 필요함. 어떻게 로그를 나눠야 할까

**SQL 파라미터 없이 쿼리 구조만 로그에 남기는 게 정답.**

```sql
-- 로그에 이렇게 남아야 함 (파라미터 없이)

SELECT * FROM user WHERE email = ?

-- 이렇게 남으면 안 됨

SELECT * FROM user WHERE email = 'user@example.com'
```

`hibernate.SQL=DEBUG`는 쿼리 구조를, `BasicBinder=TRACE`는 `?`에 들어가는 실제 값을 찍어. **SQL 구조 파악이 필요하면 `SQL=DEBUG`만, 파라미터값은 절대 TRACE 안 켜는 게 원칙**이야. stag에서도 마찬가지고.

### 브릿지
처음에 bridge는 전용 스위치 등의 하드웨어에서만 구동되는줄 알았음. kernel 단에서는 L3 ip layer부터 구성되고 NIC에서 L1, L2를 cover하는 줄 알았음. 하지만 linux kernel에서도 L2 MAC 테이블 관리 등을 담당하는 부분이 있다는 것을 알게 되었고, 이걸 직접 iptables로 만들거나 해서 조절 가능하다는 사실을 알게됨. 

또한 벽면 포트로 연결하면 ONT(Optical Network Terminal)로 연결되어 공유기를 통하지 않고 바로 온다는 사실을 알게됨. 공유기의 NAT와 방화벽 기능 등을 사용할 수 없음.
NAT, DHCP, 방화벽, bridge의 기능을 알 필요 있을듯. ip forwarding도 linux 단에서 설정 가능하다는 것도 알게됨. AWS의 포트포워딩 기술도 커널의 이런 부분을 활용하는걸까?
