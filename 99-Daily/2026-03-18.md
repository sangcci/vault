# 기록

### 알고리즘 TreeMap
백준 7662 이중 우선순위 큐 문제 푸는 과정
우선, 우선순위 큐를 2개를 이용해서 각각 최솟값, 최댓값을 담당하면 된다고 생각했음. 하지만 각각 저장할 때 한쪽 Queue는 무조건 순차탐색 O(n)을 거쳐야 했음. 시간초과로 실패.
이후에 LLM에게 물어서 TreeMap - RedBlackTree로 해결할 수 있음을 알게 됨. 처음 접함. 그래서 이것의 특징 및 과정을 자세히 정리함. BST와 Heap의 차이도 알 수 있었음. 가장 큰 차이는 Heap은 정렬할 때 단순히 트리의 부모 < 자식 관계면 되기 때문에 왼쪽, 오른쪽 자식 노드의 대소관계 따윈 필요없었음. 그래서 맨 처음 최솟값 혹은 최댓값(우선순위를 어떻게 설정하느냐에 따라 달라짐)의 삭제는 O(1)을 보장함. 근데 한쪽이 O(1)이면 나머지는 O(n) 순차탐색을 해야함. Heap 자체가 완전한 정렬은 아니기 때문에.
반면 RedBlackTree를 활용한 TreeMap은 둘 다 보장 가능함. 애초에 왼쪽 < 부모 < 오른쪽인 BST의 전위순위탐색(?) 베이스이기 때문에 완전 정렬이 보장되고, 왼쪽 자식의 값이 최솟값이고 오른쪽 자식 값이 최댓값을 절대 보장하기 때문에 트리의 높이 만큼인 O(logn)으로 확정됨. O(n)보다 훨씬 이득. 그밖에도 정렬 및 순차 탐색값도 O(logn)을 완전 보장. 
이 둘의 차이점과 어떤 상황에서 쓰일지를 정리하면 좋을듯.

### 차대절 신청 시 모집인원 카운트 로직에서 동시성 이슈 발생
모집 인원 카운트 증가 시 Lost Update 발생.
Lost update란? Read-Modify-Write 유형이며 update가 메인인 로직에서 발생. 
```
A                B
read count       
                 read count
update
commit <- lost
                 update
                 commit
```
특히 집계 컬럼인 경우 예의주시해야함.
Lost Update가 발생할 경우 데이터 정합성이 깨진다. 해결 방법은 아래 4가지를 고려했음.

#### 해결 방법 비교

**1. 집계 컬럼 제거 + 집계함수 사용**
- `passengerCount` 컬럼을 없애고, 신청 시마다 `COUNT(*)` 쿼리로 현재 인원 계산
- 인덱스 설계 필요, 조회 성능 저하 가능
- ❌ 단독으로는 불충분: 두 트랜잭션이 동시에 COUNT를 읽으면 여전히 check-then-act 경쟁 발생. DB 레벨 CHECK constraint나 락을 함께 써야 진정한 해결

**2. Serializable 격리 수준**
- 트랜잭션 격리 수준을 Serializable로 올리면 Lost Update 포함 모든 이상현상 방지
- ❌ 범위가 너무 넓음: 해당 로직 외 다른 모든 트랜잭션 성능에도 영향. 희생 대비 효과가 낮음

**3. 낙관적 락 (Optimistic Lock)**
- JPA `@Version` 컬럼으로 버전 충돌 감지, 실패 시 재시도
- 충돌이 드문 상황에서 유리 (락 대기 없음)
- ❌ 티켓팅급 동시 신청에는 불리: 충돌률이 높아지면 재시도 폭풍 → CPU 사용률 급증, throughput 오히려 저하. 신청이라는 중요한 로직에서 실패 가능성이 존재하는 것 자체가 UX에 치명적

**4. 비관적 락 (Pessimistic Lock) ← 채택**
- `rent_boarding_slots` row에 `SELECT FOR UPDATE` (쓰기 락)
- JPA `@Lock(LockModeType.PESSIMISTIC_WRITE)` 한 줄로 적용
- 동일 슬롯 신청은 직렬화됨, 다른 날짜 슬롯에는 영향 없음
- ✅ 정합성 최우선인 예약 로직에 적합

**Advisory Lock (PostgreSQL 고유)**
- MySQL의 Named Lock과 유사. "Advisory(권고적)" = OS/DB가 강제하지 않고 협력하는 코드끼리만 유효한 락
- Linux `flock()`도 같은 개념: 락을 확인하지 않는 프로세스는 그냥 접근 가능
- row가 존재하지 않아도 임의의 bigint 키로 락 가능 → check-then-act 경쟁(중복 참가자 삽입)까지 한 번에 해결 가능
- FOR UPDATE보다 shared memory만 접근하므로 오버헤드 낮음
- 활용 사례: 배치/스케줄러 중복 실행 방지, 외부 API 멱등성 보장, row 없는 상태에서 선점 필요한 경우

**5. 원자적 UPDATE**
- Read-Modify-Write 자체를 없애고 DB 단일 연산으로 처리
```sql
UPDATE rent_boarding_slots
SET passenger_count = passenger_count + :count
WHERE id = :id AND passenger_count + :count <= recruitment_count
```
- affected rows = 1 → 성공, 0 → 만석
- WHERE 절이 CHECK + ACT를 동시에 수행하므로 Check-and-Act 문제 없음 (읽기-쓰기 시간 간격 자체가 존재하지 않음)
- 추가 락 불필요, `passengerCount` 컬럼 유지 가능
- ✅ 락 오버헤드 없이 정합성 보장. 조회 일관성도 영향 없음 (Read Committed에서 커밋된 값 읽음)
- ❌ JPA 도메인 모델을 우회 → 네이티브/벌크 쿼리 필요, 영속성 컨텍스트 동기화 주의

### 부하테스트 후 발견한 이슈

**이슈 1: 로그 분석 어려움**
- AOP + MDC 트레이싱이 모든 요청에 찍혀서 Loki 로그 폭주 → 분석 불가
- 해결 방향: 테스트 시 Loki 쿼리에서 WARN/ERROR 레벨만 필터링
  ```logql
  {app="allreva"} | json | level =~ "WARN|ERROR"
  ```
- 동시성 이슈 분석 시에는 에러 코드로 직접 필터링
  ```logql
  {app="allreva"} |= "SLOT_FULL" or "RENT_JOIN_ALREADY_EXISTS"
  ```

**이슈 2: k6 스크립트 설계 문제**
- 100 VU × 1분 30초 = VU당 ~45번 요청 → 슬롯 28개에 수천 번 요청이 몰림
- 동시성 이슈를 발견하는 게 아니라 그냥 fail 수만 쌓이는 구조
- 해결 방향: `shared-iterations` executor 사용 → 50명이 딱 한 번씩 동시에 신청
  - `recruitment_count = 28` 기준, 50명 요청 시 28명만 성공해야 정상
  - `passenger_count > 28`이면 Lost Update 재현 확인
  ```js
  scenarios: {
    concurrent_apply: {
      executor: 'shared-iterations',
      vus: 50,        // 동시 실행 VU
      iterations: 50, // 전체 요청 (VU당 1번)
      maxDuration: '10s',
    }
  }
  ```
- sleep 제거, dummy-data의 recruitment_count도 28로 수정 필요
