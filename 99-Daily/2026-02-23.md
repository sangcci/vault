# 기록

### 모든 동시성 문제 공부하기
DB 레벨에서 발생할 수 있는 동시성 문제 7가지를 공부했다. 다음에는 Application단과 동시성 문제 해결 중 락을 공부할 예정이다.
- Dirty Read - Undo Log
- Non-Repeatable Read - MVCC
- Phantom Read - Lock 혹은 Serializable, 등등...
- Lost Update - MVCC X -> Lock..
- Read Skew - MVCC
- Write Skew - Lock..
- DeadLock - DB엔진이 보고 한개 강제종료..

### 프로젝트 리팩토링
리팩토링 진행이 완료되었다. 앞으로 rent, survey 로직을 더 견고히 하고, 요구사항을 더 공부해야겠다. 
- 콘서트홀 조회 시 비관적 락 - 베타락을 사용하는 것 -> update로직에서 RMW 문제가 발생해서 update원자성을 보장받지 못해 lost update(갱신 손실) 문제가 발생할 수 있음.
- 도메인 상태값이 너무 많음. 계층을 두어 관리할 필요 있음. 테이블은 그냥 json으로 묶을까도 고민중.
- 별도 테이블로 구분할 필요 없이 json으로 관리하면 될 것들이 많음. survey의 boardingDate나 concert 값들. 이들은 별로 update 즉 쓰기가 자주 발생하지 않아서 괜찮을 것 같음.
- bulk insert 데드락 조심 -> 배치 공부할 때 같이 공부해야 겠음.
- 조회수 및 별점 로직 정합성 필요할듯. 캐시도 같이 고민?
- testcontainer도입 완료.
- Elasticsearch 모두 제거 및 PostgreSQL로 전면 마이그레이션 진행. pg_trmq인가 검색할 때 도움을 주는 인덱스 이게 요물이라고 함. 이것도 같이 공부해야됌.
- 아티스트 받아올 때 배치.......
- 메서드 이름 컨벤션 필요할 듯. 핵사고날 아키텍처의 컨벤션을 참고하면 좋을 듯 하다.
