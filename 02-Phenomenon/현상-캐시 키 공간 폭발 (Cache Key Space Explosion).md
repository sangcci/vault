## 한 문장 정의

> (사전적) 검색 조건(필터, 키워드, 페이지 번호) 조합이 캐시 키 공간을 무한히 확장해 적중률이 0에 수렴하는 현상.
> (이해용) 경우의 수가 폭발하면 캐시는 DB 조회를 한 번 더 하는 메모리 낭비 레이어가 된다.

## 발생 환경

- 검색 조건(키워드, 카테고리, 정렬, 페이지 번호)이 2개 이상 조합되는 목록 API
- `@Cacheable` 기본 키 생성 전략(파라미터 조합)을 그대로 사용하는 경우

## 관찰되는 증상

- 캐시 HIT 0%, 모든 요청이 DB 조회로 이어짐
- 힙 메모리 사용량 지속 증가 (엔트리가 쌓이기만 하고 재사용 안 됨)
- eviction 빈발 → GC 부담 증가

## 원인 구조

```text
키워드: "Java" | "Spring" | "Redis" | ...   (n가지)
정렬:   "최신순" | "인기순" | ...            (m가지)
페이지: 1 | 2 | 3 | ...                     (k가지)

조합 가능한 캐시 키 수 = n × m × k → ∞
각 키는 단 한 번만 HIT 가능 → 적중률 ≈ 0%
```

- [[본질-참조 지역성 (Locality of Reference)]] 위반: 동일 데이터에 반복 접근이 발생하지 않음

## 추측되는 원인

- 동적 검색 파라미터를 캐시 키로 그대로 사용
- 캐시 적용 전 "키 공간이 유한한가" 검증 누락
