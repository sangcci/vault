## 한 문장 정의 (Definition)

> (사전적) 부모 프로세스가 자식보다 먼저 종료되어, 부모가 없는 상태로 남겨진 프로세스.
> (이해용) 부모가 사라진 자식 프로세스 — OS는 "부모 없음" 상태를 허용하지 않으므로 즉시 init이 새 부모가 된다.

## 발생 환경 (Context)

- 부모 프로세스가 자식의 종료를 `wait()`하지 않고 먼저 `exit()`한 경우
- 서버 프로세스가 crash될 때 자식 worker 프로세스들이 남는 경우
- 데몬 프로세스 생성 시 의도적으로 부모를 종료하는 경우 (double-fork 패턴)

## 관찰되는 증상 (Symptoms)

`ps` 또는 `pstree`에서 해당 프로세스의 PPID가 1로 변경됨. 프로세스 자체는 정상 실행 중이지만 관리 주체가 init으로 바뀜.

```text
[정상 상태]                     [부모 종료 후]
init (1)                         init (1)
  └── shell (500)                   ├── shell (500)
        └── myapp (501)             └── worker (502)  ← PPID가 1로 재지정
              └── worker (502)
```

## 추측되는 원인 (Root Cause)

리눅스 커널은 모든 프로세스가 유효한 부모를 가져야 한다고 설계됨. 부모 없는 프로세스는 종료 시 exit status를 회수할 주체가 없어 좀비 프로세스로 변할 수 있음. → [[개념-Init 프로세스 (PID 1)]]이 자동 입양하여 수거 역할 수행.

### 좀비 vs 고아 구분

| | 고아 프로세스 | 좀비 프로세스 |
|-|-------------|-------------|
| **상태** | 실행 중 | 종료됨 |
| **문제** | 부모가 없음 | 부모가 exit status를 회수 안 함 |
| **해결** | init이 입양 | 부모가 wait() 호출 |

## 관련 사례 (Related Cases)

- **의도적 고아 (데몬화)**: `fork()` → 부모 exit → 자식이 init의 자식이 되어 터미널 세션과 완전히 분리됨
- **init의 자동 복구**: 크리티컬 서비스가 죽으면 init(systemd)이 재시작 처리
