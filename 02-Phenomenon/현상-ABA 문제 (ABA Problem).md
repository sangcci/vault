## 한 문장 정의

> (사전적) 변수 값이 A -> B -> A로 바뀌었을 때, CAS 연산이 중간 변화를 인지하지 못하고 성공하는 현상.
> (이해용) 값은 같지만 "역사"가 다른 것을 구분하지 못하는 CAS의 한계.

---

## 발생 환경

- **CAS 기반 Lock-free 알고리즘**: 루프를 돌며 상태를 체크할 때 발생.
- **메모리 재사용**: 해제된 메모리 주소가 다시 할당되어 이전과 같은 값을 가질 때 발생.

---

## 관찰되는 증상

- **데이터 오염**: 이미 해제되었거나 변경된 데이터를 유효한 것으로 착각하여 연산 수행.
- **댕글링 포인터**: 스택/큐 등에서 이미 삭제된 노드를 가리키는 포인터를 재사용하여 런타임 오류 유발.

---

## 추측되는 원인

CAS의 맹점을 해결하기 위해 **Versioning(Stamping)** 기법을 사용합니다.
같은 `"a"` 값이라도 **(`"a"`, v1)**과 **(`"a"`, v3)**은 다른 것으로 간주하여, 그사이의 변화를 감지해낼 수 있습니다.

---

## 시각적 증거

### ABA 시나리오와 해결 (ASCII)
```text
[상황: 값 A가 B가 되었다가 다시 A가 됨]

Thread 1: Read (A, v1)
Thread 2: CAS (A, v1) -> (B, v2)  // 성공
Thread 2: CAS (B, v2) -> (A, v3)  // 성공

Thread 1: CAS ( (A, v1), (A, v3) ) 
          -> "값은 A지만 버전(v1 vs v3)이 다르네? 실패!"
```
