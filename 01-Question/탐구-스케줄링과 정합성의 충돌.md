# 탐구: 스케줄링과 정합성의 충돌

## 핵심 질문

"락(Lock)을 쥔 채로 OS에 의해 강제 선점(Preemption)당하면, 데이터 정합성은 어떻게 지켜지는가?"

## 탐구 결과

### 1. OS의 투명한 중단 (Transparency)

OS가 프로세스를 멈추고 `Ready` 리스트로 보내는 것은 하드웨어 레벨에서 상태를 저장하므로, 다시 실행될 때 프로세스는 자신이 멈췄었다는 사실조차 모릅니다. 따라서 로직 자체는 깨지지 않습니다.

### 2. 문제는 '락(Lock)'의 점유 시간

- **우선순위 역전 (Priority Inversion)**: 높은 우선순위의 스레드가 실행되어야 하는데, 낮은 우선순위의 스레드가 락을 쥔 채로 선점되어버리면 시스템 전체가 마비될 수 있습니다.
- **해결**: OS는 이럴 때 낮은 우선순위의 스레드의 우선순위를 한시적으로 높여주는 '우선순위 상속(Priority Inheritance)' 기법을 씁니다.

### 3. 하드웨어의 보증: CAS (Compare-And-Swap)

- 아주 짧은 연산(변수 증가 등)은 락을 걸지 않고 CPU 명령어 한 줄(**CAS**)로 처리합니다.
- 이 명령어는 하드웨어가 "절대 쪼개지지 않음"을 보장하므로, OS 스케줄링과 무관하게 원자성을 유지합니다.

### 4. 트랜잭션의 보호 (Isolation)

- 데이터베이스 트랜잭션은 OS 스케줄링보다 훨씬 더 높은 수준의 **격리성(Isolation)**을 제공합니다.
- OS가 나를 1초 동안 멈췄다 하더라도, DB의 락은 풀리지 않으므로 다른 트랜잭션이 내 데이터를 침범할 수 없습니다.

## 인사이트

> **"물리적인 CPU 주도권은 OS가 뺏어갈 수 있어도, 논리적인 데이터 주도권(Lock)은 명시적으로 놓아주기 전까지 보호된다."**
