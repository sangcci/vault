## 핵심 질문

> CPU의 문맥 교환, DB의 MVCC, Application의 트랜잭션 — 왜 서로 다른 계층에서 동일한 "스냅샷 저장 → 격리 실행 → 복원" 패턴이 반복되는가?

---

## 가설 및 추론

- 공유 자원을 여러 실행 단위가 번갈아 쓰는 구조 자체가 동일하기 때문
- "중간 상태 노출 방지"라는 문제의 본질이 계층과 무관하게 동일

---

## 검증 및 팩트

### 1. 세 계층의 동시성 문제 — 동일한 구조

```text
[문제 구조]  여러 실행 단위가 하나의 공유 상태를 번갈아 수정
             → 중간 상태가 다른 실행 단위에 노출
             → 결과의 정확성이 파괴됨

[CPU]   프로세스 A, B가 레지스터를 번갈아 사용
        → A의 레지스터 값이 B에게 노출/파괴

[DB]    트랜잭션 T1, T2가 같은 row를 번갈아 수정
        → T1의 중간 수정이 T2에게 노출 (Dirty Read)

[App]   스레드 1, 2가 같은 변수를 번갈아 수정
        → 경쟁 조건 (Race Condition)
```

### 2. 해결 패턴도 동일 — "Snapshot + Isolation + Restore"

| 계층 | 공유 자원 | 스냅샷 대상 | 저장 위치 | 복원 시점 |
|------|----------|------------|----------|----------|
| **CPU** | 레지스터, PC | CPU 전체 상태 | PCB (메모리) | 프로세스 재할당 시 |
| **DB (MVCC)** | Row 데이터 | 트랜잭션 시작 시점의 데이터 버전 | Undo Log / 버전 체인 | SELECT 시 해당 버전 읽기 |
| **App (STM)** | 공유 변수 | 트랜잭션 시작 시점의 변수 값 | 로컬 복사본 | Commit 시 충돌 없으면 반영 |

### 3. 왜 같은 패턴이 반복되는가

```text
근본 원인: 공유 + 교대 = 반드시 중간 상태 노출

유일한 해결 원리:
  "각 실행 단위가 자신만의 일관된 세계(Snapshot)에서 작업하고,
   다른 실행 단위의 중간 상태를 절대 보지 않게 격리한다"

이것은 CPU든 DB든 Application이든, 공유 자원 위의 동시성이 존재하는 한
구조적으로 회피할 수 없는 패턴.
```

### 4. 차이점: 충돌 해결 전략

- **CPU**: 충돌 자체가 불가능 (한 시점에 하나의 프로세스만 CPU 사용)
- **DB**: 충돌 감지 후 롤백 (Optimistic) 또는 락으로 선제 방지 (Pessimistic)
- **App**: CAS 재시도, 락, STM 롤백 등 다양

---

## 결론

- "스냅샷으로 격리"는 특정 기술의 트릭이 아니라, **공유 자원 위의 동시성이라는 문제 구조에서 필연적으로 도출되는 해법**
- CPU 레벨의 Context Switch를 이해하면 DB의 MVCC가 "당연한 귀결"로 보임
- 면접에서 "동시성 제어"를 물으면 계층을 넘나드는 이 구조적 동형을 설명할 수 있으면 깊이가 다름
