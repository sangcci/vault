# Question: 왜 Web-WAS-DB 계층 구조를 사용하는가?

## 🧐 질문의 배경
모든 백엔드 구성도를 보면 항상 Web Server, WAS, Database가 나누어져 있습니다. 한 곳에 몰아넣으면 관리도 편하고 서버 간 통신 비용(Network Latency)도 없을 것 같은데, 왜 굳이 이렇게 복잡하게 나누는 것일까? 단순히 관습적인 것인지, 아니면 거부할 수 없는 공학적 이유가 있는지 탐구해 봅니다.

---

## 💡 핵심 통찰 (Insight)
### 1. 자원의 효율적 활용과 경제성 (Scaling Strategy)
각 계층은 사용하는 하드웨어 자원의 성격이 다르며, 이를 분리해야 **비용 대비 성능**을 극대화할 수 있습니다.
- **Web Server**: 이미지, CSS 등 정적 컨텐츠 전송에 특화. CPU보다는 네트워크 I/O 자원을 주로 사용합니다.
- **WAS (CPU 집약적)**: 비즈니스 로직 연산, 데이터 가공을 담당. CPU와 메모리 자원을 많이 소모합니다.
- **DB (I/O 및 메모리 집약적)**: 데이터의 영속성 관리 및 무결성 보장. 고성능 디스크(NVMe 등)와 대용량 메모리가 필수적입니다.
- **경제적 선택**: 사용자가 늘어날 때, **비싼 DB 서버를 통째로 업그레이드(Scale-up)**하는 것보다, **상대적으로 저렴한 WAS 서버를 여러 대 늘리는(Scale-out)** 방식이 훨씬 저렴합니다.

### 2. 로직의 위치: "왜 DB에서 로직을 빼서 WAS로 옮겼는가?"
과거 레거시 시스템은 DB 내부의 **Stored Procedure**에 로직을 두었지만, 현대 아키텍처는 이를 WAS로 옮겼습니다.
- **DB 자원의 보호**: DB의 CPU는 데이터 무결성을 지키는 데만 쓰기에도 아깝습니다. 복잡한 계산 루프는 WAS의 CPU에게 떠넘기는 것이 효율적입니다.
- **유지보수의 현대화**: DB 내부 로직은 Git 버전 관리나 단위 테스트(Unit Test)가 매우 어렵습니다. WAS(Java, Python 등)로 로직을 가져와야만 **객체지향 설계**와 **CI/CD**를 온전히 활용할 수 있습니다.

### 3. 보안의 계층화 (Defense in Depth)
DB는 서비스의 '금고'와 같습니다. 사용자가 금고에 바로 접근하게 두는 것은 보안상 치명적입니다.
- **완충 지대(DMZ)**: 외부 인터넷에 노출되는 것은 Web Server뿐입니다. WAS와 DB는 사설망(Private Network)에 숨겨 해커의 직접적인 접근을 차단합니다.
- **권한 분리**: Web Server가 털려도 WAS의 인증 로직이 막아주고, WAS가 털려도 DB의 접근 제어(ACL)가 마지막 방어선 역할을 합니다.

---
## 🚀 심화 탐구: 구조적 유연성과 가용성
### 1. 장애 전파 방지 및 무중단 운영
- **장애 격리**: WAS가 로직 처리 중 에러로 뻗더라도, 앞단의 Web Server는 살아있어 사용자에게 '서비스 점검 중' 페이지를 보여주거나 다른 살아있는 WAS로 요청을 넘길(Failover) 수 있습니다.
- **업데이트의 유연성**: 새로운 기능을 배포할 때, 여러 대의 WAS를 순차적으로 재시작(Rolling Update)하면 사용자는 서비스 중단을 전혀 느끼지 못합니다.

### 2. Frontend와 Backend의 분리 (SSR vs CSR)
예전엔 서버가 화면(HTML)까지 다 만들었지만, 이제는 데이터(JSON)만 전달하는 방식으로 변했습니다.
- **멀티 플랫폼 대응**: 하나의 백엔드가 웹, iOS, 안드로이드 기기에 동일한 데이터를 제공할 수 있습니다. 각 기기는 서버로부터 받은 데이터를 바탕으로 각자의 환경에 맞는 화면을 직접 그립니다.
- **전문성의 분화**: 프론트엔드와 백엔드의 기술 스택이 고도화됨에 따라, 각 영역의 전문성을 극대화하기 위해 계층을 분리합니다.

---
## 🚀 결론: Monolithic은 정말 사라지는가?
무조건적인 분리가 정답은 아닙니다.
- **Monolithic (통합 구조)**: 초기 스타트업처럼 빠른 실행이 중요하고 시스템 복잡도가 낮을 때는 한 곳에 몰아넣는 것이 개발 속도와 비용 면에서 유리합니다.
- **Tiered Architecture (계층 구조)**: 서비스가 성장하고, 보안·확장성·유지보수가 비즈니스의 핵심 리스크가 되는 시점부터는 반드시 계층을 나누어야 합니다.

**"현재 우리 서비스의 규모에서 어떤 트레이드오프가 가장 이득인가?"**를 판단하는 것이 아키텍처 설계의 본질입니다.
