# Question: DBMS는 왜 OS의 가상 메모리를 믿지 않을까?

## 🧐 질문의 배경
`mmap()`을 쓰면 OS가 복잡한 디스크 I/O를 다 대신 해줍니다. 개발자는 그냥 메모리를 쓰듯 코드를 짜면 되는데, 왜 모든 고성능 DBMS(Oracle, MySQL, SQL Server 등)는 이 편리한 기능을 거부하고 직접 **Buffer Pool**이라는 복잡한 시스템을 밑바닥부터 만드는 것일까요?

---

## 💡 핵심 통찰 (Insight)

### 1. 트랜잭션 안전성 (Transaction Safety)
OS는 성능 최적화를 위해 RAM에 있는 데이터를 '아무 때나' 디스크에 써버릴 수 있습니다. 하지만 DB는 ACID를 위해 **"로그(WAL)를 먼저 쓰고 나서야 데이터를 써야 한다"**는 엄격한 순서가 있습니다. OS에게 제어권을 맡기면 이 순서가 뒤섞여, 장애 발생 시 복구가 불가능해집니다.

### 2. 예측 불가능한 지연 (I/O Stalls)
OS의 `mmap`은 어떤 페이지가 RAM에 있고 어떤 게 디스크에 있는지 알려주지 않습니다. DB는 쿼리를 실행하다가 갑자기 발생하는 **Page Fault**로 인해 스레드가 멈추는(Stall) 상황을 몹시 싫어합니다. 직접 관리하면 어떤 데이터를 미리 읽어올지(Prefetch), 어떤 걸 내보낼지 완벽히 통제할 수 있습니다.

### 3. 에러 처리의 조악함 (Error Handling)
디스크 데이터가 오염되었을 때, 직접 I/O를 하면 체크섬 에러를 우아하게 처리할 수 있습니다. 반면 `mmap` 도중 에러가 나면 OS는 프로세스에 `SIGBUS` 시그널을 던집니다. 이는 DB 전체를 비정상 종료시킬 수 있는 치명적인 에러 처리 방식입니다.

---

## 🚀 백엔드 개발자의 시각
"범용적인 솔루션(OS)은 모든 상황에서 '적당히' 잘 돌아가도록 설계되었지만, DB처럼 극한의 성능과 안정성을 요구하는 시스템에서는 그 '적당함'이 오히려 독이 됩니다. **직접 통제권을 쥐는 것(Direct I/O)**은 복잡하지만, 시스템의 신뢰성을 보장하는 유일한 길입니다."
