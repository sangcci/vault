## 핵심 질문 (Question)

- 왜 설정 덮어쓰기(Overwrite) 메커니즘은 편리함에도 불구하고 예측 불가능한 버그와 디버깅 지옥을 만드는가?

## 메커니즘 분석 (Mechanism)

### 1. 암시적 동작 (Implicit Behavior)
- 시스템은 덮어쓰기가 발생할 때 경고를 주지 않음. 
- 개발자는 `A`라고 믿고 있는데, 시스템은 조용히 `B`를 적용함. 이러한 **"침묵하는 변화"**가 버그의 씨앗이 됨.

### 2. 부분적 소실 문제 (Shallow Copy vs Deep Merge)
- 객체 형태의 설정을 덮어쓸 때, 전체를 교체하느냐(Overwrite) 필드만 합치느냐(Merge)에 따라 데이터 유실이 발생함.

```javascript
// Pseudo Code: Overwrite의 함정
Default = { 
  db: { url: "localhost", pool: 10, timeout: 5000 } 
}

# 시나리오: pool 사이즈만 20으로 바꾸고 싶다.
Override = { 
  db: { pool: 20 } 
}

# 결과 (Spring Boot 등 대부분의 기본 동작):
Final = { 
  db: { pool: 20 } 
} 
# -> url과 timeout 정보가 증발해버림! (Partial Data Loss)
```

### 3. 진실의 파편화 (Fragmented Truth)
- 설정의 출처가 10군데라면, 현재 값이 어디서 왔는지 확인하기 위해 10군데의 계층을 모두 머릿속으로 시뮬레이션해야 함 (인지 과부하).

## 문제점 (Problems)

- **디버깅 비용 급증**: 코드 레벨이 아닌 인프라/설정 레벨의 문제이므로 로그에 명확히 찍히지 않을 때가 많음.
- **운영 사고**: 로컬에서 검증되지 않은 환경 변수가 운영 서버 기동 시 주입되어 예기치 못한 동작(예: 잘못된 DB 연결)을 유발함.

## 해결책 (Solutions)

- **명시적 병합(Deep Merge)** 지원 도구 사용.
- **설정 출처 로깅**: 기동 시 `PropertySource` 목록과 최종 결정된 소스를 로그에 출력.
- **[[판단기준-안전한 설정 오버라이드 관리 전략]]** 수립.
