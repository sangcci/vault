## 핵심 질문 (Core Question)

> NAT 테이블과 라우팅 테이블은 둘 다 패킷의 경로에 관여하는데, 각각 무엇을 결정하는가?

---

## 가설 및 추론 (Hypothesis)

- NAT은 패킷의 주소를 바꾸고, 라우팅은 패킷이 나갈 방향을 결정하는 것 같다
- 순서가 있을 것 — 주소를 바꾼 뒤 경로를 결정해야 올바른 인터페이스로 보낼 수 있을 것
- 둘 중 하나만으로는 포워딩이 완성되지 않을 것

---

## 검증 및 팩트 (Verification)

**역할 구분**
```
NAT 테이블     "봉투 주소 라벨 교체"
               공인IP:80 → 192.168.122.10:80
               패킷 내용은 그대로, 헤더만 수정

라우팅 테이블  "어느 출구(인터페이스)로 내보낼지 결정"
               192.168.122.x → virbr0 인터페이스로
               주소는 건드리지 않음
```

**동작 시점 — 순서가 핵심**
```
외부 패킷 도착
      │
[nat/PREROUTING]   NAT: 목적지 주소 교체
      │              공인IP:80 → 192.168.122.10:80
      ↓
[Routing Table]    라우팅: 변경된 주소 기준으로 인터페이스 결정
      │              192.168.122.x → virbr0
      ↓
[filter/FORWARD]   필터: 통과 허용 여부
      │
[nat/POSTROUTING]  NAT: 출발지 주소 교체 (SNAT/MASQUERADE)
      │
[VM 도착]
```

**순서가 바뀌면 어떻게 되는가**
```
라우팅 먼저, NAT 나중이라면?
    → 라우팅 테이블은 공인IP:80을 목적지로 봄
    → 공인IP는 외부 인터페이스(eth0)로 라우팅
    → VM(192.168.122.10)에 도달하지 못함

PREROUTING에서 NAT 먼저 해야
    → 라우팅 테이블이 192.168.122.10을 보고 virbr0으로 올바르게 전달
```

**각각 없으면 어떻게 되는가**
```
NAT 없이 라우팅만   → 공인IP로 온 패킷이 VM 사설IP로 변환되지 않아 목적지 불일치
라우팅 없이 NAT만   → 주소는 바꿨지만 어느 인터페이스로 보낼지 몰라 패킷 드롭
```

---

## 결론 (Conclusion)

- NAT과 라우팅은 역할이 완전히 다른 두 단계
- **NAT = 주소 변환** (헤더 수정), **라우팅 = 경로 결정** (인터페이스 선택)
- 반드시 NAT(PREROUTING) → 라우팅 → 필터(FORWARD) 순으로 협력해야 포워딩 완성
- "DNAT을 설정했는데 왜 안 되지?"의 원인 대부분은 라우팅 또는 FORWARD 체인에 있음
