## 핵심 질문

> 코드는 Git으로 완벽하게 버전 관리된다. 그런데 왜 DB 스키마는 Git만으로 부족하고, 별도 마이그레이션 툴이 필요한가?

---

## 가설 및 추론

- 코드와 달리 스키마는 파일이 아니라 **DB 서버의 상태**다
- Git은 파일의 변경 이력을 추적하지만, DB 서버에 실제로 적용됐는지는 추적하지 못한다
- 따라서 "어떤 SQL이 어느 환경에 적용됐는지"를 별도로 관리해야 한다

---

## 검증 및 팩트

### Schema Drift 발생 메커니즘

```
초기 상태: 모든 환경이 동일
  dev   DB: users(id, name)
  stage DB: users(id, name)
  prod  DB: users(id, name)

개발자 A → dev에서 직접 ALTER TABLE users ADD COLUMN email
개발자 B → stage에서 다른 컬럼 추가
배포팀   → SQL 공유 안 됨, prod는 그대로

6개월 후:
  dev   DB: users(id, name, email, phone, address)
  stage DB: users(id, name, phone)
  prod  DB: users(id, name, email)     ← 누가 언제 적용했는지 불명
```

이 상태를 **Schema Drift(스키마 드리프트)** 라고 한다.

### 수동 SQL 방식의 3가지 실패

```
1. 멱등성 부재
   "이거 실행했어요?" "저도 했는데?" → 중복 실행 → 오류

2. 추적 불가
   6개월 전 SQL이 prod에 반영됐는지 확인 방법 없음

3. 순서 보장 불가
   V2 (컬럼 추가) 이전에 V3 (해당 컬럼 인덱스 추가) 실행 → 오류
```

### 마이그레이션 툴이 해결하는 방식

```
flyway_schema_history (DB 내부에 생성)
  version | executed_at         | success
  --------|---------------------|--------
  1       | 2026-01-01 09:00:00 | true
  2       | 2026-02-15 14:30:00 | true

→ "이 환경에 V1, V2가 적용됐다"는 사실이 DB 자체에 기록됨
→ 코드 배포 시 자동으로 미적용 마이그레이션만 순서대로 실행
→ 체크섬으로 기존 파일 변조 감지
```

### 코드-스키마 배포 타이밍 문제

```
신규 컬럼 추가 시나리오:
  1. 스키마 먼저 변경 → 앱 배포 전까지 신규 컬럼은 null
     → 구버전 앱이 null을 읽으면 NPE 가능
  2. 앱 먼저 배포 → 스키마 변경 전 신규 컬럼 접근
     → ColumnNotFoundException 즉시 오류

  해결: Expand-Contract (팽창-수축) 패턴
  1단계) Nullable 컬럼 추가 (스키마 선배포)
  2단계) 신버전 앱 배포 (새 컬럼 활용)
  3단계) NOT NULL 제약 추가 (정리)
```

---

## 결론

DB 스키마는 파일이 아닌 **서버의 상태**이기 때문에 Git만으로는 "어느 환경에 무엇이 적용됐는가"를 추적할 수 없다. 마이그레이션 툴은 변경 이력을 DB 자체에 기록함으로써 코드 배포와 스키마 변경을 동기화한다. 이것은 [[본질-코드와 상태의 비대칭성 (Code-State Asymmetry)]]의 직접적인 해결 전략이다.
