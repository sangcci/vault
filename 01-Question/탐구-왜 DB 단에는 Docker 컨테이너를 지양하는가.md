## 핵심 질문 (Core Question)

> DB는 왜 stateless 서비스와 달리 컨테이너화를 지양하는가?

---

## 가설 및 추론 (Hypothesis)

- Docker는 쉽게 띄우고 내리는 것이 핵심인데, DB는 반대로 절대 갑자기 꺼지면 안 됨
- 컨테이너가 예기치 않게 종료될 때 DB가 정상 종료 루틴을 밟지 못하면 데이터가 손상될 것
- Docker의 스토리지 레이어가 DB의 I/O 패턴에 맞지 않을 것

---

## 검증 및 팩트 (Verification)

**Stateful vs Stateless 패러다임 충돌**
```
Docker 설계 철학    컨테이너는 언제든 생성·삭제 가능한 일회용 단위
                    상태는 외부(DB, 오브젝트 스토리지)에 저장

DB 요구사항        데이터 영속성이 최우선
                    정상 종료 루틴(flush → fsync → 파일 닫기)이 반드시 필요
```

**컨테이너 크래시 → 데이터 손상 경로**
```
SIGTERM (정상 종료)   DB가 종료 신호를 받음
                       → flush: 메모리 버퍼를 디스크에 기록
                       → fsync: OS 캐시까지 디스크에 강제 기록
                       → 파일 닫기: 데이터 파일 무결성 보장

SIGKILL (강제 종료)   즉각 프로세스 종료
                       → 위 루틴 전부 생략
                       → 트랜잭션 처리 중이었다면 파일이 중간 상태로 남음
                       → 재시작 시 복구 불가 또는 데이터 손상
```

**컨테이너 크래시 예시 — 모두 SIGKILL 상황**
```
OOM Kill          메모리 제한 초과 → 커널이 컨테이너 프로세스 강제 종료
Docker 데몬 재시작 dockerd 업데이트/충돌 → 실행 중 컨테이너 전체 종료
Host 커널 패닉    Host OS 크래시 → WAL 미완료 상태로 남음
리소스 제한 초과  --memory 초과 순간 트랜잭션 처리 중 강제 종료
```

**스토리지 드라이버 문제**
```
Docker overlay2 레이어
    → 파일시스템 위에 파일시스템을 얹는 구조
    → DB의 고빈도 랜덤 I/O에 최적화되지 않음
    → 직접 블록 디바이스 접근보다 성능 저하 및 신뢰성 이슈
```

**이중 가상화 문제**
```
VM(가상 OS) 위에 Docker(컨테이너) 위에 DB
    → 가상화 레이어 2중 → I/O 지연 누적
    → 장애 발생 시 디버깅 포인트 증가
    → 성능 튜닝 시 어느 레이어 문제인지 파악 어려움
```

**인메모리 DB는 예외인가?**
```
캐시 전용 (휘발성, persistence 없음)
    → 데이터 파일 없음 → 크래시 시 파일 손상 없음
    → disk I/O 없음 → 스토리지 드라이버 문제 해당 없음
    → Docker로 운영해도 상대적으로 안전

Persistence 설정 시 (AOF / RDB snapshot)
    → 디스크에 쓰는 순간 위의 문제 동일하게 재등장
    → 프로덕션 persistence Redis도 Docker 비권장
```

---

## 결론 (Conclusion)

- 지양 이유의 핵심은 **"Docker는 SIGKILL을 전제로 설계됐고, DB는 SIGKILL을 견딜 수 없다"**
- 디스크 기반 RDB(MySQL, PostgreSQL 등)에서 특히 강하게 적용됨
- 인메모리 DB는 캐시 전용이라면 예외, persistence 설정 시 동일한 위험
- 개발·테스트 환경에서는 완전히 허용 — 지양은 **프로덕션 한정**
