# Allreva 프로젝트 핵심 설계 고려사항 및 기술 질문 리스트

본 문서는 Allreva(콘서트 차량 대절 서비스)의 백엔드 설계 과정에서 반드시 해결해야 할 핵심 기술 과제와 면접 예상 질문을 정리한 리스트입니다.

---

## 1. 좌석 예약 동시성 이슈 (Concurrency Control)
*   **핵심 과제:** 잔여 좌석이 적을 때 여러 사용자가 동시에 다인 예약을 시도할 경우, '초과 예약(Overbooking)'을 어떻게 방지할 것인가?
*   **기술적 질문:**
    *   데이터베이스의 **낙관적 락(Optimistic Lock)**과 **비관적 락(Pessimistic Lock)** 중 어떤 것이 이 서비스에 더 적합한가?
    *   Redis를 활용한 **분산 락(Distributed Lock)** 도입 시 성능상의 이점과 인프라 관리 비용의 트레이드오프는 무엇인가?
    *   Atomic Update(`UPDATE ... WHERE remaining_seats >= :count`)만으로 충분한가?

## 2. 상태 관리 및 히스토리 로그 (State & Audit Log)
*   **핵심 과제:** 예약 상태(`SURVEY` -> `PAID` -> `CONFIRMED`)의 무결성을 보장하고, 변경 이력을 어떻게 효율적으로 남길 것인가?
*   **기술적 질문:**
    *   상태 변경 로그를 저장할 때 **비동기(Event-Driven)** 방식을 도입하면 메인 로직의 성능에 어떤 도움을 주는가?
    *   비동기 도입 시, 메인 트랜잭션은 성공했으나 로그 저장에 실패했을 때의 데이터 정합성 문제는 어떻게 해결할 것인가?
    *   Audit Log 테이블의 인덱스 설계는 어떻게 조회 성능을 최적화할 것인가?

## 3. 모집 마감 체크 스케줄러 (Scheduler Implementation)
*   **핵심 과제:** 수천 개의 모집 게시물을 실시간에 가깝게 마감 처리하면서 서버 부하를 최소화하는 방법은?
*   **기술적 질문:**
    *   **1분 단위 폴링(Polling)** 방식과 **지연 큐(Delay Queue)** 방식 중 어떤 것이 확장성 면에서 유리한가?
    *   서버가 다중화(Scale-out)된 환경에서 동일한 스케줄러 작업이 중복 실행되지 않도록 보장하는 방법(예: **ShedLock**)은 무엇인가?

## 4. 다중 운행 일자 데이터 모델링 (ERD Design)
*   **핵심 과제:** 한 공연이 2~3일간 열릴 경우, 하나의 모집 게시물에서 여러 날짜별 차량을 어떻게 관리할 것인가?
*   **기술적 질문:**
    *   `Recruitment`와 `Schedule` 테이블을 **1:N 관계**로 분리할 때, 좌석 정보와 가격 정보는 어느 테이블에 위치하는 것이 정규화 관점에서 옳은가?
    *   사용자가 여러 날짜를 동시에 예약할 수 있게 할 것인가, 아니면 일자별로 별도 예약을 생성하게 할 것인가?

## 5. 주최자 인증 및 인가 (Auth & Authorization)
*   **핵심 과제:** 일반 사용자와 주최자를 어떻게 구분하며, 주최자가 본인의 게시물만 수정할 수 있도록 보장하는 방법은?
*   **기술적 질문:**
    *   **RBAC(Role Based Access Control)**와 **ABAC(Attribute Based Access Control)** 중 주최자 권한 관리에 더 적합한 모델은?
    *   Security Context에서 주최자의 ID를 검증하는 로직을 AOP나 Interceptor로 공통화하여 비즈니스 로직과 분리할 수 있는가?

## 6. 환불 절차 및 예외 처리 (Refund Process)
*   **핵심 과제:** 모집 취소(인원 미달) 시 대량 환불과 개별 취소 시 환불을 어떻게 구분하여 처리할 것인가?
*   **기술적 질문:**
    *   외부 결제 API 없이 '무통장 입금'으로 운영될 때, 관리자의 '환불 완료' 승인 프로세스를 상태도로 표현한다면?
    *   환불 과정에서 발생할 수 있는 '중복 환불' 방지를 위한 **멱등성(Idempotency)** 보장 전략은?

## 7. 알림 전송 전략 (Notification System)
*   **핵심 과제:** 사용자에게 중요한 상태 변화를 어느 시점에, 어떤 수단(SMS, Push, 알림톡)으로 보낼 것인가?
*   **기술적 질문:**
    *   알림 발송 로직을 메인 비즈니스 로직과 분리하기 위해 **Spring Event**나 **Message Queue**를 사용하는 이유는 무엇인가?
    *   대량의 알림 발송 시 시스템 성능 저하를 방지하기 위한 **Batch 발송** 또는 **Rate Limiting** 전략이 있는가?
