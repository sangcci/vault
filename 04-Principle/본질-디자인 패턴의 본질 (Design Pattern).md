**핵심 질문**: "패턴이란 무엇이며, 기술을 어떻게 정의하는가?"

## 한 문장 정의

> (사전적) 반복적으로 발생하는 설계 문제에 대한 검증된 해결책에 붙인 이름.
> (이해용) 좋은 코드에서 발견된 구조에 붙인 어휘 — 레시피가 아니라 단어.

---

## 사용 예시

1. "여기 Template Method 쓰자" → 뼈대 고정 / 단계 위임 / 상속 구조를 한 단어로 전달
2. "Kafka는 Mediator 패턴이야" → 중앙화된 통신, N:N 의존 제거를 한 문장으로 설명
3. FileVisitor를 보고 "Visitor네" → 코드 읽는 속도가 빨라짐

---

## 패턴은 발견된 것이다

```
잘못된 이해:  패턴 정의 → 코드에 적용
실제 역사:    좋은 코드 → 반복 패턴 발견 → 이름 붙임
```

GoF 저자들:
> "우리는 패턴을 만든 게 아니다. 이미 잘 동작하는 코드에서
>  반복적으로 나타나는 구조를 발견해서 이름을 붙인 것이다."

---

## 기술은 패턴으로 정의되지 않는다

기술은 **해결하려는 문제 + 감수하는 트레이드오프**로 정의된다.
패턴은 그 중 구조적 선택의 일부를 설명할 뿐이다.

```
"Kafka는 Mediator 패턴이다"
  → 맞지만, 분산 로그 / 파티셔닝 / 컨슈머 그룹 /
    오프셋 관리 / 내구성 보장이 전부 날아감

"JdbcTemplate은 Template Method다"
  → 맞지만, 커넥션 풀 관리 / 예외 변환 계층 /
    트랜잭션 통합이 전부 날아감
```

실제 기술은 단일 패턴이 아닌 **패턴의 조합**으로 구현된다.

```
walkFileTree + FileVisitor
  ├── Template Method  (DFS 뼈대 고정, 단계 위임)
  └── Strategy         (FileVisitor 통째로 교체)
      ※ 이름은 Visitor지만 Double Dispatch 없음
```

---

## 트레이드오프

| 얻는 것 | 잃는 것 |
|--------|--------|
| 설계 의도를 빠르게 전달 | 기술의 세부 특성이 가려짐 |
| 팀 내 공통 어휘 형성 | 패턴 이름에 의존한 얕은 이해 |
| 코드 구조 인식 속도 향상 | 억지 적용 시 오버엔지니어링 |

---

## 왜 사라지지 않는가

인간이 복잡한 구조를 다룰 때 **압축된 어휘**가 필요하기 때문이다.
언어가 바뀌고 프레임워크가 바뀌어도, 설계에서 반복되는 구조의 종류는
본질적으로 동일하다 — 위임, 캡슐화, 분리, 조합.

---

## 다른 모습들

- **건축**: 크리스토퍼 알렉산더의 패턴 언어 — GoF의 원형
- **언어**: 관용어 ("가시방석" → 상황 전체를 한 단어로)
- **수학**: 정리(theorem) — 반복 증명 대신 이름으로 인용
