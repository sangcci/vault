# Principle: 조합 (Composition)
**핵심 질문**: "필요한 부품을 조립해서 새로운 기능을 만들 수 있는가?"

## One-liner Definition
> (사전적) 기존 클래스들을 부품으로 사용하여 새로운 클래스를 구성하는 설계 기법.
> (이해용) **"레고 블록 조립과 같다. 자동차를 만들 때 바퀴를 상속받는 게 아니라, 바퀴 블록을 가져와서 끼우는 것."**

## Usage Examples (문장 3개)
1. "전략 패턴을 활용해 이동 로직을 객체로 분리하고, 로봇 클래스에 **조합**하여 동적인 행동 변경이 가능하게 했다."
2. "상속 대신 **조합**을 사용하여 깨지기 쉬운 기반 클래스 문제(Fragile Base Class)를 회피했다."
3. "리액트에서 컴포넌트 **조합**을 통해 작고 단순한 UI 조각들을 모아 복잡한 대시보드를 구축했다."

## Recurring Core Problem
소프트웨어 설계에서 재사용성을 높이려 할 때 상속을 사용하면 계층이 깊어지고 결합도가 지나치게 높아집니다. "상속 계층의 하단에 있는 클래스가 상위의 변경에 의해 예상치 못하게 깨지는 현상"을 어떻게 해결하고 더 유연한 재사용 구조를 만들 것인가가 핵심입니다.

## Why It Doesn't Disappear
비즈니스 요구사항은 실행 시점(Runtime)에도 유연하게 변해야 하기 때문입니다. 컴파일 타임에 결정되는 상속보다, 런타임에 부품을 갈아 끼울 수 있는 조합은 현대 소프트웨어 아키텍처의 필수적인 유연성을 제공합니다.

## Trade-offs
- **유연성 vs 복잡성**: 조합은 낮은 결합도를 제공하지만, 부품 객체들에 요청을 전달하는 위임(Delegation) 코드가 늘어나고 관리해야 할 객체 수가 많아집니다.
- **성능 vs 재사용**: 아주 미세한 수준에서 객체 생성 비용과 위임 호출로 인한 오버헤드가 발생할 수 있지만, 대부분의 비즈니스 애플리케이션에서는 무시할 만한 수준입니다.

## Appears As
- **Has-A Relationship**: 소유 또는 포함 관계
- **Strategy Pattern**: 알고리즘 부품화
- **Dependency Injection**: 외부에서 부품을 주입받음
