**핵심 질문**: "시스템 내부에서 무슨 일이 일어나는지 외부에서 어떻게 알 수 있는가?"

## 한 문장 정의

> (사전적) 시스템의 외부 출력만으로 내부 상태를 추론할 수 있는 정도.
> (이해용) 비행기의 블랙박스처럼, 시스템이 죽은 뒤에도 "그때 무슨 일이 있었는지" 재구성할 수 있는 능력.

## 사용 예시

```
옵저버빌리티의 3대 기둥
│
├── Logs    → "무슨 이벤트가 있었는가" (이산적 기록)
│              예) "2024-01-15 결제 실패, userId=A"
│
├── Metrics → "얼마나 자주/빠르게 일어나는가" (수치 집계)
│              예) 초당 에러율 0.3%, p99 응답시간 450ms
│
└── Traces  → "어디서 얼마나 걸렸는가" (분산 경로 추적)
               예) API → OrderService → PaymentService → DB: 320ms
```

1. 장애 감지: Metrics 이상 → Alert 발동
2. 원인 추적: Logs에서 에러 컨텍스트 확인
3. 병목 분석: Traces에서 어느 구간이 느린지 파악

## 트레이드오프

| 많이 수집할수록 | 적게 수집할수록 |
|---|---|
| 정확한 원인 파악 가능 | 저장·처리 비용 절감 |
| 저장/처리 비용 증가 | 장애 시 원인 불명 |
| 개인정보 유출 리스크 | 디버깅 불가 |

## 왜 사라지지 않는가

- 소프트웨어는 본질적으로 블랙박스. 프로세스 내부를 런타임에 직접 들여다볼 수 없음
- 분산 시스템이 복잡해질수록 단일 노드 디버깅만으로는 원인 규명 불가
- 예측 가능한 모든 실패를 사전에 코드로 막을 수 없으므로, **사후 관찰 능력**이 필수

## 다른 모습들

- **OS**: `/var/log/syslog`, `dmesg`, `top`, `htop`
- **DB**: Slow Query Log, `EXPLAIN ANALYZE`, Wait Events
- **네트워크**: `tcpdump`, 패킷 캡처, 라우팅 테이블
- **JVM**: GC 로그, Thread Dump, Heap Dump
