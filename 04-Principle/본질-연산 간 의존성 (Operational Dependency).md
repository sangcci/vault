**핵심 질문**: "왜 연산에는 순서가 존재하는가?"

## 한 문장 정의

> (사전적) 한 연산의 결과가 다음 연산의 입력 조건이 될 때, 두 연산 사이에 발생하는 선후 제약.
> (이해용) 아직 존재하지 않는 값을 참조할 수 없다 — 의존하는 값이 먼저 결정되어야 다음 연산이 가능하다.

---

## 사용 예시

1. **SQL 실행 순서**: GROUP BY가 완료되어야 HAVING이 집계값을 참조할 수 있음
2. **컴파일러 패스**: 심볼 테이블이 구축되어야 타입 검사가 가능
3. **빌드 의존성 그래프**: A 모듈이 빌드되어야 B 모듈이 링크 가능
4. **함수 호출 스택**: 호출자가 인자를 평가한 후에야 피호출자가 실행됨

```
[SQL]                        [빌드]
FROM → WHERE → GROUP BY      A.class → B.class → C.class
         ↑ 그룹 없으면              ↑ A 없으면
         HAVING 불가               B 링크 불가

공통 구조: 의존 대상이 먼저 존재해야 다음 연산 가능
```

---

## 트레이드오프

- **얻는 것**: 연산 결과의 예측 가능성. 입력이 확정된 후 처리하므로 일관성 보장
- **잃는 것**: 병렬 실행 불가. 의존 관계가 있는 연산은 순차 처리 강제

---

## 왜 사라지지 않는가

상태(state)를 변이하는 패러다임에서는 필연적으로 발생한다.

```
명령형/절차형: 이전 상태에 의존하는 다음 연산 → 순서 강제
객체지향:      객체 상태 변이 → 메서드 호출 순서에 의존성 발생
함수형:        순수 함수 + 불변 값 → 상태 변이 없음 → 의존성 제거 가능
               (단, 완전한 함수형도 입출력 순서는 존재)
```

> 상태가 존재하는 한, 그 상태를 만드는 연산과 소비하는 연산 사이의 선후관계는 사라지지 않는다.

---

## 다른 모습들

- **OS**: 프로세스 fork → exec 순서. 부모가 존재해야 자식 생성 가능
- **DB 트랜잭션**: BEGIN → 연산 → COMMIT. 트랜잭션 시작 전 연산 불가
- **네트워크**: TCP 3-way handshake. SYN → SYN-ACK → ACK 순서 불변
- **React**: 상태 초기화 → 렌더링 → 이펙트. 렌더 전 DOM 참조 불가
