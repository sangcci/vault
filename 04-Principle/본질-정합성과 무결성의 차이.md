**핵심 질문**: "데이터가 '맞다'는 말은 값 자체가 맞다는 뜻인가, 여러 상태가 서로 모순되지 않는다는 뜻인가?"

## 한 문장 정의

> (사전적) 무결성은 데이터 값이 정해진 규칙과 제약을 만족하는 성질이고, 정합성은 여러 데이터 상태가 시간과 관계 속에서 서로 모순되지 않는 성질이다.
> (이해용) 무결성은 쓰레기 데이터가 들어오지 못하게 막는 방어벽이고, 정합성은 여러 데이터가 서로 다른 말을 하지 않게 맞추는 흐름 관리다.

```text
무결성 Integrity
입력값 / 제약 / 도메인 규칙
        |
        v
Garbage In 방지

정합성 Consistency
동시 요청 / 캐시 / 복제 / 장애
        |
        v
서로 다른 상태의 모순 방지
```

## 사용 예시

1. `passenger_num >= 1` 제약은 무결성 문제다.
2. 동시에 좌석을 예약했는데 총 좌석보다 많이 예약되면 정합성 문제다.
3. DB는 새 값인데 cache는 옛 값을 가리키면 정합성 문제다.
4. 존재하지 않는 enum 값이 저장되면 무결성 문제다.
5. 감사 로그에는 취소 기록이 있는데 현재 상태는 승인 상태라면 정합성 문제다.

## 트레이드오프

- 무결성은 schema, constraint, validation으로 비교적 명확히 방어할 수 있다.
- 정합성은 트랜잭션, lock, version, cache invalidation, event ordering까지 함께 봐야 한다.
- 정합성을 강하게 잡으면 latency와 throughput 비용이 커질 수 있다.
- 정합성을 느슨하게 잡으면 eventual consistency를 설명하고 보정할 장치가 필요하다.

## 왜 사라지지 않는가

데이터는 한 번에 한 곳에서만 바뀌지 않는다. 웹 요청, batch, cache, message queue, DB replica, 외부 API가 함께 움직인다. 그래서 “값 자체가 맞는가”와 “상태들이 서로 맞는가”는 어떤 기술에서도 분리해서 봐야 한다.

## 다른 모습들

- DB: constraint는 무결성, transaction isolation은 정합성에 가깝다.
- Cache: key/value 형식은 무결성, DB-cache stale 문제는 정합성이다.
- Event: event schema는 무결성, event ordering과 재처리는 정합성이다.
- API: request validation은 무결성, 중복 요청 처리와 idempotency는 정합성이다.
