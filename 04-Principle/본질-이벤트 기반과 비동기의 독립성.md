**핵심 질문**: "이벤트 기반 아키텍처와 비동기 처리는 같은 것인가, 다른 것인가?"

## 한 문장 정의

> (사전적) 이벤트 기반 아키텍처는 **결합도를 낮추기 위한 설계 방식**이고, 비동기 처리는 **성능을 높이기 위한 실행 방식**이다. 둘은 독립된 개념이며, 함께 쓰면 시너지를 낸다.
> (이해용) 이벤트 기반은 "누가 듣는지 몰라도 방송하는 것", 비동기는 "답장을 기다리지 않고 다음 일을 하는 것". 방송을 비동기로 할 수도, 동기로 할 수도 있다.

---

## 사용 예시

1. "알림 발송 로직을 **이벤트 기반**으로 분리하여, 주문 서비스가 알림 서비스를 직접 의존하지 않게 만들었다."
2. "이미지 리사이징을 **비동기**로 처리하여, 업로드 API의 응답 시간을 줄였다."
3. "주문 완료 이벤트를 발행하고, 리스너에서 **비동기**로 알림과 로깅을 동시에 처리했다." (두 개념의 시너지)

---

## 트레이드오프

| | 이벤트 기반 | 비동기 |
| :--- | :--- | :--- |
| **목적** | 결합도 감소 | 성능 향상 |
| **관점** | 코드 구조 (설계) | 실행 흐름 (런타임) |
| **비용** | 흐름 추적 어려움, 디버깅 복잡 | 에러 핸들링 복잡, 트랜잭션 경계 모호 |

---

## 왜 사라지지 않는가

시스템이 커질수록 모듈 간 결합도는 필연적으로 증가하고, 동기 처리만으로는 응답 시간 요구를 만족시킬 수 없다. 이 두 문제는 독립적으로 발생하므로, 각각의 해결책도 독립적으로 존재할 수밖에 없다.

## 다른 모습들

### 이벤트 기반 (동기)

```text
OrderService                 EventPublisher              NotificationListener
     │                            │                            │
     │── publish(OrderEvent) ────>│                            │
     │                            │── notify(OrderEvent) ─────>│
     │                            │                            │── 알림 발송
     │                            │                            │   (동기 처리)
     │                            │<───────────────────────────│
     │<───────────────────────────│
     │  (리스너 완료까지 대기)
```

- 결합도는 낮아짐: OrderService는 NotificationListener를 모름.
- 하지만 **동기적으로 실행**되어 리스너 처리 시간만큼 대기.

### 비동기 (이벤트 없이)

```text
OrderService                              AsyncImageResizer
     │                                          │
     │── CompletableFuture.runAsync() ─────────>│
     │   (바로 반환, 대기하지 않음)                 │── 리사이징 수행
     │── 다음 로직 계속 ──>                        │
```

- 성능은 향상됨: 호출자가 대기하지 않음.
- 하지만 **직접 의존**: OrderService가 ImageResizer를 알고 있음.

### 시너지: 이벤트 기반 + 비동기

```text
OrderService         EventPublisher        @Async NotificationListener
     │                    │                          │
     │── publish() ──────>│                          │
     │<──(즉시 반환)──────│── 별도 스레드로 실행 ────>│
     │                    │                          │── 알림 발송
     │── 다음 로직 ──>     │                          │
```

- 결합도 감소 (이벤트) + 응답 시간 단축 (비동기) 동시 달성.
- 트랜잭션 범위에서 벗어나도 되는 로직(로깅, 알림)에 적합.

### 주의: ApplicationEventPublisher는 큐가 아니다

Spring의 `ApplicationEventPublisher`는 내부에 자료구조를 갖지 않는다. 위에서 아래로 실행되는 메서드 호출에 불과하다. 따라서 동기 모드에서는 첫 번째 리스너 처리가 끝나야 두 번째가 실행된다. 진정한 비동기 메시지 큐가 필요하면 MessageQueue(Kafka, RabbitMQ 등)를 별도로 도입해야 한다.
