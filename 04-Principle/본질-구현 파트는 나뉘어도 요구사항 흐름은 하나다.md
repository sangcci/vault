**핵심 질문**: "Front와 Back 구현이 나뉘어도 왜 요구사항은 전체 흐름으로 이해해야 하는가?"

## 한 문장 정의

> (사전적) 요구사항 흐름은 화면·사용자 행동·API·DB·예외 정책이 한 작업 안에서 이어지는 전체 경로다.
> (이해용) Front와 Back은 작업 구역이 다를 뿐, 사용자가 겪는 일은 하나의 흐름으로 움직인다.

> "Front와 Back이 구현 파트에서는 나눠져있지만, 요구사항을 느끼는 것은 다 동일하게 흐름을 전체적으로 알아야 된다고 생각." — 사용자 메모, 2026-06-10

---

## 사용 예시

1. 상품 목록 UI에서 정렬 기준을 모르면 Back API의 쿼리 조건과 인덱스 필요성을 질문하기 어렵다.
2. 무한 스크롤인지 페이지 방식인지 모르면 Front 상태 관리와 Back 페이징 응답 구조를 함께 결정할 수 없다.
3. 상품상세 화면에서 어떤 정보가 한 번에 필요한지 모르면 API를 쪼갤지 묶을지 판단하기 어렵다.

```text
사용자 행동
└─ 화면 흐름
   └─ Front 상태 / 이벤트
      └─ API 요청 / 응답
         └─ Back 정책 / 검증
            └─ DB 조회 / 저장
               └─ 사용자에게 다시 보이는 결과
```

---

## 트레이드오프

- 맡은 파트만 보면 구현 속도는 빠를 수 있지만, 흐름 밖의 질문을 놓쳐 재작업이 생긴다.
- 전체 흐름을 보면 초반 이해 비용은 늘지만, 누락된 정책과 협의 지점을 빨리 발견한다.
- Front/Back 경계를 존중하되, 요구사항 질문은 경계를 넘어야 한다.

---

## 왜 사라지지 않는가

사용자는 Front와 Back을 나눠서 경험하지 않는다. 사용자는 화면에서 행동하고 결과를 본다. 그 사이에 Front 상태, API, Back 정책, DB가 연결되어 있기 때문에 구현 파트가 나뉘어도 요구사항 흐름은 하나로 남는다.

```pseudo
function understandRequirementFlow(feature) {
    userAction = readUserAction(feature.ui)
    frontNeed = inferFrontState(userAction)
    apiNeed = inferApiContract(frontNeed)
    backPolicy = inferBusinessRule(apiNeed)
    dataNeed = inferDataAccess(backPolicy)
    return connect(userAction, frontNeed, apiNeed, backPolicy, dataNeed)
}
```

---

## 다른 모습들

- UI 요구사항 분석: 화면에 보이는 항목이 API와 DB 조건으로 이어진다.
- API 설계: 응답 단위는 Front 사용 흐름과 Back 변경 가능성을 같이 본다.
- 코드 리뷰: 내 파트 코드만 보지 않고 호출 흐름과 영향 범위를 같이 본다.
- 협업: 내가 Front를 몰라도 된다는 태도는 질문 부족으로 이어질 수 있다.
