**핵심 질문**: "왜 어떤 시스템은 이전 요청을 기억하지 못하는가?"

## 한 문장 정의

> (사전적) 각 요청이 이전 요청과 완전히 독립적으로 처리되며, 시스템이 클라이언트의 상태를 보존하지 않는 설계 원칙.
> (이해용) 매 요청마다 처음 만나는 사람처럼 대하는 것 — 기억하려면 요청자가 직접 모든 맥락을 가져와야 한다.

## 사용 예시

1. **HTTP**: 각 요청은 독립적. 쿠키·세션·JWT로 클라이언트가 직접 상태를 주입.
2. **LLM**: 모델 자체는 기억 없음. 매 요청마다 전체 대화 기록을 컨텍스트에 담아 전송.
3. **Serverless (Lambda)**: 함수 호출 간 상태 없음. 상태는 외부 DB에 위임.
4. **REST API**: 서버가 클라이언트 상태를 저장하지 않음. 확장성의 근거.

## 트레이드오프

- **장점**: 수평 확장이 쉬움 — 어떤 서버가 요청을 받아도 동일하게 처리 가능
- **단점**: 매 요청마다 상태를 다시 전송해야 하므로 페이로드가 커짐 (LLM의 Context Window 압박이 대표적)

## 왜 사라지지 않는가

상태를 서버에 두면 특정 서버에 종속되어 확장이 어려워진다. 시스템이 커질수록 상태를 클라이언트나 외부 저장소로 밀어내는 것이 불가피하다. 분산 시스템의 본질적 한계이기 때문에 기술이 바뀌어도 이 패턴은 반복된다.

## 다른 모습들

- **OS**: 인터럽트 핸들러는 호출될 때마다 레지스터 상태를 스택에 저장/복원 — 핸들러 자체는 무상태
- **DB Connection Pool**: 커넥션은 상태를 갖지만, 쿼리 단위로 보면 각 쿼리는 독립적
- **LLM + 하네스**: 무상태 모델 위에 대화 기록 주입으로 "유상태처럼 보이게" 만드는 구조

```
[ 유상태처럼 보이는 LLM 대화 ]

요청 1: [시스템 프롬프트 + "안녕"]          → "안녕하세요"
요청 2: [시스템 프롬프트 + "안녕" + "안녕하세요" + "이름이 뭐야?"]  → "저는 Claude입니다"
         ↑
         LLM은 매번 처음 보는 텍스트 덩어리를 받을 뿐
```
