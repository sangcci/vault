**핵심 질문**: "공격자의 비용이 방어자의 비용보다 항상 높게 만드는 것이 가능한가?"

## 한 문장 정의

> (사전적) 시스템이 인가된 주체에게만 자원에 접근을 허용하고, 인가되지 않은 접근·변조·유출을 방지하는 성질.
> (이해용) **"자물쇠를 여는 비용이 문을 부수는 비용보다 항상 커야 한다."** 완전한 보안은 없고, 공격 대비 방어 비용 비율을 조절하는 것이 핵심.

## 사용 예시

1. "BCrypt의 Cost Factor를 높여 해시 연산에 200ms가 걸리게 설정하면, 브루트포스 공격자는 초당 5번밖에 시도 못 한다."
2. "TLS 핸드셰이크에서 비대칭키(RSA)로 키를 교환하고, 실제 데이터는 대칭키(AES)로 암호화하여 속도와 보안을 동시에 확보한다."
3. "Access Token 유효기간을 15분으로 짧게 유지해 탈취 시 피해를 시간적으로 제한한다."

## 트레이드오프

- **보안 강도 vs 성능**: BCrypt Cost Factor 상승 → 해시 시간 지수 증가 → 정상 사용자 경험 저하
- **보안 강도 vs 편의성**: Token 수명 단축 → 재인증 빈도 증가 → UX 악화
- **방어 깊이 vs 복잡도**: 계층적 방어(Defense in Depth)는 보안을 높이지만 운영 복잡도를 증가시킨다

## 왜 사라지지 않는가

데이터는 비즈니스 자산이고 공격자의 동기(금전·정치·경쟁)는 사라지지 않기 때문이다. 컴퓨팅 성능이 증가할수록 과거의 방어 수준은 무력화되므로, 보안은 한 번 설정하고 끝나는 것이 아닌 지속적인 갱신이 필요하다.

## 다른 모습들

- **저장 보안**: AES-256 암호화 — 디스크·DB에 저장된 민감 데이터 보호
- **전송 보안**: TLS 1.3 — 네트워크 구간 도청·변조 방지 ([[개념-PKI (Public Key Infrastructure)]])
- **인증 보안**: BCrypt / Argon2 — 패스워드를 느린 해시로 저장해 유출 시 역추적 비용 극대화
- **접근 제어**: RBAC / ReBAC — 인가된 주체만 자원에 접근
