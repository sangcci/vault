**핵심 질문**: "왜 코드는 git으로 롤백할 수 있는데, DB 스키마는 롤백이 어려운가?"

## 한 문장 정의

> (사전적) 코드는 배포할 때마다 이전 버전을 대체하지만, 상태(데이터·스키마)는 이전 상태 위에 누적된다는 수명주기의 근본적 차이.
> (이해용) **"코드는 교체되는 부품이고, 상태는 쌓이는 퇴적층이다."**

## 사용 예시

1. "git revert로 코드를 되돌려도 이미 실행된 `ALTER TABLE`은 되돌아오지 않는다. 이 **비대칭성** 때문에 Flyway 같은 마이그레이션 툴이 필요하다."
2. "Terraform은 `.tf` 파일(코드)과 `terraform.tfstate`(상태)를 별도로 관리한다. 코드를 지워도 클라우드 리소스(상태)는 남기 때문이다."
3. "이벤트 소싱에서 이벤트(상태)는 절대 삭제하지 않는다. 코드 버그로 잘못 처리했어도 이벤트 자체는 진실로 남는다."

## 비대칭 구조도

```
코드의 수명주기
  v1.0 ──→ v1.1 ──→ v1.2
             ↑ 배포 시 이전 버전 교체됨 (대체)

상태의 수명주기
  schema_v1
     + ALTER TABLE (누적)
  schema_v2
     + ALTER TABLE (누적)
  schema_v3
             ↑ 이전 상태는 현재 상태 안에 녹아 있음 (누적)

결과:
  코드 롤백 → git revert → 즉시 가능
  상태 롤백 → 역방향 SQL 작성 → 데이터 손실 위험 + 복잡
```

## 트레이드오프

- **얻는 것**: 비대칭성을 인정하고 마이그레이션을 코드처럼 관리하면 → 배포 신뢰성, 환경 간 일관성 확보
- **치르는 비용**: 스키마 변경은 항상 전진(forward-only)해야 함. 롤백 대신 fix-forward 전략이 강제됨
- **숨은 비용**: 코드와 스키마의 배포 타이밍을 맞춰야 함 → Expand-Contract 패턴 같은 다단계 배포가 필요해짐

## 왜 사라지지 않는가

데이터는 비즈니스 자산이기 때문이다. 코드는 재실행하면 다시 만들 수 있지만, 사용자가 입력한 데이터는 한 번 사라지면 복구 불가능하다. 이 비대칭성은 "데이터를 영구 보존해야 한다"는 영속성의 요구가 존재하는 한 절대 사라지지 않는다.

## 다른 모습들

- **DB**: 스키마 마이그레이션 (Flyway, Liquibase) — 상태 변경 이력을 코드처럼 버전 관리
- **인프라**: Terraform state 파일 — 선언된 코드(`.tf`)와 실제 상태(`.tfstate`) 분리 관리
- **이벤트 소싱**: 이벤트 로그는 append-only — 과거 이벤트 수정 불가, 새 이벤트로만 보정
- **캐시**: 코드 배포 후 캐시 무효화 — 캐시(상태)는 코드 배포와 자동 동기화되지 않음
