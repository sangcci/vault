**핵심 질문**: "왜 시스템은 논리적으로 순서가 보이는데도, 실제 저장은 전혀 다른 배치를 가질 수 있는가?"

## 한 문장 정의

> 논리적 순서(조회·정렬·탐색의 순서)와 물리적 순서(디스크·페이지·블록의 배치)는 서로 독립이므로, 시스템은 필요에 따라 둘을 분리하거나 다시 정렬할 수 있다.

## 핵심 그림

```text
논리 순서(사용자가 보는 순서)
key1 → key2 → key3 → key4

물리 순서(실제 저장 위치)
page 8 → page 2 → page 19 → page 3

조회는 논리 순서를 따르지만,
I/O는 물리 배치에 의해 흔들린다.
```

## 왜 이 원리가 필요한가

- 논리적으로는 정렬된 상태를 유지하면서, 물리적으로는 삽입/수정 비용을 낮출 수 있다.
- 필요하면 `CLUSTER`처럼 물리 배치를 다시 정렬할 수 있다.

> "With no advance preparation, the system would have to scan the entire `test1` table, row by row" — [PostgreSQL Documentation: Indexes Introduction](https://www.postgresql.org/docs/current/indexes-intro.html)
> "it might only have to walk a few levels deep into a search tree." — [PostgreSQL Documentation: Indexes Introduction](https://www.postgresql.org/docs/current/indexes-intro.html)
> "When a table is clustered, it is physically reordered based on the index information." — [PostgreSQL Documentation: CLUSTER](https://www.postgresql.org/docs/current/sql-cluster.html)
> "no attempt is made to store new or updated rows according to their index order." — [PostgreSQL Documentation: CLUSTER](https://www.postgresql.org/docs/current/sql-cluster.html)
> "the scan will still have to visit all 10000 rows" — [PostgreSQL Documentation: Using EXPLAIN](https://www.postgresql.org/docs/current/using-explain.html)
