**핵심 질문**: "왜 동일한 소스 코드가 실행되는 물리적/논리적 환경(Dev, Test, Prod)을 철저히 분리해야 하는가?"

## 구조도 (Diagram)

```text
   [ Dev Env ]        [ Test Env ]        [ Prod Env ]
  +-----------+      +-----------+      +------------+
  |  Mock DB  |      |  Test DB  |      |  Real DB   |
  +-----------+      +-----------+      +------------+
        |                  |                   |
  <----------- ( Isolation Barrier ) ----------->
        |                  |                   |
  [ Local Mac ]      [ CI Server ]      [ Cloud/Home ]
```

## 한 문장 정의 (Definition)

> (사전적) 소프트웨어 시스템의 신뢰성과 안전성을 보장하기 위해 개발, 테스트, 운영 등 서로 다른 목적을 가진 환경을 물리적 또는 논리적으로 분리하는 원칙.
> (이해용) 개발자의 실수나 특정 서비스의 장애가 실제 사용자 데이터나 다른 시스템(운영 환경)에 절대로 영향을 미치지 않도록 **장애 전파 경로를 차단(Fault Propagation Prevention)**하는 방어막을 치는 것.

## 사용 예시 (Examples)

1. **데이터베이스 분리**: 로컬 개발용 DB, 통합 테스트 전용 DB, 실제 서비스용 운영 DB를 분리하여 운영 데이터 오염 방지.
2. **네트워크 격리**: VPC(Virtual Private Cloud) 등을 활용해 외부망과 내부망을 분리하고, 각 환경 간의 직접적인 통신 차단.
3. **가상화 및 컨테이너**: 홈서버 구축 시 VM(가상 머신)을 통해 Web, WAS, DB 환경을 격리하여, 특정 서버의 침해나 장애가 물리 서버 전체로 확산되는 것을 방지.
4. **CI/CD 파이프라인**: 빌드된 아티팩트는 동일하지만, 배포되는 환경에 따라 각기 다른 설정값(API Key, DB 주소)을 주입.

## 트레이드오프 (Trade-off)

- **이득**: 운영 안정성 극대화, 데이터 무결성 보장, 독립적인 테스트 가능성 확보.
- **손실**: 환경 구축 및 관리 비용 증가(인프라 비용), 각 환경의 동기화 문제(환경 간 설정 불일치로 인한 오류 발생 가능성).

## 왜 사라지지 않는가 (Persistence)

- 기술이 발전하여 컨테이너(Docker)나 가상화 기술이 등장하더라도, "실수할 수 있는 인간"과 "데이터의 가치"는 변하지 않기 때문에 사고를 방지하기 위한 격리 원칙은 모든 아키텍처의 근간이 됨.

## 다른 모습들 (Polymorphism)

- **OS**: 프로세스 간 메모리 공간 격리(Memory Isolation).
- **Container**: 리소스 제어(Cgroups) 및 네임스페이스(Namespaces)를 통한 프로세스 수준의 격리.
- **Architecture**: 마이크로서비스(MSA) 간의 데이터베이스 독립성 보장.
