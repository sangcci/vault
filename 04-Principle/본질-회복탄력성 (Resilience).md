# Principle: 회복탄력성 (Resilience)
**핵심 질문**: "시스템이 무너지지 않는 것보다 중요한 것은 무엇인가? 장애 상황에서 어떻게 품격 있게 무너지고 빠르게 일어날 것인가?"

## One-liner Definition
> (사전적) 시스템이 장애나 비정상 상황에서도 핵심 기능을 유지하거나 빠르게 복구하여, 장애 전파를 막는 능력.
> (이해용) **"오뚝이와 같다. 장애로 잠시 쓰러질 수는 있지만, 스스로 중심을 잡고 빠르게 다시 일어서는 시스템의 복구력."**

## Usage Examples (문장 3개)
1. "외부 결제 시스템의 장애가 우리 서비스 전체로 번지지 않도록 서킷 브레이커를 도입하여 **회복탄력성**을 높였다."
2. "네트워크 오류 시 지수 백오프(Exponential Backoff) 기반의 재시도 로직을 통해 일시적 결함에 대한 **회복탄력성**을 확보했다."
3. "핵심 기능이 마비되더라도 최소한의 읽기 기능은 제공하는 대체 동작(Fallback)을 설계하여 시스템의 **회복탄력성**을 실천했다."

## Recurring Core Problem
복잡한 현대 시스템에서 장애는 피할 수 없는 상수입니다. 사소한 외부 API 지연이나 하드웨어 결함이 연쇄 장애(Cascading Failure)로 이어져 시스템 전체를 마비시키는 상황을 어떻게 격리하고, 수동 개입 없이 자동으로 시스템을 정상화할 것인가가 핵심 문제입니다.

## Why It Doesn't Disappear
분산 시스템은 수많은 의존성으로 엮여 있어 하나가 완벽해도 전체가 안전할 수 없기 때문입니다. '절대 죽지 않는 시스템'은 불가능하므로, '장애를 안고도 살아남는 시스템'을 만드는 회복탄력성은 가용성 확보의 필수 전략입니다.

## Trade-offs
- **안정성 vs 복잡성**: 서킷 브레이커, 벌크헤드, 폴백 로직 등을 추가할수록 시스템의 전체적인 코드 복잡도가 증가하고 테스트 케이스가 방대해집니다.
- **사용자 경험 vs 데이터 최신성**: 장애 시 폴백(예: 캐시된 옛날 데이터 보여주기)을 사용하면 서비스는 유지되지만, 사용자에게 일시적으로 부정확한 정보를 제공할 수 있는 위험을 감수해야 합니다.

## Appears As
- **Circuit Breaker**: 장애 전파 차단기
- **Retry & Backoff**: 지능적 재시도
- **Bulkhead Pattern**: 선박의 격벽처럼 장애 구역 격리
- **Graceful Degradation**: 점진적 기능 저하 (핵심 기능 우선 유지)
