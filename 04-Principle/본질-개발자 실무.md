**핵심 질문**: "개발자의 실무는 개발을 하는 것인가, 일을 개발로 풀어내는 것인가?"

## 한 문장 정의

> (사전적) 개발자의 실무는 주어진 일을 이해하고, 그 일을 개발이라는 수단으로 풀어내는 활동이다.
> (이해용) 코드를 많이 쓰는 사람이 아니라, 현실의 일을 코드로 해결 가능한 형태로 바꾸는 사람이 실무 개발자다.

> "우선 대 전제는, 개발자 실무가 \"일을 잘하는 것\"이고, 일을 개발로 풀어내는것이 개발자의 역할이라고 함." — 사용자 메모, 2026-06-10

---

## 사용 예시

1. UI 계획서를 보고 바로 API를 만들지 않고, 상품 중복 노출 가능성·노출 개수·정렬 기준·페이지 방식을 먼저 질문한다.
2. `ProductCategoryEntity` 같은 연결 Entity 이름을 정할 때, 팀이 어떤 개념을 상위로 보는지 먼저 확인한다.
3. 상품상세 조회 API를 만들 때, 현재 화면만 보지 않고 기획 변경 가능성과 Front/Client 협의 비용까지 같이 본다.

```text
일
└─ 요구사항 / 운영 맥락 / 제약
   └─ 개발자의 해석
      ├─ 무엇을 질문할까?
      ├─ 어떤 개념이 중심인가?
      ├─ 어떤 변경이 자주 올까?
      └─ 무엇을 얻고 무엇을 포기할까?
         └─ 코드 / API / DB 모델 / 운영 방식
```

---

## 트레이드오프

- 개발을 목표로 보면 빠르게 구현할 수 있지만, 실제 일이 무엇인지 놓칠 수 있다.
- 일을 목표로 보면 질문과 협의 시간이 늘지만, 구현 후 누락·재작업·API 버전 증가를 줄일 수 있다.
- 정답을 찾으려 하면 결정이 늦어지고, 트레이드오프를 인식하면 상황에 맞는 선택을 설명할 수 있다.

---

## 왜 사라지지 않는가

기술은 바뀌어도 일은 항상 사람·조직·시장·운영 제약 안에서 발생한다. 그래서 개발자는 특정 기술을 적용하기 전에, 지금 풀어야 하는 일이 무엇인지 먼저 해석해야 한다.

```pseudo
function practicalDevelopment(work, context) {
    problem = understand(work, context)
    questions = findMissingInformation(problem)
    options = designTechnicalOptions(problem)
    decision = compareTradeOffs(options, context)
    return implement(decision)
}
```

## 다른 모습들

- 요구사항 분석: 화면에 보이는 요소보다 숨은 정책과 예외를 질문한다.
- 코드 읽기: 코드가 어떻게 생겼는지보다 왜 그렇게 결정됐는지를 추적한다.
- 모델링: 테이블·Entity 이름보다 어떤 개념이 중심인지 먼저 본다.
- API 설계: 한 번에 크게 조회할지, 변경 가능성 기준으로 나눌지 판단한다.
- 협업: Front와 Back 구현은 나뉘어도 요구사항 흐름은 하나로 이해한다.
