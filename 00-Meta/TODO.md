### cache
- cache evict race condition 상황 - DB commit 안된 상태에서 동시에 다른 트랜잭션이 조회 요청을 하고, 미리 tx 전에 cache evict되어 DB에서 구버전 데이터를 조회해서 캐시에 저장함 -> 정합성 문제 발생. cache가 이전 버전의 데이터를 가리킴.
- TransactionAwareCacheManagerProxy를 통해 evict가 commit 후로 미뤄짐 + 실패 시 commit이 아닌 rollback되었으므로 캐시를 비우지 않음 -> spring에서 제공하는거겠지?
- 하지만 이 경우 Cache Stampede 현상 발생할 수 있음. Evict는 캐시를 단순히 비우는 작업이라서 비운 이후에 동시에 조회가 1000건이 발생하면 DB Connection 부족으로 예외를 발생시킬 수 있음. 
- write-through는 rollback을 직접 구현해야 되므로 캐시 저장소와 서버가 분리되어 있는 한 원자성 문제 발생
- @CachePut도 업데이트를 하긴 하나, commit 타이밍이 잘못되면 똑같이 Stampede 현상 발생할 수 있음. 
- Redis로 글로벌캐시를 사용해서 관리 포인트 줄이고, 노드 간 캐시 불일치를 완화할 수 있음. 하지만 DB와 캐시 사이의 dual write consistency를 막을 수는 없다.

## 모듈, 아키텍처에 관한 관점
"단순한 심리적 거부감은 지양하고 확실한 근거와 판단 기준을 세워야 한다"
- 비즈니스 로직에 대한 이해 -> 새로운 신규 입사자가 코드를 읽고 쉽게 이해할 수 있도록 + 문서의 특성을 지닌 코드 -> 문서화의 단점 - 관리하기 어려움. 코드 변경 시 문서까지 반영하기 매우 힘듬. 버전 관리가 한 번 망가지면 계속 망가짐
- 아키텍처 트래이드 오프 - 단순한 CRUD성 프로젝트 + 유지보수할 일이 거의 없음 -> 전형적인 layered architecture 선택
- 현대 어플리케이션은 매우 복잡 + 지속적인 요구사항 반영 및 수정, develop -> service 코드를 보면 비즈니스 로직을 담았다기 보단 구현에 가까운 느낌 -> 비즈니스를 발전시키는데 상당한 걸림돌. -> 좀 더 추상화의 계층을 도입할 필요가 있다고 느낌
- 관습적인 layered architecture 지양 -> 각 layer가 어떤 역할을 하는지 정도는 알고 있었으나 더 나은 아키텍처를 위한 고민은 안해보거나 수동적인 스탠스였음
- "처음부터 쪼개지 말고, 임계점에서 레고 블록처럼 분리하라"
- "멀티모듈도 모든 문제는 해결하지 못한다"
- "굳이 domain을 순수하게 유지할 필요는 없다. 실리를 위해 spring core, tx, context 같은 기본 프레임워크의 bean 주입 및 생명주기 관리 기능은 필요하다고 본다" ""프레임워크 종속성을 제로(0)로 만들겠다"는 이상주의에 갇혀 수많은 보일러플레이트 코드를 양산하는 것보다, **스프링의 은혜를 가볍게 입으면서 비즈니스 요구사항을 명확하게 코드로 드러내는 것**"
- 우선, 아키텍처를 설계하기 전 "소프트웨어가 왜 생겨났을지"에 대한 고민부터 출발해야 한다. 결국 우리가 만드는 소프트웨어는 현실 세계에서 어떤 병목이 있는지 등의 문제점을 해결하기 위해 만든다. 즉, 비즈니스 업무를 자동화하는 등의 효율성을 살려 이용자에게 편의, 안전을 제공하는 것이다. 이유가 명확하면 업무는 그 이유에 따라서 중요해질 수 밖에 없는법. 중요성을 깨닫자.
- 비즈니스 요구사항은 개발의 용어가 포함될 수 있다. 즉, 비즈니스는 기획자, 디자이너, 개발자, 도메인 전문가, 현업자 등 여러 인원들이 공통으로 사용하는 유비쿼터스 언어로 구성되는데, 꼭 그럴 필요 없이 개발자끼리의 언어도 유비쿼터스 언어가 될 수 있다. 가령 응답시간 개선이라던지 등
- 도메인에 따라 아키텍처를 다르게 구성하는게 좋다. 업무 로직이 간단하고, 유지보수 주기가 매우 긴 경우에는 layered architecture로 충분하다고 생각한다. 심지어 그냥 JPA에 Entity 바로 연결해서 transactional script로 구성해도 된다고 생각한다. 그만큼 업무 규칙이 간단하기 때문에 인간의 인지 능력 안에서 업무 규칙을 다 외울 수 있고 그만큼 코드 이해가 빠를 수 있다.
- 하지만 도메인이 복잡하거나 도메인을 완벽하게 분석하지 못했다면 지금의 아키텍처가 크게 도움될 수 있을거라고 생각한다. 비즈니스 usecase를 최대한 직관적으로 읽히도록, 그래서 문서를 보지 않고도 어느정도 usecase의 흐름을 이해할 수 있도록 구성되어 있다. 그리고 db나 다른 의존성들을 모듈로 분리해놓음으로써 기술 교체 확장성까지 열어두었다.

core 모듈
- core 모듈은 다른 모듈을 몰라야 한다. 즉, api, batch 모듈이나 support 모듈(db, cache, client 등)을 몰라야 한다. 의존성의 방향을 항상 일정하게 만들자.
- application - domain 사이 새로운 레이어 implement를 도입한다.
	- 해당 레이어는 트랜잭션 범위 제어 및 비즈니스 업무 1가지를 맡게 된다.
		- ex) RentReader, RentSeatReserver ...
	- 행위자를 네이밍으로 명확하게 분류하면서 역할을 확실하게 부여한다.
	- application layer는 이런 implement 레이어의 행위자들을 적극 활용해서 usecase를 오케스트레이션 한다.

- domain layer는 ddd에 based한다. 
- domain layer는 비즈니스 업무 rule을 정의한다. 또한 도메인 엔티티를 가져오거나 저장하거나 삭제할 수 있는 Repository interface를 둔다. 행위를 정의한다. - `*Redaer` `*Writer`가 이들을 한번 더 감싸서 엔티티를 가져오는것과 null 체크를 담당하는 행위자가 된다. 

- support 모듈은 구현체들의 모임.
- api 모듈도 특정 통신 프로토콜 및 서버에 의존한 모듈. 

- 모듈 경계는 기술이나 벤더사 이름 보다는 개념(역할) 이름을 우선으로 한다
	- 그러다가 특정 벤더사나 기술이 너무 비대해지면(?) 기술 이름을 적고 모듈로 분리. 그 이전에는 개념(역할)로 모듈 분리하고 기술 의존은 내부 패키지로 추가 및 구분.

- 비즈니스 로직 구분은 "기술 로직" vs "비즈니스 로직"으로 구분한다. OAuth2는 기술 로직에 해당한다. 이 기술이 바뀌어도 비즈니스 core는 변하지 않는다. 하지만 인가에 관한 로직은 비즈니스 로직에 깊게 연관되어 있을 확률이 높다. "인가코드를 토큰으로부터 가져와서 비즈니스 로직에서 역할을 부여한다", "인증하는 방식은, 로그인 하는 방식은 기술에 의존한다."
