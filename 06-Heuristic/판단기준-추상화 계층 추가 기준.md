## 판단 기준

- 숨기려는 대상에 **복잡성이 있다** → 추상화 계층 추가 가치 있음
- 숨기려는 대상이 **이미 충분히 명확하다** → 추가하면 의미없는 간접 참조만 늘어남

## 효과적인 상황

```
// 저장소가 바뀔 수 있음 → Repository 인터페이스로 추상화
interface UserRepository {
    User findById(Long id);
}
// MySQL → MongoDB 교체 시 구현체만 바꾸면 됨
```

감추는 대상(DB 접근 방식)이 변경 가능성이 있고 복잡성을 가질 때.

## 실패하는 상황

```java
// Entity 안에 한 번 더 감싸기
class User {
    static User load(Long id) {
        return repository.findById(id); // ← findById 하나를 숨김
    }
}

// 호출부
User user = User.load(1L);        // 안에서 뭘 하는지 불분명
User user = userRepository.findById(1L); // 동일한 동작, 더 명확
```

`repository.findById(id)` 자체가 이미 명확한 추상화야.
복잡성 없는 것을 한 번 더 감싸면 **코드 읽는 사람이 내부를 확인해야 하는 비용**만 추가됨.

## 추상화 추가 전 체크리스트

```
1. 숨기려는 로직이 자주 바뀌는가?         → YES면 추상화 가치 있음
2. 호출부가 내부를 알 필요가 없는가?       → YES면 추상화 가치 있음
3. 동일한 로직이 여러 곳에서 반복되는가?   → YES면 추상화 가치 있음
4. 위 세 가지 모두 NO                     → 의미없는 간접 참조
```
