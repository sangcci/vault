## 1. 면접 전략 및 말하기 방법

### 핵심 출처
- **"면접 전에 꼭 알아야함"** (조회수 2,526)
- **"면접 무조건 알아야할 3가지"** (조회수 2,322)

### 🎯 핵심 전략 (5년차 개발자, 5회 이직 경험 기반)

#### 1️⃣ 난감한 질문에 자신감 유지하기

**상황:**
- 공백기, 잦은 이직, 개인적 약점을 공격하는 듯한 질문
- 말을 더듬거나 허둥지둥하는 순간 → **즉각 탈락**

**대응 방법:**
```
❌ 잘못된 반응: "음... 그게... 사실은..."
✅ 올바른 반응: "쉬고 싶었습니다" / "준비 기간이 좀 길었습니다"
```

**백엔드 개발자 관점:**
- 프로덕션 장애 대응과 동일: 당황하지 않고 사실을 명확하게 전달
- 문제를 인정하고 해결책을 제시하는 태도

#### 2️⃣ "면접관은 그냥 아저씨다" 마인드

**핵심 원칙:**
- 간절함이 과하면 무조건 떨어진다
- 면접관을 벌써부터 상사처럼 대하면 불합격
- 실력이 부족해도 합격하려면 → 면접을 잘 봐야 함

**실전 적용:**
- 가볍게 웃으면서 대화
- 긍정적으로 계속 이야기
- **농담도 가끔 OK**

**실제 사례:**
```
Q: "그동안 공백기였다가 왜 다시 준비하는 거예요?"
A: "아, 주식하는데 많이 잃었어요. 다시 복구해야 합니다."
→ 실제로 합격!
```

**백엔드 개발자 관점:**
- 코드 리뷰처럼 접근: 방어적이지 않고 자신감 있는 토론
- 동료로서 기술적 대화를 나누는 느낌

#### 3️⃣ 내가 여기서 무엇을 할지 명확하게 알고 가기

**가장 중요한 포인트!**

**준비 방법:**
1. 채용공고의 지원 자격, 우대 사항 분석
2. 그들의 아키텍처가 머릿속에 그려져야 함
3. 내가 했던 프로젝트와 매핑
4. 도움을 줄 수 있으면 지원, 없으면 지원 X

**면접에서:**
```
❌ 뻔한 대답: "귀사의 비전에 공감해서..."
✅ 구체적 답변: "제가 보니 TPS X를 처리하시는데,
   저는 Y에서 Z로 개선한 경험이 있습니다.
   Kafka를 활용한 이벤트 드리븐 아키텍처로..."
```

## 2. 채용공고 분석 방법

### 핵심 출처
- **"이력서 떨어지는 5가지 이유"** (조회수 2,684)

### 🔍 이력서가 떨어지는 5가지 이유

#### 1️⃣ 고집을 버리지 못함

**현실:**
- 멘토링 받고도 안 고치는 사람 → 계속 떨어짐
- 합격 연락 오는 사람 → 곧바로 수정

**교훈:**
- 피드백을 받으면 즉시 반영
- 자기 생각에 갇히지 말기

#### 2️⃣ 면접관 입장을 고려하지 않음

**면접관의 하루:**
- 10개 이상의 이력서 검토
- 면접 진행
- 과제 평가

**결과:**
→ 이력서를 보는 시간은 **몇 초**

**적용 방법:**
- 스캔 가능하도록 구조화
- 핵심만 부각
- 읽기 편한 형식

#### 3️⃣ 공고를 보고 아키텍처 매핑 안 함

**올바른 프로세스:**

```
1. 지원 자격 + 우대사항 읽기
   ↓
2. 그들의 아키텍처 머릿속에 그리기
   ↓
3. 내 프로젝트와 매핑
   ↓
4. 기여 가능 여부 판단
   ↓
5. 기여 가능하면 지원, 아니면 X
```

**예시:**
```
공고: "Microservices with Kafka, Redis 캐싱"
→ 내 경험: "이벤트 드리븐 아키텍처, Redis 세션 관리"
→ 매핑됨! 지원 Go
```

#### 4️⃣ 너무 과도한 어필

**문제:**
- 모든 것을 다 쓰려고 함
- "기본기를 쓰라"고 하니 "1+1=2" 수준을 씀

**해결:**
- 적절한 수준의 기본기
- 채용 공고 수준에 맞는 내용

#### 5️⃣ 필살기만 쓰는 사람

**면접관의 생각:**
- "기본기는 잘 알겠지" 라고 **절대 생각 안 함**

**황금 비율:**
```
📊 60-70% 기본기 + 30% 나만의 강점
```

**백엔드 예시:**
```
✅ 기본기 (60-70%):
- HTTP/REST API 설계
- 데이터베이스 정규화 및 인덱싱
- 트랜잭션 관리
- 에러 핸들링

✅ 나만의 강점 (30%):
- Redis Lua 스크립트 최적화
- gRPC 스트리밍 구현
- 분산 추적 시스템 구축
```

### 성능 테스트 결과

- TPS: 800
- 평균 응답 시간: 200ms

**최적화 적용:**

✅ 예상 트래픽의 3배 부하 테스트 통과
✅ Redis 장애 시 fallback 동작 확인
✅ DB 커넥션 고갈 시나리오 테스트

### 배포
- Blue-Green 배포로 무중단 배포

2. **요구사항 이상의 검증**

3. **시스템 전체 영향도 고려**
   - 사이드 이펙트 분석
