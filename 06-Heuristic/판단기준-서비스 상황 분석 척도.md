## 판단 기준

- 서비스 상황은 기능 목록이 아니라 규모·성장·변경·인력·운영 역량을 함께 본다.
- 같은 기능도 유저 수, 데이터 수, 판매량 증가, 요구사항 변경 빈도, 개발자 수, DevOps 유무에 따라 다른 선택이 된다.
- 상황 분석의 목적은 멋진 설계를 고르는 것이 아니라, 지금 팀이 감당 가능한 선택을 찾는 것이다.

> "유저 10000명+, 상품 10000개+, 판매량 계속 증가중, 요구사항 매일 변함, 개발자는 3명밖에 없음 + devops 팀 없음" — 사용자 메모, 2026-06-10

---

## 분석 척도

```text
서비스 상황 분석 척도
├─ 유저 규모
│  └─ 얼마나 많은 사람이 동시에 영향을 받는가?
├─ 데이터 규모
│  └─ 조회·정렬·검색 비용이 커지는가?
├─ 성장 추세
│  └─ 지금보다 부하와 판매량이 계속 늘어나는가?
├─ 변경 빈도
│  └─ 요구사항이 자주 바뀌어 API/모델 변경 비용이 커지는가?
├─ 팀 규모
│  └─ 개발자 수로 유지보수 가능한 복잡도인가?
└─ 운영 역량
   └─ DevOps 없이 배포·모니터링·장애 대응을 감당할 수 있는가?
```

## 효과적인 상황

- 기능 구현 전에 현재 서비스 맥락을 먼저 정리해야 할 때.
- API를 크게 묶을지, 변경 가능성 기준으로 나눌지 판단할 때.
- 운영 인력이 부족한 상황에서 복잡한 인프라나 구조를 도입할지 결정할 때.
- 면접이나 협업에서 “왜 그 선택을 했는가”를 상황 기반으로 설명해야 할 때.

---

## 실패하는 상황

- 유저 수와 상품 수만 보고 트래픽 문제로 단정하면 요구사항 변경 비용을 놓친다.
- 요구사항 변경만 보고 구조를 과하게 나누면 개발자 3명이 유지보수하기 어려워진다.
- DevOps 팀이 없는데 운영 복잡도가 큰 선택을 하면 장애 대응과 배포 부담이 커진다.
- 판매량 증가를 무시하면 상품상세·주문·결제처럼 매출과 연결된 화면의 변경 민감도를 낮게 본다.

---

## 상황 문장을 보는 법

| 문장 | 떠올릴 판단 |
|---|---|
| 유저 10000명+ | 장애 영향 범위와 동시 요청 가능성을 본다 |
| 상품 10000개+ | 조회·정렬·검색·페이징 비용을 본다 |
| 판매량 계속 증가중 | 매출 직결 화면과 API의 안정성·변경 민감도를 본다 |
| 요구사항 매일 변함 | API/모델을 크게 묶었을 때 변경 비용을 본다 |
| 개발자 3명 | 설계 복잡도와 유지보수 가능 범위를 본다 |
| DevOps 팀 없음 | 운영 자동화·모니터링·장애 대응 부담을 본다 |
