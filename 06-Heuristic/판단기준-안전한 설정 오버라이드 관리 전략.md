## 로깅 및 검증 로직 (Pseudo Code)

```java
public class ConfigValidator {
    public void validateOnStartup() {
        // 1. 모든 설정 소스 출력 (출처 추적용, 값은 출력하지 않음)
        env.getPropertySources().forEach(s -> logger.info("Loaded Source: " + s.getName()));

        // 2. 핵심 설정 로깅 (보안 마스킹 적용)
        String dbUrl = env.getProperty("db.url");
        String dbOrigin = env.getPropertyOrigin("db.url");
        
        // 값(Value)보다는 출처(Origin)를 로깅하는 것이 보안상 안전함
        logger.info(String.format("Final DB URL Source: %s", dbOrigin));
        
        // 민감 정보는 반드시 마스킹 처리
        String password = env.getProperty("db.password");
        logger.info("DB Password: " + mask(password)); 

        // 3. 필수 설정 누락 시 기동 차단 (Fail-Fast)
        if (dbUrl == null) throw new CriticalConfigMissingException("db.url");
    }
    
    private String mask(String value) {
        return (value == null || value.isEmpty()) ? "null" : "******";
    }
}
```

## 판단 기준 (Criteria)

### 1. 계층의 최소화 (KISS 원칙)
- **기준**: 설정 계층은 3단계를 넘지 않는 것이 권장됨 (Default -> Profile -> Env).
- **이유**: 계층이 깊어질수록 인간의 인지 능력이 섀도잉 현상을 추적하지 못함.

### 2. 명시적 오버라이드 (Explicit over Implicit)
- 공통 설정(`application.yaml`)에 너무 많은 것을 넣고 프로파일에서 깎아내는 방식보다는, 각 프로파일에서 **필요한 값을 명시적으로 정의**하는 방식을 선호함.

### 3. Fail-Fast 전략
- "덮어쓰겠지"라고 막연히 믿지 말고, 중요한 설정값이 예상 범위 밖일 경우 애플리케이션 기동을 실패(Panic/Exception)시키는 검증 로직을 추가함.

### 4. 보안 가시성 (Secure Visibility)
- **원칙**: 운영 환경에서는 **값(Value)보다 출처(Source)**를 로깅하는 데 집중함.
- **기준**: `password`, `secret`, `token`, `key` 키워드가 포함된 키는 절대 평문 로깅 금지.
- **효과**: "값이 무엇인가"가 아니라 "어디서(어느 파일/환경변수) 왔는가"만 알아도 설정 충돌 이슈의 대부분을 해결할 수 있음.

## 효과적인 상황 (Effective When)

- 마이크로서비스 아키텍처(MSA)와 같이 수많은 설정이 환경별로 다르게 주입되는 환경.
- 인프라 변경이 잦고 환경 변수 활용도가 높은 클라우드 네이티브 환경.

---

## 실패하는 상황 (Fails When)

- 설정 파일이 너무 여러 군데로 파편화되어 있어, 전체 그림을 한눈에 볼 수 없을 때.
- 로컬 개발 환경과 운영 환경의 설정 소스 종류(OS Env vs Yaml)가 완전히 달라 재현이 불가능할 때.
