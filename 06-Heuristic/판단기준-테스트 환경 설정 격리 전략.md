## 판단 기준 (Criteria)

테스트 환경 설정을 구성할 때, **"메인(Main) 설정의 상속"**과 **"완벽한 격리"** 중 무엇을 선택할지에 대한 기준입니다.

1.  **메인 설정 상속 (Default-Overriding)**
    - **적합한 경우**: 애플리케이션의 핵심 로직이 설정값에 민감하게 반응하여, 운영과 최대한 유사한 환경에서 검증이 필요할 때.
    - **장점**: 운영 환경과 테스트 환경 간의 괴리(Configuration Drift)로 인한 "내 컴퓨터에선 됐는데 운영에선 안 돼요" 현상을 방지함.
    - **주의**: 메인 설정의 변경이 테스트에 의도치 않은 영향을 줄 수 있으며, 디버깅 시 어떤 설정값이 주입되었는지 추적하기 어려울 수 있음.

2.  **완벽한 격리 (Full Isolation)**
    - **적합한 경우**: 외부 API 호출, 시크릿 매니저(Secret Manager) 연동, 특정 인프라 의존성이 강해 로컬/CI 환경에서 실행이 불가능할 때.
    - **장점**: 테스트 실행의 독립성과 속도가 보장되며, 테스트 환경만을 위한 특수한 값(Mock 주소 등)을 명확하게 제어할 수 있음.
    - **주의**: 테스트 설정만 업데이트하고 운영 설정을 누락하는 실수가 발생할 수 있음.

## 효과적인 상황 (Effective When)

- **상속 기반 전략**: 프로젝트 초기 단계이거나 설정 항목이 많지 않아 계층 구조를 한눈에 파악할 수 있을 때.
- **격리 기반 전략**: MSA 구조에서 수많은 외부 시스템과 연동하거나, 테스트 실행 속도가 개발 생산성에 직결될 때.

## 실패하는 상황 (Fails When)

- **암묵적 상속의 함정**: `application.yml`에 정의된 설정이 `application-test.yml`에서 명시적으로 덮어씌워지지 않아, 테스트 코드에서 실제 운영 DB나 외부 운영 서버로 요청을 보내는 사고가 발생할 때. (치명적)
- **과도한 모킹(Mocking)**: 설정을 너무 철저히 격리한 나머지, 실제 운영 환경에서는 설정 오류(Type Mismatch 등)로 구동되지 않는데 테스트만 통과하는 경우.

## 권장 전략 (Recommendation)

- **"Main 기반의 신뢰, 예외적 격리"**: 기본적으로는 `Main` 설정을 따르되, 테스트하기 어려운 요소(외부 API, 보안 변수)만 `Testcontainers`나 `WireMock` 주소로 명시적으로 덮어쓰는 방식을 권장함.
- **환경 변수 활용**: 파일 기반 설정보다는 환경 변수를 통해 주입받도록 설계하여, 실행 시점에 설정을 유연하게 교체할 수 있게 함.
