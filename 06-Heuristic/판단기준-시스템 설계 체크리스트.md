# Heuristic: 시스템 설계 시 6가지 질문으로 검증한다

## 기준

시스템 설계나 기술 선택 시 다음 6가지 질문을 순서대로 확인한다.

1. **가용성은 충분한가?** (Availability)
2. **성능 병목은 어디인가?** (Bottleneck)
3. **확장 전략은 무엇인가?** (Scalability)
4. **데이터 정합성은 어떻게 보장하는가?** (Consistency)
5. **장애 시 시스템은 어떻게 동작하는가?** (Resilience)
6. **운영과 모니터링은 쉬운가?** (Observability)

## 효과적인 경우

### 설계 단계

- 아키텍처 리뷰 시 체크리스트로 활용
- 기술 선택 의사결정 문서 작성 시
- 장애 사후 분석(Post-mortem) 시

### 면접 단계

- 시스템 설계 문제 답변 전 머릿속 체크
- "이 설계의 문제점은?" 질문 대비
- 답변 구조화

### 실무 적용 예시

**사례: 캐싱 도입 검토**

1. 가용성: 캐시 장애 시 DB 직접 조회 가능? (Fallback)
2. 병목: 어느 구간이 느린가? (DB 조회 vs 연산)
3. 확장: 캐시 서버 확장 계획은?
4. 정합성: 캐시 무효화 전략은? (TTL, Invalidation)
5. 장애: 캐시 서버 다운 시 시나리오는?
6. 운영: 캐시 히트율 모니터링 가능?

## 실패하는 경우

- 1-2개만 체크하고 나머지 무시
- 체크리스트를 암기만 하고 상황에 적용 못 함
- 모든 항목을 완벽하게 충족하려다 과도한 설계
- 우선순위 없이 모든 항목 동등하게 취급

## 각 항목별 핵심 키워드

### 1. 가용성
- SPOF 제거
- 다중화 (Active-Standby, Active-Active)
- Failover
- SLA 목표 (99.9%, 99.99%)

### 2. 성능 병목
- Latency (지연시간)
- Throughput (처리량)
- DB 쿼리
- 네트워크 I/O
- CPU / 메모리

### 3. 확장성
- Horizontal Scaling (수평 확장)
- Vertical Scaling (수직 확장)
- Stateless 설계
- Partitioning / Sharding

### 4. 데이터 정합성
- ACID
- Eventual Consistency
- 트랜잭션 격리 수준
- 락 전략

### 5. 장애 대응
- Circuit Breaker
- Retry / Timeout
- Graceful Degradation
- Bulkhead

### 6. 운영 / 모니터링
- Logging
- Metrics (Prometheus, Grafana)
- Tracing (Jaeger, Zipkin)
- Alerting

## 우선순위 결정 기준

모든 항목을 동시에 만족시킬 수 없으므로, 서비스 특성에 따라 우선순위 결정.

### 금융 서비스
1. 데이터 정합성 (최우선)
2. 가용성
3. 장애 대응
4. 성능
5. 확장성
6. 운영성

### 소셜 미디어
1. 확장성 (최우선)
2. 성능
3. 가용성
4. 장애 대응
5. 운영성
6. 데이터 정합성 (Eventual Consistency 허용)

### 스타트업 초기
1. 운영성 (빠른 배포/수정)
2. 성능 (사용자 경험)
3. 가용성
4. 나머지는 필요할 때 추가
