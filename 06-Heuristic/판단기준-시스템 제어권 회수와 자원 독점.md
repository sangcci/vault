# Heuristic: 시스템 제어권 회수와 자원 독점

## 기준

애플리케이션이 요구하는 성능과 정합성이 운영체제(OS)의 **범용적 관리 범위를 벗어날 때**, OS 캐시를 우회하고 **자원 제어권을 직접 회수(Direct Control)**해야 한다.

## 언제 제어권을 가져와야 하는가? (Context)

1. **성능의 예측 가능성(P99 Latency)이 핵심일 때**: 
   - OS의 무작위적인 페이지 교체나 스케줄링이 서비스의 응답 속도에 튀는 지점(Spike)을 만든다면, 메모리나 CPU 코어를 독점적으로 점유해야 한다.
2. **데이터의 절대적 정합성이 필요할 때**: 
   - 하부 레이어의 원자적 쓰기 단위가 믿음직스럽지 못할 때, 상위 레이어에서 직접 로그를 남기고 복구 로직을 짜야 한다. (`[[본질-최소 단위의 불일치]]`)
3. **자원 소모가 비효율적일 때 (Double Buffering)**: 
   - 애플리케이션도 캐시하고 OS도 캐시하면서 메모리가 낭비되는 상황이라면 한쪽(주로 OS)을 꺼야 한다.

## 구체적인 실천 방안

### 1. Direct I/O (`O_DIRECT`)
- 파일 읽기/쓰기 시 OS 페이지 캐시를 거치지 않고 직접 디스크와 통신한다. 현대 고성능 DB의 기본 설정이다.

### 2. 메모리 고정 (`mlock`)
- 특정 메모리 영역을 'Non-swappable'로 설정하여, 절대 디스크로 쫓겨나지 않도록 RAM에 고정한다.

### 3. CPU Affinity (Set Affinity)
- 특정 프로세스나 스레드를 특정 CPU 코어에 박아버려, 불필요한 문맥 교환과 캐시 미스를 방지한다.

## 만능 문장 템플릿

> "OS는 **범용적인 공평함**을 추구하지만, 고성능 시스템은 **특화된 독점**을 필요로 합니다. 제어권을 직접 가져오는 것은 구현 복잡도를 높이지만, 시스템의 성능 한계를 돌파하기 위한 필수적인 선택입니다."
