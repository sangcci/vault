# Heuristic: SI 채용공고 분석 및 지원 전략

**핵심 질문**: "채용공고의 텍스트 너머의 '실무 요구사항'을 어떻게 읽어낼 것인가?"

## 📍 기준 1: 아키텍처 매핑 (Architecture Mapping)

채용공고를 분석할 때는 단순한 기술 나열이 아닌, 회사의 **시스템 구조와 나의 경험을 연결**하는 과정이 필요합니다.

1.  **자격요건 분석**: "우리는 이런 환경에서 당장 일할 사람이 필요하다."
    - Java/Spring, RDBMS, SQL 숙련도 → 비즈니스 로직 구현 속도와 데이터 처리 능력 중심.
2.  **우대사항 분석**: "현재 우리가 수주한 프로젝트의 페인포인트는 이것이다."
    - 대용량 트래픽, MSA, Kafka → 시스템 고도화 및 분산 환경 경험 요구.
3.  **지원 동기 전략**: "나의 [X 프로젝트] 경험이 귀사의 [Y 아키텍처] 문제를 해결하는 데 기여할 수 있다."

## 📍 기준 2: 비즈니스 모델의 이해 (독립형 vs 수주형)

돈을 누가 주느냐에 따라 개발자에게 요구하는 핵심 역량과 일하는 방식이 완전히 달라집니다.

| 구분 | 독립형 (B2C/서비스) | 수주형 (B2B/SI) |
| :--- | :--- | :--- |
| **수입원** | 일반 대중 (서비스 사용료/광고) | 기업/기관 (프로젝트 구축 대금) |
| **핵심 역량** | **사업 이해도** 및 서비스 성장 기여 | **기술적 숙련도** 및 정확한 구현 |
| **지식의 폭** | 넓고 다양하게 (풀스택적 사고) | 특정 분야를 깊고 정확하게 (전문화) |
| **업무 형태** | 지속적인 개선과 빈번한 회의 | 명확한 분업과 납기 중심의 실행 |
| **목표** | 사용자 경험 및 비즈니스 임팩트 | 약속된 품질과 기한(Deadline) 준수 |

- **분석 팁**: 공고에 "도메인 경험 우대"가 강조되어 있다면, 이는 해당 비즈니스 로직의 복잡도가 높아 **기술력만큼이나 업무 도메인 지식**이 중요한 수주형 프로젝트일 확률이 높습니다.

## ✅ 효과적인 경우

- **실무 관점 어필**: "JPA를 썼다"보다 "N+1 문제를 해결하여 DB 부하를 X% 줄였다"는 수치제시가 SI의 효율성 관점에 부합함.
- **거대 시스템 경험 강조**: 공공, 금융 등 SI에서만 다룰 수 있는 대규모 트래픽이나 복잡한 정합성 처리 경험을 어필할 때.

## ⚠️ 실패하는 경우

- **박애주의적 지원**: 모든 회사에 똑같은 이력서를 보내는 경우.
- **기술 나열**: 내가 사용한 기술이 회사의 시스템에 어떻게 도움이 될지 설명하지 못함.
- **정처기 경시**: SI에서 정보처리기사는 단순 자격증이 아닌 '인건비 산정 기준'임을 간과함.
