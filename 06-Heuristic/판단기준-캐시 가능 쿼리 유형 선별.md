## 판단 기준

```text
쿼리의 키 공간이 유한한가?
  ├─ Yes (정형 쿼리) → 로컬 캐시 적용 가능
  │    예: 최신순 첫 페이지, 인기순 상위 N개, 키워드 없는 목록
  │
  └─ No (동적 쿼리) → 로컬 캐시 부적합
       ├─ 인스턴스 간 일관성 필요 → Redis (글로벌 캐시)
       └─ 형태소/유사도 검색 필요  → Elasticsearch 등 검색 엔진
```

- **정형 쿼리 판별 기준**: 파라미터 조합 수가 배포 시점에 예측 가능하고 런타임에 증가하지 않음

---

## 효과적인 상황

- "최신순 첫 페이지", "인기순 상위 N개" 처럼 접근 패턴이 수렴하는 쿼리
- 검색 키워드 없이 정렬·카테고리만으로 조회하는 API
- TTL을 짧게 두어 일부 불일치를 허용할 수 있는 조회성 데이터

---

## 실패하는 상황

- 키워드 × 필터 × 페이지 조합 → [[현상-캐시 키 공간 폭발 (Cache Key Space Explosion)]] 발생
- 다중 인스턴스 환경에서 로컬 캐시 적용 → 서버마다 다른 결과 반환
