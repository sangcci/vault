# Heuristic: 동시성 이상 현상 분류 체계 (3-Layer)

동시성 문제를 마주했을 때, 다음의 3가지 기준(Layer)으로 분석하면 원인 파악과 해결책 선정이 명확해집니다.

## Layer 1: 발생 계층 (Scope)
> **"어디서 터졌는가?"**

1. **Application Layer (OS/Runtime)**
    - 멀티스레드 환경에서 CPU 캐시, 공유 메모리(Heap) 접근 시 발생.
    - 해결책: `synchronized`, `volatile`, `Lock/Mutex`, `AtomicVariable`.
2. **Database Layer (Storage)**
    - 여러 트랜잭션이 동일한 레코드/페이지에 접근할 때 발생.
    - 해결책: `Isolation Level`, `Pessimistic/Optimistic Lock`.

---

## Layer 2: 문제의 성격 (Nature)
> **"어떻게 터졌는가?"**

1. **Safety (안정성) 위배**
    - 결과값이 틀린 경우. (데이터 오염, 정합성 붕괴)
    - 예: 계좌 잔액이 100원인데 동시에 100원씩 두 번 출금되어 -100원이 됨.
2. **Liveness (활동성) 위배**
    - 시스템이 멈추거나 응답하지 않는 경우.
    - 예: 교착 상태(Deadlock)에 빠져 모든 요청이 타임아웃됨.

---

## Layer 3: 코드 패턴 (Pattern)
> **"어떤 로직에서 터졌는가?"**

1. **RMW (Read-Modify-Write)**
    - 값을 읽고(`R`), 수정하고(`M`), 다시 쓰는(`W`) 과정 사이에 다른 개입이 발생.
    - **핵심**: "수정 전의 값이 최신임을 보장할 수 있는가?"
2. **CTA (Check-then-Act)**
    - 조건을 확인하고(`Check`), 결과에 따라 행동(`Act`)하기 직전에 조건이 변함.
    - **핵심**: "확인한 상태가 실행 시점에도 유효한가?"

---

## 동시성 이상 현상 매트릭스 (Quick Map)

| 계층 | 문제 성격 | 이상 현상 (Phenomenon) | 주요 원인 패턴 |
| :--- | :--- | :--- | :--- |
| **DB** | Safety | **Dirty Read**, **Lost Update**, **Read/Write Skew** | RMW |
| **DB** | Safety | **Phantom Read** | CTA |
| **App** | Safety | **Race Condition (Data)** | RMW |
| **App** | Safety | **Visibility Issue** | Hardware Cache |
| **App** | Liveness | **Deadlock**, **Livelock**, **Starvation** | Lock Contention |
