## 판단 기준

```text
버전 고정 여부?
  → 모든 소프트웨어 fixed 유지. 마이너도 자동 업데이트 금지.
  → "오늘 빌드 = 내일 빌드" — 내가 통제하지 않는 변화는 시스템에 들어오면 안 됨.

업데이트 실행 주체?
  → 사람이 직접. 체인지로그 확인 → breaking issue 검토 → 관련 코드 QA 후 진행.

메이저 버전 업데이트 시점?
  → 메이저 출시 후 → 다음 패치 한 번 더 나온 뒤 올릴 것 (초기 버그 회피).

보안 패치는?
  → 예외 없이 즉시 적용. 협상 대상 아님.
```

---

## 효과적인 상황

- 장기 운영 서비스의 안정적인 의존성 관리
- 신규 팀원 합류 시에도 동일한 빌드 결과 보장

---

## 실패하는 상황

- "최신 버전 유지"를 자동화하면 예상치 못한 breaking change로 프로덕션 장애 발생
- 우리는 남의 베타테스터가 아님 — 조립식 개발 지양
- 결과론적 동작을 추구하지 않으면 우연에 기대는 도박이 됨
