# Heuristic: 비기능 요구사항 수치 결정 시 공식 표준과 업계 관행을 근거로 삼는다

## Criteria (기준)

소프트웨어의 비기능적 품질을 정의할 때, 주관적 추측이 아닌 아래의 공식 표준(ISO, RFC, NIST) 및 검증된 업계 관행을 기준으로 수치를 결정한다.

### 1. 성능 (Performance) 판단 수치
*   **API 응답 시간 목표**
    *   **200ms (평균)**: Google Core Web Vitals의 INP(Interaction to Next Paint) "Good" 등급 기준.
    *   **100ms (즉각)**: 사용자가 지연을 느끼지 못하는 한계치 (Nielsen, 1993, "Response Time Limits").
    *   **1.0s (흐름 유지)**: 사용자의 사고 흐름이 끊기지 않는 최대 한계 (Nielsen, 1993).
    *   **500ms (p95)**: 운영 안정성을 고려한 업계 보수적 SLO(Service Level Objective) 관행.
*   **DB 커넥션 풀 (HikariCP) 공식**
    *   공식: `Pool Size = Tn × (Cm - 1) + 1` (Tn: 최대 스레드 수, Cm: 한 스레드당 동시 보유 커넥션 수)
    *   출처: [HikariCP Wiki - About Pool Sizing](https://github.com/brettwooldridge/HikariCP/wiki/About-Pool-Sizing)

### 2. 보안 (Security) 판단 수치
*   **비밀번호 해싱 (BCrypt)**
    *   **Cost Factor 10 이상**: 현재 하드웨어 성능 기준 최소 권장치 (업계 표준은 12 권장).
    *   출처: [OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)
*   **암호화 표준 프로토콜**
    *   **AES-256**: [NIST FIPS PUB 197](https://csrc.nist.gov/pubs/fips/197/final) 표준. 대칭키 암호화의 사실상 표준.
    *   **TLS 1.3**: [RFC 8446](https://datatracker.ietf.org/doc/html/rfc8446) 표준. (보안 취약한 1.0/1.1은 [RFC 8996](https://datatracker.ietf.org/doc/html/rfc8996)에 의해 폐지됨)
*   **인증 토큰 수명 (OAuth 2.0)**
    *   **Access Token**: 5~60분 (단기), **Refresh Token**: 1~2주 (장기) 권장.
    *   출처: [RFC 6749 - The OAuth 2.0 Authorization Framework](https://datatracker.ietf.org/doc/html/rfc6749)

### 3. 가용성 및 일관성 (Availability & Consistency)
*   **가용성 목표 (SLA)**
    *   **99.9% (Three Nines)**: 연간 최대 다운타임 8.76시간 허용. Enterprise급 서비스의 기본 목표.
    *   출처: [ISO/IEC 25010:2023 - Systems and software Quality Requirements and Evaluation](https://www.iso.org/standard/78176.html)
*   **데이터베이스 격리 수준**
    *   **READ COMMITTED**: 일반 웹 서비스의 기본 설정 (SQL-92 표준 기반).
    *   출처: [ISO/IEC 9075:1992 - SQL Standard](https://www.iso.org/standard/16663.html)

### 4. 품질 관리 (Quality)
*   **테스트 코드 커버리지**
    *   **80%**: 업계의 보편적 "Golden Standard". 파레토 원칙(80/20)에 근거하여 가성비가 가장 높은 구간.

## Effective When (효과적인 경우)

*   **아키텍처 설계 및 인프라 구축 초기**: 서비스의 목표 성능(TPS, Latency)과 가용성 지표를 정량적으로 설정해야 할 때.
*   **이해관계자 설득 및 면접**: 특정 기술 결정(예: 왜 BCrypt 12를 썼는가?)에 대해 주관적 의견이 아닌 "공식 표준"이라는 강력한 근거가 필요할 때.
*   **부하 테스트 결과 해석**: 테스트 결과로 나온 수치가 "정상" 범위인지 "위험" 범위인지 판단하는 기준점으로 활용.

---

## Fails When (실패하는 경우)

*   **특수 도메인 환경**: 의료/항공/실시간 금융 등 "표준 이상의 극단적 안정성"이 요구되거나, 반대로 성능보다 "기능 구현 속도"가 압도적으로 중요한 초기 스타트업 단계.
*   **인프라 제약 조건**: 네트워크 레이턴시가 물리적으로 200ms를 넘는 환경(해외 통신 등)에서 표준 응답 시간 수치를 강요할 때.
*   **수치 지상주의**: 80% 커버리지라는 숫자 자체에 매몰되어, 정작 중요한 비즈니스 로직에 대한 테스트 질이 떨어지는 경우.
