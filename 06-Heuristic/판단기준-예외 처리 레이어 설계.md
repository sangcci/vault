## 판단 기준

```
예외 처리 책임 분배:

Service Layer           ← 핵심 예외 처리 위치
  └── flush() 명시 + try-catch + CustomException 변환
  └── 이유: @Transactional 흐름 안에서 flush 타이밍 제어 가능
  └── 이유: Lazy Loading이 Service 안에서 발생 → Session 살아있는 시점에 처리

Infrastructure Layer    ← 예외 종류 변환까지만
  └── DataIntegrityViolationException → DomainException 변환 허용
  └── 비즈니스 메시지 결정 ❌  → Service 책임

ControllerAdvice        ← HTTP 응답 매핑 + 로깅만
  └── CustomException → HTTP 상태 코드 변환
  └── flush 보장 불가, Lazy Loading 예외 처리 불가
  └── 비즈니스 로직 ❌
```

---

## 효과적인 상황

- `@Transactional` 안에서 DB 제약 예외(Unique, FK) 처리할 때
- JPA Lazy Loading이 Service 내에서 일어날 때
- 예외 원인(제약명)으로 비즈니스 메시지를 결정해야 할 때
- 트랜잭션 흐름을 Service 코드만 읽고 파악해야 할 때

---

## 실패하는 상황

**Infrastructure에 핵심 예외 처리 집중 시:**
- Repository 메서드만 봐선 `@Transactional` 범위와 flush 타이밍 파악 불가
- Service 코드에서 트랜잭션 흐름이 불투명해짐

**ControllerAdvice 단독 처리 시:**
- Service를 거치지 않고 예외가 올라오므로 flush 타이밍 보장 불가
- Hibernate `LazyInitializationException` 등 Session이 닫힌 후 발생하는 예외 처리 불가
- `@Transactional` 내부에서 발생한 예외인지 판단할 컨텍스트 없음
