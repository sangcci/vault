## 판단 기준

- 일반적인 OLTP 서비스 → **3NF**까지 적용
- 결정자가 후보키가 아닌 관계가 존재하고 갱신 이상이 우려됨 → **BCNF**까지 적용
- 조회 성능이 쓰기 정합성보다 중요한 경우 (OLAP, 대시보드 등) → **의도적 비정규화** 고려
- 4NF 이상 (다치 종속, 조인 종속) → 특수한 경우 외 실무에서 드물게 적용

## 효과적인 상황

- Surrogate key + Entity별 테이블 분리가 이미 되어 있는 경우 → 자연스럽게 3NF 달성
- 도메인 모델이 명확하게 정의된 경우

## 실패하는 상황

- 하나의 테이블에 여러 도메인 개념을 혼재시킨 경우
- 복합 자연키를 기본키로 사용한 경우 → 부분 종속 발생 가능성 높음
- BCNF로 분해 시 후보키 정보가 손실될 수 있음 (무손실 분해 여부 확인 필요)
