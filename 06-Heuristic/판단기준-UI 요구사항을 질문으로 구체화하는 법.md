## 판단 기준

- UI 계획서를 보면 화면에 보이는 요소보다 먼저 빠진 정책을 질문한다.
- 질문은 기획자를 공격하는 것이 아니라, 기획자·사업가가 가진 의도와 뉘앙스를 개발 가능한 조건으로 바꾸는 과정이다.
- Front와 Back 구현 파트는 나뉘어도 요구사항 흐름은 하나이므로, 화면 흐름 전체를 알아야 좋은 질문이 가능하다.

> "간단한 UI 계획서 같은걸 보고 계속 질문해보는거임." — 사용자 메모, 2026-06-10

---

## 질문화 흐름

```text
UI 계획서
├─ 화면에 보이는 것
│  └─ 목록 / 버튼 / 텍스트 / 이미지
└─ 화면에 안 보이는 정책
   ├─ 중복 가능성
   ├─ 노출 개수
   ├─ 페이지 방식
   ├─ 노출 정보
   ├─ 카테고리별 수량
   └─ 정렬 기준
      └─ API / DB / Front 상태 / 예외 정책
```

```pseudo
function turnUiPlanIntoQuestions(uiPlan) {
    visibleElements = readVisibleElements(uiPlan)
    hiddenPolicies = findMissingPolicies(visibleElements)
    questions = askAbout(hiddenPolicies)
    return convertToDevelopmentConstraints(questions)
}
```

---

## 효과적인 상황

- UI 계획서에는 화면만 있고 정책·예외·정렬·노출 기준이 비어 있을 때.
- 상품 목록, 카테고리, 상품상세처럼 Front와 Back 데이터 흐름이 강하게 연결될 때.
- 기획 변경이 잦아 초기에 질문하지 않으면 구현 후 재작업이 커질 때.
- 내가 맡은 파트 밖의 흐름을 몰라 질문조차 못 하는 상태를 피해야 할 때.

---

## 실패하는 상황

- 화면에 보이는 것만 구현하면 중복·정렬·개수·페이징 정책을 뒤늦게 발견한다.
- Back만 안다고 생각하고 Front 흐름을 모르면 어떤 데이터가 언제 필요한지 질문하지 못한다.
- 질문을 귀찮은 확인 절차로 여기면 기획자·사업가의 의도를 이해하지 못한다.
- 정책 질문 없이 API를 만들면 구현 후 미비한 점을 수정하는 시간 낭비가 생긴다.

---

## UI 계획서에서 뽑을 질문

| 질문 축 | 실제 질문 |
|---|---|
| 중복 가능성 | 2개 카테고리에 1개 상품이 중복으로 뜰 수 있는가? |
| 노출 개수 | 화면에 상품을 몇 개까지 보여주는가? |
| 페이지 방식 | 페이지 방식인가, 스크롤 방식인가? |
| 노출 정보 | 상품 카드에 어떤 정보를 보여주는가? |
| 카테고리별 수량 | 카테고리별 상품 개수도 보여줘야 하는가? |
| 정렬 기준 | 최신순, 인기순, 판매순 중 어떤 기준으로 정렬하는가? |
