## 판단 기준

```
패턴으로 "이해"할 때  → 써라
패턴으로 "설계"할 때  → 멈춰라
```

- 기존 코드를 보며 "왜 이렇게 생겼지?" → 패턴으로 의도 파악 ✓
- 팀원에게 설계 설명 → 패턴 어휘로 소통 압축 ✓
- "여기 Visitor 패턴 적용해야겠다"며 코드 시작 → 억지 적용 위험 ✗
- "이건 Visitor니까 이렇게만 해야 해" → 경직된 사고 ✗

---

## 효과적인 상황

**코드 리딩**
```
코드 읽다가 accept() + visit() 발견
  → "Visitor 구조네"
  → DFS? 타입 분기? Double Dispatch? 를 한번에 떠올림
  → 읽는 속도 급상승
```

**설계 소통**
```
"여기 Template Method 쓰자"
  한 문장으로 전달되는 것:
  - 알고리즘 뼈대 고정
  - 가변 단계는 서브클래스 위임
  - 상속 구조
```

**면접 / 시험**
- 패턴 이름 → 의도 + 구조를 연결할 수 있으면 충분
- 정처기: "어떤 상황에 어떤 패턴?" → 패턴별 핵심 의도 한 줄이 전부

## 실패하는 상황

**패턴 이름으로 기술을 정의하려 할 때**
```
"Kafka는 Mediator 패턴이다" (X)
  → 분산 로그, 파티셔닝, 컨슈머 그룹이 전부 날아감

올바른 이해:
"Kafka는 서비스 간 의존을 중재한다는 점에서
 Mediator와 결을 같이하지만,
 내구성 / 분산 / 재처리 특성이 핵심이다"
```

**패턴 적용을 목표로 설계할 때**
```
문제:   파일 트리 순회가 필요함
나쁜 접근: "Visitor 패턴 써야겠다" → 억지로 accept/visit 구조 설계
좋은 접근: 문제를 풀다 보니 → 나중에 보니 Visitor 구조가 나와있음
```

**하나의 패턴으로 기술 전체를 이해했다고 착각할 때**
```
JdbcTemplate = Template Method  ← 맞지만
JdbcTemplate ≠ Template Method만  ← 틀림

실제: Template Method + Strategy + 예외 추상화 + 커넥션 관리
```

## 기억할 것

실제 기술은 단일 패턴이 아닌 **패턴의 조합**이다.
패턴은 그 조합의 일부를 설명하는 어휘일 뿐이다.

```
패턴 = 원자(atom)
기술 = 분자(molecule)

"이 분자는 수소다" (X)
"이 분자에 수소가 포함되어 있다" (O)
```
