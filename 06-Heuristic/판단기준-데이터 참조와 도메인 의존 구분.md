## 판단 기준

- 단순 식별자, enum, 문자열, 타입 코드만 가진 경우에는 대체로 **데이터 참조**로 본다.
- 상대 도메인의 객체를 직접 들고 메서드를 호출하거나 정책을 알아야 하면 **도메인 의존**으로 본다.
- 상대 도메인의 변경이 내 비즈니스 로직 수정으로 전파되면 결합도가 높다고 본다.
- 상대 도메인의 이름이나 분류값만 저장하고 내 규칙이 독립적으로 유지되면 결합도가 낮다고 본다.
- 비즈니스 unique 값이 있더라도 그 값이 변경될 수 있다면, 참조용으로는 의미 없는 인공 식별자를 두는 편이 변경 전파를 줄인다.

> "X is _coupled_ to Y if a change to Y can potentially require a change in X" — [SEforSDL, Software Design Fundamentals](http://se-education.org/se-book/designFundamentals/index.html)

---

## 판단 흐름

```text
Review가 Product를 의존하는가?
│
├─ Product 객체를 필드로 직접 가지는가?
│   └─ 예 → 의존 가능성 증가
│
├─ Product의 메서드나 정책을 호출하는가?
│   └─ 예 → 도메인 의존
│
├─ Product 정책 변경 시 Review 로직도 바뀌는가?
│   └─ 예 → 높은 결합
│
└─ ReviewType.PRODUCT 같은 분류값만 가지는가?
    └─ 예 → 데이터 참조에 가까움
```

## 예시

### 낮은 결합: 데이터로 참조한다

```java
class Review {
    private ReviewType reviewType; // PRODUCT
}
```

```text
Review
 └─ ReviewType.PRODUCT
    └─ 리뷰 대상 분류값
       └─ Product 정책을 직접 알지 않음
```

이 경우 `Review`는 `Product`라는 도메인 객체의 정책이나 내부 구조를 모른다. 단지 “이 리뷰가 상품 리뷰다”라는 분류 데이터를 가진다.

> "Two modules are data coupled if they communicate by passing parameters." — [UNC COMP 145, Intermodule Coupling](https://www.cs.unc.edu/~stotts/COMP145/coupling.html)

### 높은 결합: 도메인에 의존한다

```java
class Review {
    private Product product;

    boolean canBeDisplayed() {
        return product.isPublished() && product.isReviewable();
    }
}
```

```text
Product 정책 변경
  ↓
Product#isReviewable 의미 변경
  ↓
Review#canBeDisplayed 판단도 수정 필요
  ↓
변경 전파 발생
```

이 경우 `Review`는 `Product`의 상태와 정책을 알아야 자기 로직을 수행한다. `Product` 정책 변경이 `Review` 변경으로 이어질 수 있으므로 도메인 의존이 생긴다.

### 식별자는 변경 전파를 끊는 간접 참조다

비즈니스에는 `email`, `productCode`, `orderNumber`처럼 unique한 값이 있을 수 있다. 하지만 이런 값은 비즈니스 의미를 가지기 때문에 정책 변경에 따라 바뀔 수 있다. 이 값을 다른 도메인이 직접 참조하면, 값의 변경이 참조하는 쪽으로 전파된다.

> "Because they have a business meaning, if that meaning changes (eg. they change their surname), then that value NEEDS to change." — [SSW Rules, Use Natural or Surrogate Primary Keys](https://www.ssw.com.au/rules/use-natural-or-surrogate-primary-keys)

```text
비즈니스 키 직접 참조
Product
 └─ productCode = "ABC-001"

Review
 └─ productCode = "ABC-001"

productCode 정책 변경
  ↓
Product.productCode 변경
  ↓
Review.productCode / FK / API / 캐시 키까지 전파 가능
```

```text
인공 식별자로 간접 참조
Product
 ├─ id = 1                  ← 안정적인 참조용 식별자
 └─ productCode = "ABC-001" ← 변경 가능한 비즈니스 값

Review
 └─ productId = 1

productCode 변경
  ↓
Product 내부 값만 수정
  ↓
Review.productId는 그대로
```

따라서 인공 식별자 `id`는 단순 편의가 아니라, 변경 가능한 비즈니스 값과 안정적인 참조 대상을 분리해 결합도를 낮추는 장치로 볼 수 있다. 핵심은 unique 여부가 아니라 **그 값이 비즈니스 의미를 가져 변경될 수 있는가**다. 자세한 키 선택 기준은 [[개념-대리키 (Surrogate Key)]]와 [[개념-복합키 (Composite Key)]]를 참조한다.

## 효과적인 상황

- 도메인 클래스 간 연관관계를 설계할 때.
- enum, type code, id, 객체 참조 중 무엇을 둘지 판단할 때.
- “이 클래스가 저 도메인을 의존하는가?”를 코드 리뷰에서 설명해야 할 때.
- 결합도를 낮춘다는 말이 실제 코드에서 무엇을 의미하는지 확인할 때.

---

## 실패하는 상황

- 이름이 등장한다는 이유만으로 의존이라고 판단하면 과하게 분리할 수 있다.
- 객체 참조와 메서드 호출이 있는데도 단순 데이터라고 보면 변경 전파를 놓친다.
- 결합도를 0으로 만드는 것만 목표로 삼으면 필요한 협력까지 제거되어 도메인 모델이 빈약해질 수 있다.
- enum 값 하나를 둔 것과 상대 도메인의 정책을 호출하는 것을 같은 강도의 결합으로 보면 설계 판단이 흐려진다.

## 판단 질문

| 질문 | 의미 |
|---|---|
| 값만 저장하는가, 행동을 호출하는가? | 데이터 참조와 도메인 의존의 1차 구분 |
| 상대 도메인의 정책을 알아야 하는가? | 비즈니스 규칙 결합 여부 |
| 상대 도메인 변경 시 내 로직도 바뀌는가? | 변경 전파 여부 |
| id/type/enum만으로 충분한가? | 낮은 결합으로 유지 가능한지 |
| unique한 비즈니스 값이 변경될 수 있는가? | 인공 식별자로 변경 전파를 끊어야 하는지 |
| 객체 참조가 실제 협력을 표현하는가? | 필요한 결합인지 과한 결합인지 |
