## 판단 기준

- 실무를 잘하려면 요구사항·코드·개념·상황을 따로 외우지 않고, 하나의 흐름으로 느껴야 한다.
- 요구사항에서 질문을 만들고, 코드에서 결정 이유를 추적하고, 개념 사이의 격벽을 정리하고, 상황에 맞는 선택으로 마무리한다.
- 정답을 찾는 순서가 아니라 맥락을 읽는 순서로 사용한다.

> "요구사항 느끼기(생각하는법, 질문하는법) - 코드 느끼기(왜 이렇게 구현하도록 결정했는지?) - 개념 정리(개념과 격벽. 이건 제미니의 개발실무 강의에서 나온 내용) - 상황 정리(상황에 따른 선택)" — 사용자 메모, 2026-06-10

---

## 4단계 흐름

```text
개발 실무를 느끼는 4단계
├─ 1. 요구사항 느끼기
│  └─ 화면/기획서를 보고 빠진 질문을 만든다
├─ 2. 코드 느끼기
│  └─ 왜 이렇게 구현하도록 결정했는지 추적한다
├─ 3. 개념 정리
│  └─ 개념의 경계와 격벽을 구분한다
└─ 4. 상황 정리
   └─ 현재 맥락에서 어떤 선택이 맞는지 판단한다
```
---

## 효과적인 상황

- UI 계획서만 있고 세부 정책이 비어 있을 때.
- 이미 구현된 코드를 읽으며 팀의 결정 이유를 파악해야 할 때.
- `Product`, `Category`, `ProductCategory`처럼 여러 개념이 엮여 경계가 흐릴 때.
- 유저 수·상품 수·판매량·변경 빈도·팀 규모 같은 현재 상황을 기준으로 선택해야 할 때.

---

## 실패하는 상황

- 요구사항을 느끼지 않고 바로 구현하면 뒤늦게 빠진 정책을 발견한다.
- 코드 모양만 보고 결정 이유를 추적하지 않으면 같은 실수를 반복한다.
- 개념의 격벽을 정리하지 않으면 Entity·API·화면 단위가 뒤섞인다.
- 상황 정리 없이 일반론으로 선택하면 팀 규모와 운영 역량에 맞지 않는 설계를 하게 된다.

## 단계별 질문

| 단계 | 질문 |
|---|---|
| 요구사항 느끼기 | 이 화면에서 기획자가 아직 말하지 않은 정책은 무엇인가? |
| 코드 느끼기 | 이 코드는 왜 이렇게 생겼고, 어떤 제약 때문에 이 선택을 했는가? |
| 개념 정리 | 어떤 개념이 중심이고, 어떤 개념은 부수 정보인가? |
| 상황 정리 | 지금 규모·변경 빈도·팀 역량에서 감당 가능한 선택은 무엇인가? |
