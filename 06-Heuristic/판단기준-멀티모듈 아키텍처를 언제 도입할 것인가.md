## 판단 기준

- 단순 CRUD이고 유지보수 주기가 길다 → 전형적인 layered architecture를 유지한다.
- 업무 규칙이 늘고 Service가 절차 나열로 비대해진다 → core 내부 구조부터 정리한다.
- DB, cache, 외부 client, notification sender 같은 기술 교체 지점이 반복된다 → support adapter를 분리한다.
- API 서버와 batch/scheduler의 배포 주기, 장애 격리, 자원 튜닝 기준이 달라진다 → 실행 모듈을 분리한다.
- 팀원이 코드만 보고 usecase 흐름을 따라가기 어렵다 → 모듈과 패키지로 업무 경계를 드러낸다.
- “멋있어 보여서” 또는 “DDD처럼 보여서”가 이유다 → 도입하지 않는다.

```text
작은 CRUD
  |
  v
Layered Architecture
  |
  |  업무 규칙 증가 / 기술 교체 지점 반복 / 실행 단위 분리 필요
  v
Core 구조 정리
  |
  |  의존성 방향 강제 필요
  v
Multi-module
```

## 효과적인 상황

- core가 api, batch, support 구현체를 몰라야 할 때.
- 도메인별 요구사항이 계속 추가되어 구조적 경계가 필요할 때.
- 외부 기술을 adapter로 숨기고 port/adapter 형태로 관리하고 싶을 때.
- 코드 구조 자체가 문서 역할을 해야 할 때.

## 실패하는 상황

- 모듈을 너무 일찍 쪼개면 Gradle 설정, 테스트 wiring, 의존성 관리 비용이 먼저 커진다.
- 팀이 경계 규칙을 공유하지 않으면 멀티모듈이어도 순환 의존과 우회 참조가 생긴다.
- domain을 순수하게 만들겠다는 목표만 앞서면 보일러플레이트가 늘고 코드 의도가 흐려질 수 있다.

## 실전 규칙

```text
core
- business usecase
- domain rule
- port interface

api / batch
- 실행 진입점
- protocol adapter

support
- DB / Redis / S3 / OAuth / FCM 구현체
```

처음부터 모든 것을 분리하지 않는다. 변경 임계점이 왔을 때 레고 블록처럼 분리한다.
