## 판단 기준

### 경로 A — Lock 경합 방지 (WHERE 없는 쿼리)

```
UPDATE/DELETE 실행 전:
  1. EXPLAIN으로 인덱스 사용 여부 확인
  2. SELECT로 영향 행 수 먼저 확인
  3. 대량 업데이트 → 배치/LIMIT 분할 처리

WHERE 없는 UPDATE의 Lock 확산은 DB마다 다름:

  [MySQL MyISAM]
    → 진짜 Table Lock
    → SELECT 포함 모든 읽기/쓰기 차단 (가장 심각)

  [MySQL InnoDB]
    → Full Scan → 모든 Row에 Row Lock (기술적 Row Lock 집합)
    → MVCC: SELECT는 통과, INSERT/UPDATE/DELETE는 blocking
    → 실질적으로 Table Lock과 동일한 효과

  [PostgreSQL]
    → ROW EXCLUSIVE Lock(테이블 수준, 경량) + 각 Row Lock
    → MVCC: 새 행 INSERT는 통과
    → 기존 행 UPDATE/DELETE만 blocking
    → InnoDB보다 완화되나, 기존 행 UPDATE 多 워크로드에서는 동일하게 위험

공통 결과: 커넥션 점유 시간 증가 → [[현상-커넥션 풀 고갈 (Connection Pool Exhaustion)]]
```

### 경로 B — 트랜잭션 범위 최소화 (외부 I/O)

```
@Transactional 안에 외부 I/O가 있는가?
  │
  ├── NO  → 유지, 범위를 최대한 짧게
  │
  └── YES → 외부 호출을 @Transactional 밖으로 분리 (유일한 해결책)
              │
              └── 외부 호출 실패 시 DB 롤백도 필요한가?
                    │
                    ├── NO  → 분리만으로 해결
                    └── YES → 분산 트랜잭션 영역
                               Saga / Transactional Outbox 패턴 사용
```

## 효과적인 상황

- 외부 API 호출, 파일 처리 등 느린 작업이 `@Transactional`과 섞인 경우
- 커넥션 풀 고갈 징후(타임아웃 에러) 원인 진단

## 실패하는 상황

- `PROPAGATION_NOT_SUPPORTED`로 커넥션 반납을 기대하는 경우
  → 외부 트랜잭션을 suspend할 뿐, 커넥션은 반납되지 않음 (오히려 2개 점유)
- 외부 호출 실패 시 DB 롤백까지 `@Transactional` 하나로 묶으려는 경우
  → 근본적으로 분산 트랜잭션 문제 — 애플리케이션 레이어에서 해결 불가
