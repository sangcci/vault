## 판단 기준

- 변경 경계는 같은 데이터를 다루는지가 아니라, 같은 이유로 바뀌는지를 기준으로 나눈다.
- 같은 데이터를 다루더라도 행위자, 목적, 권한, 트래픽 패턴, 장애 허용 범위가 다르면 유지보수 라이프사이클이 다르다.
- 변경 라이프사이클이 다른 기능을 하나의 API, 모듈, DTO, 서비스에 묶으면 서로 다른 변경 이유가 한 코드에 누적되어 결합도가 높아진다.
- Admin API와 일반 사용자 API 분리는 변경 경계가 API 설계에 드러난 대표 사례이며, 자세한 기준은 [[판단기준-API 경계는 데이터가 아니라 변경 이유로 나눈다]]에서 다룬다.

> "The Single Responsibility Principle (SRP) states that each software module should have one and only one reason to change." — [Clean Coder Blog, The Single Responsibility Principle](https://blog.cleancoder.com/uncle-bob/2014/05/08/SingleReponsibilityPrinciple.html)

---

## 판단 흐름

```text
같은 Product 데이터를 사용한다
│
├─ 고객 사용 흐름
│  └─ 변경 이유: UX, 성능, 공개 스펙 안정성
│
├─ 운영자 업무 흐름
│  └─ 변경 이유: 운영 정책, CS, 검수, 권한, 감사 로그
│
├─ 정산 업무 흐름
│  └─ 변경 이유: 수수료, 매출 집계, 회계 정책
│
└─ 결론
   └─ 같은 데이터지만 변경 이유가 다르므로 변경 경계 분리
```

```pseudo
function shouldSplitChangeBoundary(features):
    if features.changeForSameReason:
        return "같은 유지보수 단위 가능"

    if features.changeForDifferentReasons:
        return "변경 경계 분리 우선"

    if features.haveDifferentActors:
        return "변경 경계 분리 검토"

    if features.needDifferentSecurityTrafficFailurePolicy:
        return "운영 경계도 함께 분리 검토"

    return "문맥 판단"
```

## 예시

### 나쁜 기준: 데이터가 같으니 하나로 묶는다

```text
Product라는 같은 데이터
├─ 고객 화면 변경
├─ 운영 정책 변경
├─ 정산 정책 변경
└─ 검색 색인 정책 변경

→ 한 유지보수 단위에 누적
```

이 구조는 처음에는 단순해 보이지만, `Product`라는 데이터 이름 하나 아래에 서로 다른 변경 이유가 섞인다. 고객 화면 변경, 운영 정책 변경, 정산 정책 변경, 검색 색인 정책 변경이 모두 같은 유지보수 단위에 누적된다.

### 좋은 기준: 변경 이유로 나눈다

```text
Product 관련 변경 경계
├─ 고객 사용 경계
│  └─ UX, 성능, 공개 스펙
│
├─ 운영 업무 경계
│  └─ 검수, 권한, 감사 로그
│
├─ 정산 경계
│  └─ 수수료, 매출 집계, 회계 정책
│
└─ 검색 경계
   └─ 색인 필드, 검색 노출 정책
```

같은 `Product`를 다루더라도 각 경계가 바뀌는 이유가 다르다. 이때 변경 경계를 나누면 한쪽의 변경이 다른 쪽의 유지보수 주기를 끌고 가지 않는다.

## API 경계와의 관계

변경 경계는 API보다 넓은 개념이다. API 경계, 모듈 경계, DTO 경계, 서비스 경계는 모두 변경 경계가 코드 구조로 드러난 모습이다.

```text
변경 경계
├─ API 경계
├─ 모듈 경계
├─ DTO 경계
└─ 서비스 경계
```

Admin API와 일반 API를 나누는 문제는 변경 경계가 API 설계에 적용된 사례다. 자세한 판단 기준은 [[판단기준-API 경계는 데이터가 아니라 변경 이유로 나눈다]]를 참조한다.

## 효과적인 상황

- 같은 도메인 데이터를 여러 행위자가 서로 다른 목적으로 사용할 때.
- Admin API와 일반 사용자 API를 같은 컨트롤러나 서비스에 둘지 고민할 때.
- 화면/API/모듈/DTO/서비스 경계를 데이터 이름 기준으로 나누고 있는지 의심될 때.
- 트래픽, 권한, 장애 허용 범위, 배포 주기가 서로 다른 기능이 한 단위에 섞여 있을 때.

---

## 실패하는 상황

- 같은 데이터를 다룬다는 이유만으로 API를 하나로 합치면, 서로 다른 변경 이유가 한 코드에 누적된다.
- Admin 기능의 무거운 조회나 내부 정책 변경이 일반 사용자 API의 성능·안정성 요구와 충돌할 수 있다.
- 반대로 변경 이유가 같은 기능까지 무조건 나누면 요청 수, 중복 DTO, 조합 복잡도가 늘어난다.
- 경계를 권한만으로 보면 운영 라이프사이클, 트래픽 패턴, 장애 영향 차이를 놓친다.

## 판단 질문

| 질문 | 의미 |
|---|---|
| 같은 데이터라서 묶는가, 같은 변경 이유라서 묶는가? | 경계 기준 점검 |
| 사용하는 행위자가 다른가? | 고객·관리자·정산·검색 등 라이프사이클 차이 |
| 변경을 요구하는 사람이 다른가? | 고객 UX, 운영 정책, 정산 정책 등 변경 축 |
| 권한·감사 로그·보안 요구가 다른가? | Admin/API 분리 필요성 |
| 트래픽 패턴과 장애 허용 범위가 다른가? | 배포·운영 단위 분리 필요성 |
| 함께 배포되고 함께 장애 나도 괜찮은가? | 유지보수 단위 검증 |
