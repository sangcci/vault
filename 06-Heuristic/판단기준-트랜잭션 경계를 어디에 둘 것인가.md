## 판단 기준

- 여러 저장 작업이 하나의 업무로 성공/실패해야 한다 → application service에 둔다.
- dirty checking으로 aggregate 상태 변경을 감지해야 한다 → application service에 둔다.
- 동시성 제어와 일관된 조회 스냅샷이 필요하다 → application service에 둔다.
- 단일 저장/삭제이고 repository 구현체 트랜잭션으로 충분하다 → service에 무조건 붙이지 않는다.
- 외부 API 호출이 오래 걸린다 → DB 트랜잭션 안에 넣지 않는다.

```text
request
  |
  v
@Transactional usecase
  - read aggregate
  - apply domain rule
  - save changes
  - raise event
```

## 효과적인 상황

- 한 usecase 안에서 여러 aggregate나 여러 table을 함께 바꿀 때.
- cache evict, event publish, audit log append 타이밍을 commit 기준으로 맞춰야 할 때.
- service 전체에 습관적으로 `@Transactional`이 붙어 경계가 흐려졌을 때.

## 실패하는 상황

- 모든 public method에 붙이면 읽기/쓰기 경계가 흐려진다.
- 외부 network I/O를 transaction 안에 넣으면 lock 보유 시간이 늘어난다.
- self-invocation 때문에 기대한 transactional proxy가 적용되지 않을 수 있다.
