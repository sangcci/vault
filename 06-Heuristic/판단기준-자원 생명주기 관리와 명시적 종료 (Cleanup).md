# Heuristic: 자원 생명주기 관리와 명시적 종료 (Cleanup)

## 기준

외부 자원(파일, 네트워크, 메모리, 파이프)을 사용하기 위해 연결을 생성했다면, 사용이 끝나는 즉시 **명시적으로 연결을 종료(`close()`)**해야 한다.

## 왜 중요한가? (Reasoning)

1. **종결 신호(EOF) 전달**: 
   - 파이프나 소켓의 경우, 쓰는 쪽에서 연결을 닫지 않으면 읽는 쪽은 데이터가 더 올 것이라 믿고 영원히 대기하게 된다. (`[[본질-교착 상태 (Deadlock)]]` 유발)
2. **자원 고갈 (Leak)**: 
   - OS가 할당할 수 있는 파일 디스크립터(FD), 포트 번호, 메모리는 유한하다. 닫지 않은 연결이 쌓이면 시스템 전체가 마비된다.
3. **데이터 정합성**: 
   - 버퍼링된 데이터가 있는 경우, `close()` 호출 시점에 남은 데이터가 물리적 장치로 완전히 플러시(Flush)된다.

## 적용 사례

### 1. 유닉스 파이프
- 쉘이 자식들에게 입출력을 넘겨준 뒤 자기 자신의 파이프 FD를 닫지 않으면, 자식들은 파이프가 아직 살아있다고 판단하여 종료되지 않는다.

### 2. Java InputStream / OutputStream
- `try-with-resources` 구문을 사용하지 않고 수동으로 열었다면 `finally` 블록에서 반드시 `close()`를 호출해야 한다. 그렇지 않으면 핸들이 유지되어 파일을 삭제하거나 수정할 때 에러가 발생할 수 있다.

### 3. 네트워크 소켓 / DB Connection
- 연결을 끊지 않고 방치하면 서버의 FD 제한에 걸리거나 DB 커넥션 풀이 고갈되어 새로운 요청을 처리할 수 없게 된다.

## 만능 문장 템플릿

> "자원 관리는 단순히 메모리를 아끼는 것이 아니라, **상대방 프로세스나 시스템에게 작업의 종료(EOF)를 명시적으로 알리는 동기화 행위**입니다."
