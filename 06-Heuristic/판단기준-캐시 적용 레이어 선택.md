## 판단 기준

```text
Repository에 @Cacheable을 두어야 하는가?
  ├─ JPA Entity를 반환하는가?
  │    └─ Yes → Service 레이어로 이동
  │              이유: detached Entity → LAZY 연관관계 접근 시 LazyInitializationException
  └─ DTO / Projection을 반환하는가?
       └─ Yes → Repository도 허용
                 이유: 영속성 컨텍스트와 무관한 객체, lazy 문제 없음

외부 API / 레거시 시스템 호출인가?
  └─ Yes → Repository도 허용 (JPA 무관)
```

- **원칙**: Entity를 캐시하지 않는다. 캐시 단위는 비즈니스 로직 완료 후 결과물(DTO)이어야 한다.

## 핵심 근거 — LazyInitializationException

```text
[요청 1] 캐시 MISS
  트랜잭션 시작 → EntityManager A 생성
    └→ Order 조회 → managed 상태
         └→ @Cacheable이 Order 객체를 캐시에 저장
  트랜잭션 종료 → EntityManager A 닫힘 → Order는 detached

[요청 2] 캐시 HIT
  트랜잭션 시작 → EntityManager B 생성 (Order를 모름)
    └→ 캐시에서 detached Order 반환
         └→ order.getItems() 호출 (LAZY)
              └→ EntityManager A는 닫혔고, B는 이 객체를 모름
                   └→ 💥 LazyInitializationException
```

- 같은 트랜잭션 안이더라도 **캐시에서 꺼낸 Entity는 현재 EntityManager에 등록되어 있지 않다.**
- 연관관계 속성(LAZY)을 초기화하려면 살아있는 Session이 필요한데, 해당 Entity와 연결된 Session은 이미 닫혀있다.

## 효과적인 상황

- Service에 @Cacheable + DTO 반환: 비즈니스 로직 완료 후 결과를 캐시, 무효화 조건이 비즈니스 이벤트와 일치
- Repository에 @Cacheable + DTO projection: `@Query`로 DTO 직접 반환 시 lazy 문제 없음

## 실패하는 상황

- Repository에 @Cacheable + JPA Entity 반환: detached Entity의 LAZY 연관관계 접근 시 예외
- EAGER로 바꿔서 해결 시도: N+1 문제 및 불필요한 데이터 로딩 발생

## 캐싱 단위(Granularity) 선택

```text
리스트 캐싱                      개별 객체 캐싱
──────────────────────────────  ──────────────────────────────
장점: 구현 단순, 단일 엔트리      장점: 변경된 객체만 정밀 무효화
단점: 1개 변경 → 전체 무효화     단점: 별도 캐싱 레이어 필요
      메모리 낭비                       목록 조합 시 N번 캐시 조회
      stale 데이터 범위 넓음
```

**선택 기준**:
- 목록 내 개별 항목의 변경 빈도가 낮고 전체 갱신이 자주 발생 → 리스트 캐싱
- 항목별 변경 빈도가 높고 무효화 비용이 중요 → 개별 객체 캐싱 + 별도 조합 레이어

**별도 캐싱 레이어 패턴**:
```text
CacheService (캐싱 전담 빈)
  ├─ @Cacheable("product")  findById(id) → 개별 DTO 캐싱
  └─ @Cacheable("productList") findTop10() → 리스트 캐싱 (별도 네임스페이스)

@CacheEvict("product") 로 개별 무효화
@CacheEvict("productList") 로 목록 무효화 — 독립적으로 제어 가능
```
