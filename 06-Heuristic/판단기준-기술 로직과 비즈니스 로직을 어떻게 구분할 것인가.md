## 판단 기준

- 기술이 바뀌어도 core 규칙이 변하지 않는다 → 기술 로직이다.
- 제품의 정책이나 사용자 권한 의미가 바뀔 때만 변한다 → 비즈니스 로직이다.
- 특정 vendor, protocol, SDK 이름이 필요하다 → api/support adapter로 밀어낸다.
- “누가 무엇을 할 수 있는가”를 판단한다 → core의 usecase/domain 규칙으로 끌어올린다.
- “어떻게 인증 정보를 얻는가”를 처리한다 → 기술 로직으로 둔다.

```text
Kakao OAuth2 / JWT / FCM / Redis / S3
        |
        v
adapter: 기술을 말함
        |
        v
port: core가 이해하는 말
        |
        v
core: 제품 규칙을 말함
```

---

## 효과적인 상황

- OAuth2 provider, FCM, DB, cache 같은 기술이 core 코드에 새고 있을 때.
- 인증과 인가가 섞여서 “로그인 방식”과 “권한 판단”이 같은 코드에 있을 때.
- 알림 전송 기술과 알림 대상 결정 정책이 한 클래스에 모여 있을 때.

---

## 실패하는 상황

- 모든 Spring 의존성을 제거하는 것을 목표로 삼으면 현실 비용이 커질 수 있다.
- 비즈니스 언어를 너무 좁게 잡으면 “응답 시간 개선”, “검색 동기화” 같은 개발 품질 언어를 놓친다.
- 기술 로직을 무조건 나쁘게 보면 운영상 필요한 실행 정책까지 도메인에서 숨기기 어렵다.

---

## 예시

| 항목 | 분류 | 이유 |
|---|---|---|
| OAuth2 authorization code 교환 | 기술 로직 | provider가 바뀌면 구현이 바뀐다 |
| JWT parsing | 기술 로직 | token format과 security infrastructure 문제다 |
| 현재 사용자가 모집글을 수정할 수 있는지 | 비즈니스 로직 | 권한 정책과 도메인 상태 판단이다 |
| FCM token 저장 | 기술 로직 | push provider 저장 방식이다 |
| 알림을 누구에게 보내야 하는지 | 비즈니스 로직 | 제품 정책이다 |
