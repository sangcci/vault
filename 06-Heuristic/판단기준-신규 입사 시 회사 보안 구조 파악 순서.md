## 판단 기준

MAC, RBAC, DAC 레이어 순으로 파악한다. **위에서 아래로** — 강제 정책부터 세부 권한 순.

```
1단계: MAC 파악  → 조직이 강제하는 절대 금지 항목
2단계: RBAC 파악 → 내 역할(Role)에 부여된 권한 범위
3단계: DAC 파악  → 내가 소유하거나 위임받은 리소스 권한
```

## 1단계 — MAC: 무엇이 원천 차단되는가

> "아무리 해도 안 되는 것"을 먼저 파악

| 확인 항목 | 방법 |
|---|---|
| 사용 가능한 클라우드 리전 | AWS SCP 또는 인프라팀 문의 |
| 차단된 서비스 목록 | 직접 콘솔 접속 시도 → 에러 메시지 |
| 네트워크 레벨 차단 | 외부 IP 접근 가능 여부, VPN 필수 여부 |
| 보안 정책 문서 | 회사 내부 위키, 보안 서약서 |

```
실전 질문:
"이 환경에서 퍼블릭 S3 버킷을 만들 수 있나요?"
→ 된다면 SCP 미설정 / 안 된다면 MAC으로 차단됨
```

## 2단계 — RBAC: 내 역할의 권한 범위는 어디까지인가

> "내 Role로 무엇을 할 수 있는가"

| 확인 항목 | 방법 |
|---|---|
| 부여된 IAM Role/Group | AWS 콘솔 → IAM → 내 계정 확인 |
| 접근 가능한 서비스 목록 | 콘솔 직접 탐색, 403 에러 지점 확인 |
| 배포 권한 여부 | CI/CD 파이프라인 접근 가능 여부 |
| DB 접근 권한 | 읽기 전용인지, 쓰기 가능한지 |

```
실전 질문:
"프로덕션 RDS에 직접 접속할 수 있나요?"
→ 안 된다면 RBAC으로 분리된 것 (개발/운영 Role 분리)
```

## 3단계 — DAC: 내가 소유·위임받은 리소스는 무엇인가

> "내가 직접 제어할 수 있는 것"

| 확인 항목 | 방법 |
|---|---|
| 담당 서비스의 리소스 목록 | 팀 내 문서, S3 버킷/EC2 태그 확인 |
| 리소스 공유 정책 | 타 팀에 어떤 권한을 위임할 수 있는지 |
| 시크릿·환경변수 관리 주체 | AWS Secrets Manager, Parameter Store 접근 여부 |

## 효과적인 상황

- 입사 첫 1~2주 온보딩 기간
- 인프라·백엔드 직무 (권한 구조가 업무에 직결)
- 보안 사고 발생 시 원인 레이어 특정

## 실패하는 상황

- 문서가 없는 스타트업: MAC 정책 자체가 미설정일 수 있음 → 직접 시도해보며 파악
- 권한이 개인에게 직접 부여된 경우 (RBAC 미적용): Role 개념 없이 계정마다 다름

## 실전 온보딩 체크리스트

```
[ ] VPN 없이 접근 가능한 시스템 목록 파악      ← MAC
[ ] 내 IAM 계정의 Policy 문서 확인             ← RBAC
[ ] 프로덕션/개발 환경 Role 분리 여부 확인      ← RBAC
[ ] 내 담당 S3 버킷·EC2 인스턴스 태그 확인     ← DAC
[ ] 시크릿 관리 시스템 접근 가능 여부 확인      ← DAC/RBAC
[ ] CloudTrail 감사 로그 활성화 여부 확인       ← MAC (감사 무결성)
```
