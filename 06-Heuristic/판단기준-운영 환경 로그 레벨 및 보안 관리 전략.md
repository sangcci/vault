## 로깅 전략 (Pseudo Code)

```java
public class SecurityLogger {
    public void logRequest(Request req) {
        // 1. 운영 환경 로그 레벨 체크 (INFO 이상만)
        if (Env.isProd() && req.level < Level.INFO) return;

        // 2. 민감 데이터 마스킹 (Masking)
        String payload = req.getPayload();
        payload = payload.replaceAll("password\":\"[^\"]+\"", "password\":\"***\"");
        payload = payload.replaceAll("ssn\":\"[^\"]+\"", "ssn\":\"***\"");

        // 3. 구조화된 로그 출력 (Structured JSON)
        System.out.println(String.format(
            "{\"traceId\":\"%s\", \"level\":\"%s\", \"msg\":\"%s\", \"data\":\"%s\"}",
            req.getTraceId(), req.getLevel(), req.getMessage(), payload
        ));
    }
}
```

## 판단 기준 (Criteria)

### 1. 로그 레벨 설정 (Standard: INFO)
- **왜 INFO인가?**: 
    - `DEBUG`/`TRACE`는 시스템의 세부 동작을 모두 기록하므로 성능 저하와 스토리지 고갈을 유발함.
    - 운영 중에는 비즈니스 흐름의 주요 변곡점(로그인, 결제, 에러 발생 등)만 추적할 수 있으면 충분함.
- **Exception Case**: 특정 버그 추적이 필요한 경우에만 일시적으로 특정 패키지의 로그 레벨을 `DEBUG`로 상향 조정함.

### 2. 로그에 절대 포함하지 말아야 할 것
- **민감 데이터**: 비밀번호, API Key, 계좌번호, 주민번호, 개인 이메일 등.
- **SQL 실 파라미터**: 보안 및 개인정보 유출 방지를 위해 `?` 형태의 구조만 남김.
- **대용량 바이너리**: 이미지 데이터, 대형 JSON 등은 로그 저장소를 마비시킴.

### 3. 로그 보관 및 관리 전략
- **Retention Policy**: 법적 근거 또는 비즈니스 요구사항에 따라 30일, 90일 등 명확한 보관 주기 설정.
- **망 분리**: 모니터링 시스템(Grafana, Loki 등)이 외부망과 통신하더라도, 원본 로그 저장소는 내부망에서 접근 제어를 엄격히 해야 함.

## 효과적인 상황 (Effective When)

- 대규모 트래픽이 발생하는 서비스에서 시스템 안정성을 유지해야 할 때.
- 개인정보 보호 규정(GDPR, ISMS 등)을 준수해야 하는 환경.

## 실패하는 상황 (Fails When)

- 로그가 너무 없어 장애 발생 시 원인 파악이 아예 불가능한 경우 (적절한 에러 스택 트레이스 누락).
- 로그 시스템 자체가 보안 취약점이 되어 데이터가 외부로 유출되는 경우.

## 실무 팁 (Tips)

- **[[개념-구조화된 로그 (Structured Logging)]]**: JSON 형식으로 로그를 남겨 Elasticsearch/Loki 등에서 필드 기반 필터링 및 메트릭 집계가 가능하도록 함. `traceId` 필드로 단일 요청 전체 흐름 재구성.
- MDC (Mapped Diagnostic Context): 요청별 고유 ID를 ThreadLocal에 저장해 로그에 자동 주입하는 메커니즘. [[개념-구조화된 로그 (Structured Logging)]]와 결합 시 효과 극대화.
