## 핵심 질문 (Core Question)

> "Linux는 OS가 아니라 커널이다"는 말이 있다. 그렇다면 우리가 OS라고 부르는 것의 경계는 어디인가? 데스크톱 환경, 쉘, 네트워크 매니저는 OS의 일부인가, 그냥 앱인가?

## 가설 및 추론 (Hypothesis)

- 커널 공간에서 실행되는 것 = 커널, 사용자 공간에서 실행되는 것 = 애플리케이션
- 따라서 OS ≈ 커널이 아닐까?

## 검증 및 팩트 (Verification)

### 1. CPU 실행 모드: 선은 존재한다

```text
┌─────────────────────────────────────────┐
│  사용자 공간 (User Space / Restricted)   │
│  - 안전한 명령어 서브셋만 실행 가능        │
│  - 하드웨어 직접 접근 불가                │
│  - 커널에 시스템 콜로 위임                │
├─────────────────────────────────────────┤
│  커널 공간 (Kernel Space / Privileged)   │
│  - 모든 CPU 명령어 실행 가능              │
│  - 페이지 테이블, 인터럽트, 하드웨어 제어 │
└─────────────────────────────────────────┘
```

이 선을 기준으로 커널/비커널을 나눌 수 있다.

### 2. "OS처럼 보이는 것"들의 실제 위치

```text
[커널 공간] 메모리 관리 / 스케줄러 / 파일시스템 드라이버 / 네트워크 스택
─────────────────────────── 경계선 ───────────────────────────
[사용자 공간] 쉘 / display server (Wayland/X11) / 데스크톱 환경 (GNOME/KDE)
             네트워크 매니저 / 패키지 매니저 / SSH 서버 / 로깅 데몬
```

우리가 "Ubuntu답다", "macOS답다"고 느끼게 하는 것들은 **전부 사용자 공간 프로세스**.

### 3. init이 OS의 나머지를 소환한다

커널은 [[개념-Init 프로세스 (PID 1)]] 하나만 직접 생성. 이후 모든 서비스 인프라는 init이 fork+exec로 만들어냄.

```text
커널 (Privileged)
  └──[직접 생성]──→ init (PID 1)
                       └──[fork+exec]──→ networkd, sshd, display server ...
                                               └──[fork+exec]──→ 사용자 앱
```

### 4. 그래서 Linux 배포판이 존재한다

```text
Linux 커널 (공통)
    +
사용자 공간 구성 (배포판마다 상이)
    = Ubuntu (GNOME + systemd + apt)
    = Fedora (KDE + systemd + dnf)
    = Alpine (busybox + openrc)
```

커널 개발자는 init 이후 무슨 일이 벌어지는지 책임지지 않음 — 그게 배포판의 역할.

## 결론 (Conclusion)

- **1차 경계 (명확)**: 커널 공간에서 실행 = 커널. 나머지는 "OS의 사용자 공간 부분" 또는 "애플리케이션"
- **2차 경계 (불명확)**: 사용자 공간 중 어디까지가 OS이고 어디서부터가 앱인가 — 명확한 정의 없음
- 핵심 통찰: **Linux는 커널이고, Ubuntu/Fedora 같은 배포판이 비로소 OS에 가깝다**
