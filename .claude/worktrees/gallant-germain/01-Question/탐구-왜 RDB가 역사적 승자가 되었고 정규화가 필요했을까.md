# Question: 왜 RDB(테이블 구조)가 역사적 승자가 되었고, 정규화가 필요했을까?

## 🧐 질문의 배경

단순히 "정규화는 중복을 없애는 것"이라고만 외우기엔 부족함이 있습니다. 1970년대에도 NoSQL과 유사한 그래프/계층형 DB가 있었음에도 불구하고, 왜 2차원 표(Table) 형태의 RDB가 수십 년간 IT 세계를 지배하게 되었는지, 그 본질적인 이유와 시대적 맥락을 이해하고 싶습니다.

## 💡 핵심 통찰 (Insight)

### 1. 과거 모델의 한계: "데이터 종속성과 절차적 접근"
RDB 이전의 모델들은 데이터 간의 관계를 물리적인 **포인터(주소)**로 직접 연결했습니다. 이는 프로그램 코드와 데이터 구조가 강력하게 결합되는 결과를 초래했습니다.
- **절차적 접근 (How):** 개발자가 데이터를 찾기 위해 **"어떤 경로로 이동할지"**를 루프와 조건문으로 직접 코딩해야 했습니다.
- **COBOL 예시 (네트워크 모델):**
    ```cobol
    MOVE 'IT' TO DEPT-ID IN DEPARTMENT.
    FIND ANY DEPARTMENT USING DEPT-ID. 
    FIND FIRST EMPLOYEE WITHIN DEPT-EMP-SET. /* 부서-직원 사이의 물리적 연결(Set)을 직접 참조 */
    
    PERFORM UNTIL DB-STATUS-OK = FALSE
        GET EMPLOYEE
        DISPLAY EMP-NAME
        FIND NEXT EMPLOYEE WITHIN DEPT-EMP-SET /* '다음 데이터의 주소'를 직접 호출 */
    END-PERFORM.
    ```
- **치명적 단점 (유지보수의 지옥):** * **구조 변경의 취약성:** 만약 `부서-직원` 사이에 `팀`이라는 계층이 추가되면, 기존의 연결선(`DEPT-EMP-SET`)이 끊어지므로 위 코드는 에러가 발생합니다. 개발자는 모든 루프를 뜯어고쳐 `부서 -> 팀 -> 직원` 순으로 다시 경로를 짜야 합니다.
    - **물리적 종속성:** 데이터가 디스크의 다른 위치로 이동하여 주소(Pointer)가 바뀌면, 프로그램은 엉뚱한 곳을 참조하게 됩니다.

### 2. RDB의 혁명: "데이터 독립성과 선언적 접근"
E.F. Codd는 데이터의 **물리적 위치(Storage)**와 **논리적 로직(Code)**을 완전히 분리하여 **데이터 독립성**을 확보했습니다.
- **선언적 접근 (What):** 개발자는 경로를 고민하지 않고 SQL을 통해 **"무엇을"** 원하는지만 선언합니다.
    ```sql
    /* 구조가 바뀌어 Join이 늘어나도 '데이터를 찾는 방식'은 DB가 고민합니다 */
    SELECT EMP_NAME FROM EMPLOYEE JOIN DEPT ON ... WHERE DEPT_NAME = 'IT';
    ```
- **Optimizer의 역할:** 경로 탐색의 책임이 **DB 엔진(Optimizer)**으로 넘어갔습니다. 데이터 구조가 바뀌거나 데이터가 디스크 어디에 있든, Optimizer가 실시간으로 최적의 실행 계획을 세웁니다.
- **결과:** 개발자는 "서울역 가는 법(How)"을 외우는 대신, "서울역으로 가달라(What)"고 말하기만 하면 됩니다. 덕분에 데이터 구조가 변해도 비즈니스 로직은 안전하게 보호됩니다.

### 3. 하드웨어적 배경: "저장 공간이 금값이었던 시대"
RDB가 주류가 된 1970~80년대는 하드디스크 비용이 극도로 비쌌습니다.
- **전략:** "비싼 디스크 공간을 아끼기 위해, 상대적으로 저렴한 CPU 연산을 더 많이 사용하자."
- **정규화(Normalization)의 경제학:** 데이터를 극한으로 쪼개 중복을 제거함으로써 저장 공간을 절약하고, 대신 **CPU(Join 연산)**를 써서 데이터를 조립하는 경제적 트레이드오프를 선택한 것입니다.

## 🚀 심화 탐구: RDB의 생존 이유와 이상 현상의 본질
### 1. 이상 현상(Anomaly)과 정규화의 인과관계
- **원인:** 데이터를 편의상 한 테이블에 뭉쳐 **중복 저장(비정규화)**했기 때문입니다.
    - **삽입 이상:** 부서 정보만 넣고 싶은데 직원 정보가 없어 입력을 못 함.
    - **삭제 이상:** 직원을 지웠더니 부서 정보까지 사라짐.
    - **갱신 이상:** 중복된 데이터 중 하나만 수정되어 데이터가 꼬임.
- **처방:** 이러한 이상 현상을 치료하기 위해 수학적 근거(함수적 종속성)를 바탕으로 테이블을 쪼개는 **'무손실 분해(정규화)'**를 수행합니다.
### 2. 왜 여전히 RDB인가? (vs NoSQL)
- **책임의 소재:** RDB는 데이터 무결성의 책임을 **DB 엔진(ACID)**이 집니다. 반면 NoSQL은 그 책임을 **애플리케이션(개발자)**에게 넘깁니다.
- **철학의 차이:** RDB는 "성능을 희생하더라도 정확성(Consistency)을 보장한다"는 철학입니다.

## 🚀 시대의 변화와 NoSQL의 귀환
오늘날 다시 NoSQL이 주목받는 이유는 하드웨어 환경이 정반대로 변했기 때문입니다.
1. **Storage is Cheap:** 디스크 비용이 거의 공짜가 되어 중복 저장이 죄악이 아닙니다.
2. **CPU is the Bottleneck:** 데이터 양이 폭증하며 수억 건의 **Join**을 감당하는 CPU 연산 비용이 가장 커졌습니다.
3. **결론:** 이제는 **CPU 비용(Join)을 아끼기 위해**, 저렴한 **디스크 공간을 낭비(중복 저장/NoSQL)**하는 새로운 전략이 성립하게 되었습니다.
