## 핵심 질문

"성능을 약간 희생하면서까지 Spring의 거대한 IoC/DI 컨테이너를 써야만 하는 이유는 무엇인가?"

## 탐구 결과

### 1. 성능을 팔아 '유연성'과 '생산성'을 사다

- **Cost**: 리플렉션 사용, Proxy 객체 생성, 부팅 시간 지연, 메모리 추가 점유.
- **Gain**: 비즈니스 로직과 인프라 설정의 완벽한 분리. 코드 한 줄 수정 없이 DB나 메시지 큐를 교체할 수 있는 확장성.

### 2. 테스트 가능한 아키텍처 (Testability)

- 의존성을 외부에서 주입해주므로, 테스트 시점에 실제 DB 대신 가짜(Mock) 객체를 끼워 넣는 것이 매우 쉽습니다. 이것이 견고한 대규모 시스템을 만드는 핵심 비결입니다.

### 3. AOP를 통한 공통 관심사 제거

- 트랜잭션 관리, 로깅, 권한 체크 같은 지저분한 코드를 비즈니스 로직에서 제거할 수 있습니다. Spring이 객체를 주입할 때 몰래 '프록시'를 끼워 넣어 기능을 덧붙여주기 때문에(IoC) 가능합니다.

### 4. 이미 증명된 마틴 파울러의 통찰

- 마틴 파울러는 2004년 아티클에서 컴포넌트 조립의 주도권을 외부에 넘기는 것이 소프트웨어 복잡도를 관리하는 결정적 열쇠임을 설파했습니다.

## 인사이트

> **"Spring은 단순한 라이브러리가 아니라, 거대한 Object Graph를 관리하는 지능형 공장이다. 개발자가 비즈니스에만 집중하게 만드는 것이 이 비싼 프레임워크의 진정한 가치다."**
