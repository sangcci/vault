## 한 문장 정의 (Definition)

> (사전적) 수신자가 먼저 브로커에 아웃바운드 연결을 맺고 세션을 유지한 채, 이후 들어오는 요청을 그 기존 세션을 통해 역방향으로 전달받는 기법.
> (이해용) 문이 잠긴 방에서 나가는 문을 먼저 열어두고, 외부 요청이 그 통로를 타고 들어오게 하는 것.

## 해결하는 문제 (Problem Solved)

- 방화벽·NAT·ISP 포트 차단으로 인바운드 포트를 열 수 없는 환경
- 공인 IP가 없거나 포트 포워딩 권한이 없는 환경 (가정용 회선, 사내망 등)

## 치르는 비용 (Cost/Trade-off)

- 브로커(중간 서버)에 의존 → 브로커 장애 시 연결 전체 단절
- 연결 단절 시 재수립 로직 필요 (`restart: unless-stopped` 등)
- 브로커 경유로 인한 지연(latency) 추가
- 직접 연결 대비 데이터가 제3자 브로커를 경유

## 동작 원리 (Mechanism)

**일반 인바운드 방식 (포트 개방 필요)**
```
외부 사용자  →→→  공인IP:80  →→→  서버
             방화벽·ISP가 막으면 통신 불가
```

**아웃바운드 터널링 방식 (포트 개방 불필요)**
```
[수신 서버]  →→→  아웃바운드 연결 수립  →→→  [브로커]
             persistent session 유지
             (WebSocket과 동일한 구조)

[외부 사용자]  →→→  [브로커]  →→→  기존 세션 재사용  →→→  [수신 서버]
               인바운드 연결 없이 역방향 전달
```

**세션 유지 방식**
```
WebSocket        HTTP 업그레이드 후 TCP 세션 유지 → 양방향 통신
아웃바운드 터널링  브로커에 먼저 연결 후 세션 유지 → 역방향 요청 수신
```

사용하는 프로토콜은 구현마다 다름:
```
SSH 역터널  →  TCP (소켓 유지)
QUIC 기반   →  UDP (QUIC이 신뢰성·순서·혼잡제어를 직접 구현 → TCP처럼 동작)
HTTP/2      →  TCP (스트림 멀티플렉싱으로 단일 연결에서 다중 요청 처리)
```

## 구현 예시

| 구현체 | 프로토콜 | 특징 |
|--------|----------|------|
| SSH `-R` 옵션 | TCP | OS 기본 제공, 단순 포트 포워딩 |
| Cloudflare Tunnel | QUIC → HTTP/2 fallback | 4개 PoP에 동시 연결, ISP 차단 우회 |
| ngrok | HTTP/2 | 개발·테스트 용도, 임시 공개 URL |
