# Principle: 가용성 (Availability)
**핵심 질문**: "가용성은 단순히 '서버가 켜져 있는 것' 이상의 어떤 의미를 갖는가?“

 ![InkDrawing](<97-Image/Ink/Drawing/2026.2.6 - 17.14pm.svg>)

## One-liner Definition
> (사전적) 시스템이 장애 없이 정상적으로 서비스를 제공할 수 있는 성질.
> (이해용) **"사용자가 필요로 하는 바로 그 순간에, 막힘없이 서비스를 사용할 수 있는 상태"**.

## Usage Examples (문장 3개)
1. "클라우드 제공업체는 99.9% 이상의 **가용성**을 보장하는 SLA를 체결했다."
2. "서킷 브레이커를 도입하여 외부 API 장애 시에도 우리 시스템 전체의 **가용성**을 보호했다."
3. "무중단 배포 전략을 통해 코드 업데이트 중에도 서비스의 **가용성**을 유지하고 있다."

## Recurring Core Problem
모든 시스템 부품(서버, 네트워크, DB)은 반드시 고장 납니다. 특정 부품의 실패가 서비스 전체의 중단으로 이어지는 '단일 장애점(SPOF)'을 어떻게 제거하고 서비스를 지속할 것인가가 핵심 문제입니다.

## Why It Doesn't Disappear
비즈니스 가치가 인터넷을 통해 24/7 제공되는 현대 환경에서, 1분의 중단은 곧 매출 손실과 사용자 신뢰 하락으로 직결되기 때문입니다.

## Trade-offs
- **비용 증가**: 동일한 자원을 2배 이상 유지(이중화)해야 하므로 인프라 비용이 기하급수적으로 늘어납니다.
- **일관성 희생**: 분산 환경에서 가용성을 높이면(AP), 데이터 일치 속도가 늦어질 수 있습니다. ([[본질-일관성의 종류 (Consistency Types)]] 참고)

## Appears As
- **SLA/SLO**: 가동 시간 수치화 (99.99%)
- **Redundancy**: 액티브-스탠바이, 멀티 리전
- **Failover**: 장애 시 자동 전환 메커니즘
