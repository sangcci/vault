# Principle: 단일 스레드 모델

**핵심 질문**: "복잡한 동기화 비용을 없애기 위해 병렬성을 포기하고 무엇을 얻었는가?"

## 한 문장 정의
> (사전적) 하나의 프로세스에서 한 번에 하나의 명령만 처리하는 실행 모델.
> (이해용) **줄 서서 먹는 맛집** (주방장은 한 명이지만, 손님 응대와 요리를 기가 막히게 빨리 전환함).

## 사용 예시
1. Redis는 데이터의 **원자성(Atomicity)**을 보장하고 락(Lock) 관리를 없애기 위해 단일 스레드를 사용합니다.
2. Node.js는 I/O가 많은 웹 서버 환경에서 **컨텍스트 스위칭 비용**을 줄이기 위해 이 모델을 채택했습니다.
3. JavaScript 브라우저 환경은 DOM 조작의 복잡성을 피하기 위해 메인 스레드 하나만 사용합니다.

## 트레이드오프 (Trade-off)
- **Gain (얻는 것):** 원자성 보장, 데드락 없음, 컨텍스트 스위칭 비용 제거.
- **Loss (잃는 것):** 멀티 코어 활용 불가, CPU 집약적 작업 시 전체 블로킹 발생.

## 왜 사라지지 않는가 (Persistence)
- 인간의 사고 방식(순차적 처리)과 가장 유사하며, 디버깅이 쉽고 예측 가능하기 때문입니다.

## 다른 모습들 (Polymorphism)
- OS: 인터럽트 핸들러의 순차적 처리.
- Network: 비동기 논블로킹 I/O (Reactor Pattern).
