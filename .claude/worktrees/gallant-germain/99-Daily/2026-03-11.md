### 점점 높아져보이는 이력서 컷
점점 신입의 프로젝트 경험 레벨이 높아지고 있는 느낌이 든다. 이는 AI의 등장으로 인해 더 가속화되고 있으며, 이런 상황일수록 이 현상에 대해 깊이 분석하고 이해하는 것이 중요하다. 왜 레벨이 높아지고 있을까?

AI가 잘하는 분야를 대체하고 있는데, 그중에서 코드 수정 및 초안 작성, 코드 분석 및 문제 분석 및 해결책 제안은 솔직히 어느 누구보다 잘한다고 생각한다. 문제는 LLM의 기억력이다. 한마디로 비유하면 "기억력이 제로인 아인슈타인"이다. 기억력이 좋지 않아서 대화를 길게 유지할수록 우리가 시킨 Task 중 일부를 수행하지 않고 넘어가는 경우가 생기며 정보 자체의 할루시네이션 현상도 플러스 알파로 등장한다. 그래서 우리는 기억력이 좋지 않은 아인슈타인을 잘 다루는 방법에 집중해야 한다.
기억력이 좋지 않다는 말은, 결국 전체적으로 볼 때 Task를 정리하는 과정에서 인간이 개입해야 하며, 할루시네이션 현상이 아직 존재한다는 것은, 아직 우리가 가진 경험 지식을 바탕으로 선택하고 판단해야 하는 능력이 필수라는 것이다.

아직까지는 비전공자들의 바이브코딩으로 만든 프로그램들의 허점들이 많이 보인다(특히 보안쪽). 또한 전공자들도 AI에게 전권을 맡겨 소프트웨어를 구축하거나 유지보수할수도 없다 - 위와 같은 문제들 때문에. 

그러나 분명한건, AI가 등장하기 전보다 전공자들이 시니어의 영역에 빠르게 도달한다는 점이다. LLM과 함께라면 우리가 굳이 stackoverflow를 뒤져가며 찾을 필요가 없어진다. 우리보다 훨씬 검색 및 분석을 잘 한다는 점은 이미 저명한 사실이다. 그래서 주니어들이 설 자리가 빠르게 사라지고 있다. 공급은 많아지고 수요는 적어지는 상황이 발생하니 회사 입장에서는 수많은 지원자들중 제일 잘하는 1명을 채용할 것이며 자연스럽게 이력서와 포트폴리오는 화려해진다.

한편으로는 다른 시각이 존재한다. LLM이 등장하기 전에도 우리는 고수들의 코드를 복사 붙여넣기해서 해결해왔다는 점이다(상위 1% 제외). 이 현상을 LLM이 대체했을 뿐이며 시장이 단순히 현상의 겉면만 보고 판단해서 과열되었다는 점. 버블이 꼈다는 것이다.

LLM은 언어 모델이다. 즉 우리보다 언어적으로 표현함에 있어서 압도적으로 능력이 뛰어나다. 즉, 말을 엄청나게 논리적으로 잘하는 분들이 몇명 모여서 우리가 질문한걸 분석해서 알려준다고 생각하면 된다.

### 그리디 알고리즘
1. 그리디 알고리즘이란?
각 단계에서 **지금 당장 가장 최선인 선택**을 하는 방식.  
전체를 다 따져보지 않고, 국소적 최적 선택이 전체 최적으로 이어지는 경우에 사용.
2. 최적화 문제 접근법 분류
```
최적화 문제
├── 그리디   : 선택 기준 미리 정해서 한 방향으로 선택, 뒤돌아보지 않음
├── DP       : 부분 문제 탐색 + 결과 저장 + 재활용
└── 완전탐색 : 저장/기준 없이 전부 탐색 (경우의 수 작을 때만)
```
- 그리디 핵심: **"왜 이 기준이 최적인가"** 를 납득할 수 있어야 함
- DP 핵심: **"부분 문제를 어떻게 정의하는가"**
- 그리디가 안 될 때 DP를 고려
3. BOJ 1931 — 회의실 배정
**유형:** Interval Scheduling Maximization (구간 스케줄링 최적화)
**풀이 전략:**
	1. 끝시간 오름차순 정렬 (끝시간 같으면 시작시간 오름차순)
	2. 앞에서부터 탐욕 선택: `next.start >= curr.end` 이면 선택
**왜 끝시간 기준인가:**
```
끝시간이 빠름 = 회의가 일찍 끝남 = 뒤에 여유 시간이 많음 = 선택지가 많음
```

**증명 방식 — 교환 논증 (Exchange Argument):**

> 최적해에서 첫 번째 선택을 끝시간이 가장 빠른 회의로 교체해도  
> 다음 선택 조건이 나빠지지 않는다 → 개수가 줄지 않는다  
> → 끝시간 기준 그리디가 항상 최적

**시간 복잡도:** O(n log n) (정렬 지배)

4. 흔한 실수 — 이중 루프
```java
// 잘못된 코드: O(n²)
for (int i = 0; i < n; i++) {
    for (int j = i+1; j < n; j++) { ... }
}

// 올바른 코드: O(n)
Meeting curr = meetings[0];
for (int i = 1; i < n; i++) {
    if (meetings[i].start >= curr.end) { ... }
}
```
i=1, i=2... 시작은 i=0보다 선택지가 줄거나 같다 → 외부 루프 불필요
5. 그리디 유형 분류

|유형|특징|예시|
|---|---|---|
|정렬 기반 선택|기준 정렬 후 순차 선택|회의실 배정|
|우선순위 큐|매 순간 최솟값/최댓값|다익스트라|
|교환 논증|교체해도 나빠지지 않음 증명|스케줄링|
|구간 커버링|최소 구간으로 전체 덮기|선분 덮기|
6. solve.ac Class 로드맵 (참고)

```
Class 1-2  : 완전탐색, 기본 구현
Class 3-4  : 정렬, BFS/DFS, 그리디, DP 기초  ← 현재
Class 5-6  : 그래프 심화, DP 심화
```
