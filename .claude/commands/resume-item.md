# 이력서 한 줄 + 포트폴리오 항목 생성

사용자가 해결한 문제를 불완전한 문장으로 줬을 때, 아래 두 가지를 생성한다.

**입력:** $ARGUMENTS

---

## 작업 순서

### Step 1 — 입력 분석

입력 문장에서 다음 4가지 요소를 추출하거나 합리적으로 추론한다:
- **도메인**: 어떤 서비스/기능 영역인가 (예: 게시글 조회, 예약 동기화)
- **문제**: 무엇이 문제였는가 (증상 또는 원인)
- **해결**: 어떤 방식/기술로 해결했는가
- **결과**: 어떤 효과가 있었는가 (수치가 있으면 포함, 없으면 정성적으로)

추론한 내용은 사용자에게 먼저 확인하지 않고 초안에 반영한다.
단, 결과 수치가 전혀 없을 경우 `[수치 미확인 — 직접 입력 필요]` 표시를 남긴다.

---

### Step 2 — 이력서 한 줄 생성

아래 형식을 따른다:

```
• [도메인]에서 [문제]를 [해결 방식]으로 해결해 [결과]
```

**원칙:**
- 한 줄 (60자 내외)
- 기술 키워드 자연스럽게 포함
- "단순히 구현했습니다" 형태 금지
- 수치가 있으면 포함

---

### Step 3 — 포트폴리오 항목 생성

아래 구조를 마크다운으로 작성한다:

```markdown
## [이력서 한 줄을 그대로 제목으로]

### 전체 아키텍처
> draw.io / excalidraw로 작성할 다이어그램 구성 요소 제안:
> - [컴포넌트 A] → [컴포넌트 B] → [컴포넌트 C]
> - 표현해야 할 흐름: ...

### 문제 원인
- [문제 1: 어떤 도메인에서 어떤 증상이 발생했는가]
- [문제 2: 근본 원인]
- [문제 3: 영향 범위]

### 해결 과정
- [해결 1: 어떤 방식을 선택했는가]
- [해결 2: 구체적인 구현 흐름]
- [해결 3: 예외/엣지케이스 처리]
- [해결 4: 테스트 방법 — 로컬/Docker 등]

### 결과
- [결과 1: 정량적 개선 수치 또는 정성적 효과]
- [결과 2: 추가 효과]
- [결과 3: 확장 가능성 또는 시스템 안정성 변화]
```

**원칙:**
- 구현 코드 캡처 절대 포함하지 않음
- 아키텍처 그림 구성 요소만 텍스트로 제안 (draw.io/excalidraw로 직접 작성 유도)
- 결과 수치가 없으면 `[수치 미확인 — 직접 입력 필요]` 표기
- 입력에서 추론한 항목은 `(추론)` 표시

---

### Step 4 — 출력 형식

다음 순서로 출력한다:

1. **이력서 한 줄** (바로 복사 가능한 형태)
2. **포트폴리오 항목** (마크다운 그대로)
3. **추론한 항목 목록** — 사용자가 직접 확인·수정해야 할 부분을 마지막에 정리

---

## 참고 원칙 (생각등대 가이드 기준)

- 이력서 한 줄: "문제 + 해결 + 결과 + 도메인" 형태
- 포트폴리오: 문제 3줄 / 해결 3~4줄 / 결과 3줄
- 면접관은 위에서 아래로 읽으므로 문제 해결 역량을 최상단에
- 기술 스택 키워드는 문장 안에 자연스럽게 녹이기
- 수치가 설득력을 결정함
